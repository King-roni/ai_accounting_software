## Migration tooling

Per `Docs/sub/policies/supabase_migration_tooling_policy.md`:

- **Tool:** Supabase CLI only. No Prisma Migrate, no raw `psql` scripts.
- **Filename:** `YYYYMMDDHHMMSS_<descriptive_snake_case>.sql` (generated by `supabase migration new`).
- **Header block:** every file starts with `-- Migration: <name> / -- Ticket: <ref> / -- Block: <NN — name> / -- Description:`.
- **One file, one logical change.**
- **CREATE TABLE → must include indexes + RLS in same file** (lint rule). **Exception: 0001/0002 baseline migrations explicitly defer RLS to Phase 05** (see B02·P01 phase doc DoD: "schema is open until Phase 05"). This README is the explicit waiver record for the lint rule on Phase-01 baseline migrations only.
- **Destructive changes** (`DROP TABLE`, `DROP COLUMN`, `ALTER COLUMN … TYPE`, `DROP INDEX`, `TRUNCATE`) require a paired rollback script in `rollbacks/` with the same prefix.

### Supabase Auth dashboard config (manual — Stage 7-2)

The MCP can't write these settings; they require dashboard access at
`https://supabase.com/dashboard/project/noxvmnxrqlzsdfngfiww/auth/providers`.

**Email provider**
- Enable Email signin: **ON**
- Confirm email: **ON** (required per the phase doc DoD — "login refused until email is verified")
- Secure email change: ON
- Secure password change: ON

**URL configuration** (`/auth/url-configuration`)
- Site URL: `http://localhost:3000` (dev) — staging/prod URLs added as they land
- Redirect URLs allowlist:
  - `http://localhost:3000/auth/callback`
  - `http://localhost:3000/reset-password`

**Password policy** (`/auth/policies`)
- Minimum length: **12** (matches `password_policy.md`)
- Required characters: digits + uppercase + lowercase + special (matches `password_policy.md`)
- Leaked password protection: **ON** (Supabase native HIBP integration; spec's
  custom HIBP hook is deferred — Supabase's built-in coverage is sufficient for MVP)

**SMTP** (`/auth/templates`)
- Supabase platform email is enabled by default and works for local dev (rate-limited).
- Production needs a custom SMTP — candidate providers per
  `Docs/sub/integrations/transactional_email_service_integration.md`:
  Resend / Postmark / SES (EU region).

**Rate limits** (`/auth/rate-limits`)
- Sign-in: 30/hour per IP (Supabase default; spec's
  `rate_limit_configuration_policy.md` allows tuning later).

### Stage 7-2 deviations from the spec (recorded)

- **Custom-auth elements deferred** — the spec's `users.password_hash` (bcrypt
  in app code), `password_reset_tokens` table, separate `sessions` table,
  5-concurrent-session limit, role-based absolute timeout (8h std / 12h
  accountant), and explicit HIBP k-anonymity check are all replaced by
  Supabase Auth's native equivalents. Phase 02 schema is unchanged from
  Phase 01 — no new tables. If a later phase needs the granular
  spec-described controls, they land as Supabase Auth hooks on Block 06's
  audit infrastructure.
- **Audit events deferred** — `LOGIN`, `LOGIN_FAILED`, `SIGNUP`, etc. require
  Block 05's audit-log emission API which doesn't exist yet. Supabase Auth
  emits its own platform audit log; the application audit-event mapping
  lands when B05·P02 is built.
