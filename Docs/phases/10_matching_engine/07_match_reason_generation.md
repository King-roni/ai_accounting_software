## Phase Goal

Produce two artefacts per match record: a structured signal breakdown (audit-friendly, queryable) and a plain-language reason (user-friendly, the thing the user actually reads). After this phase, every match record carries both — `match_signals` JSONB for the engine and `match_reason_plain_language` text for the user.

## Dependencies

- Phase 02 (scoring engine produces the signal breakdown that gets stored)
- Block 04 Phase 03 (`match_records.match_signals`, `match_reason_plain_language` columns already provisioned)
- Block 06 Phase 02 (Privacy Gateway)
- Block 06 Phase 04 (prompt registry — match-reason prompts registered there)
- Block 06 Phase 10 (plain-language pipeline — `generatePlainLanguage(MATCH_REASON, ...)`)

## Deliverables

- **Structured match reason** stored in `match_records.match_signals` JSONB:
  - `score_breakdown`: per-signal contribution (`signal_name → { value, weight, contribution }`).
  - `match_level` (1–4).
  - `decision_factors`: the salient signals that drove the level decision (e.g., `["amount_exact", "supplier_exact", "date_within_3d"]`).
  - `cross_currency`: when applicable, the FX rate used and source (`bank_paired_leg` vs `ecb_fallback`).
  - `cross_period`: when applicable, the days between transaction date and document date.
- **Plain-language reason** stored in `match_records.match_reason_plain_language`:
  - Generated by Phase 10 of Block 06: `generatePlainLanguage('MATCH_REASON', { match_level, score_breakdown, decision_factors, transaction_summary, document_summary })`.
  - **Tier 2 default** (per Block 06 Phase 10).
  - **Tier 3 escalation** triggered explicitly when:
    - Cross-currency case (more complex to explain in natural language).
    - Cross-period case (need to explain the time gap).
    - Two or more ambiguous signals (Levels 2/3 cases where the explanation is non-obvious).
  - Output schema (per Block 06 Phase 10's title/description constraints): single-paragraph, max ~300 chars.
- **Sample outputs:**
  - **Level 1 example:** *"Matched: invoice amount EUR 49.00, transaction amount EUR 49.00, supplier Google Ireland Ltd, invoice dated one day before the payment."*
  - **Level 2 example:** *"Likely match: amount EUR 49.00 and currency match exactly; supplier 'Google Ireland Ltd' matches 'Google IE' with 92% confidence; you've confirmed this supplier 4 times before."*
  - **Level 3 example:** *"Possible match: amount matches, but supplier name and date are different. Please review."*
  - **Cross-currency example:** *"Matched: invoice USD 100.00 converts to EUR 91.20 at the bank's recorded rate (1.097 USD/EUR); transaction amount EUR 91.20."*
- **Cache integration:**
  - Plain-language calls go through Block 06 Phase 09's within-run cache; identical structured inputs return cached strings.
- **Reason regeneration:**
  - When a match's signals change (e.g., user edits the match), the plain-language reason is regenerated. The old reason is preserved in audit (the match record's history) but `match_reason_plain_language` carries the current value.
- **Audit events** (`<DOMAIN>_<PAST_VERB>` convention):
  - `MATCHING_REASON_GENERATED` (with tier used)
  - `MATCHING_REASON_CACHE_HIT`
  - `MATCHING_REASON_REGENERATED`
  - `MATCHING_REASON_FALLBACK_APPLIED`
- **Failure handling (Tier 2/3 AI failure on plain-language generation):**
  - When Block 06 Phase 10's plain-language call fails (timeout, rate limit, model error, schema-validation failure after retries), the match record is NOT held — the run continues.
  - `match_reason_plain_language` is populated with a **deterministic structured-fallback string** assembled from `match_signals.decision_factors` (e.g., *"Match details: amount exact, supplier exact, date within 1 day. Plain-language summary unavailable; structured signals only."*). The fallback is generated locally with no AI call; sub-doc captures the exact template per match level.
  - `match_signals` JSONB still carries the full structured breakdown — the failure only affects the plain-language string.
  - A `LOW`-severity review issue is raised in `'Possible Wrong Match'` noting the plain-language fallback, with a "Regenerate" action that re-attempts the call.
  - `MATCHING_REASON_FALLBACK_APPLIED` audit event captures the failure category (`AI_TIMEOUT`, `AI_SCHEMA_VALIDATION_FAILED`, `AI_RATE_LIMITED`, `AI_OTHER`).
  - **Contract alignment:** this is the same fallback semantics Phase 09's `matching.generate_reasons` registration declares; both phases must agree on the placeholder text shape and the LOW-issue raising rule.

## Definition of Done

- Every match record produced by Phase 02 has both `match_signals` JSONB populated and `match_reason_plain_language` populated within the same workflow phase.
- Plain-language reasons fit Block 06 Phase 10's output schema constraints.
- Cross-currency matches escalate to Tier 3 as expected.
- Cross-period matches explain the time gap in user terms.
- Repeated identical structured inputs return from cache (verified via test).
- Editing a match regenerates the plain-language reason; the old value is in audit.
- Tests cover at least one example per match level + cross-currency + cross-period.
