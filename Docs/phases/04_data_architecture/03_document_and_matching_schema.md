## Phase Goal

Add the operational tables for documents (invoices, receipts, contracts) and the match records linking transactions to documents. After this phase, Blocks 09 and 10 have the schema they need for intake, extraction, and matching.

## Dependencies

- Phase 01 (hashing)
- Phase 02 (`transactions` table; `match_records` FK to it)
- Block 02 Phase 05 (RLS)

## Deliverables

- **`documents` table:**
  - `id` (UUID v7), `organization_id`, `business_id`
  - `source` (`EMAIL`, `DRIVE`, `MANUAL`, `INVOICE_GENERATOR`)
  - `source_location` (e.g., Gmail message id, Drive file id, upload path; provenance for the audit trail)
  - `file_id` (Supabase Storage path; nullable for `Document Stub` records that document an exception with no file)
  - `original_filename`, `document_hash`
  - `document_type` (`INVOICE`, `RECEIPT`, `CONTRACT`, `PROOF_OF_PAYMENT`, `BANK_EVIDENCE`, `STUB`, `OTHER`)
  - **Extracted fields:** `supplier_name`, `supplier_address`, `supplier_country`, `supplier_vat_number`, `invoice_number`, `invoice_date`, `due_date`, `service_period_start`, `service_period_end`, `amount_subtotal`, `amount_total`, `currency`, `vat_amount`, `vat_rate`, `payment_reference`, `client_name`
  - `line_items` (JSONB) — preserved for Block 11's multi-line consolidation (Stage 1: one ledger entry, line detail on document)
  - `extraction_status` (`DISCOVERED`, `INGESTED`, `EXTRACTED`, `LINKED_CANDIDATE`, `MATCHED`, `DISMISSED`)
  - `extraction_confidence_per_field` (JSONB — confidence score per extracted field)
  - `ocr_text_reference` (Storage path for OCR text in the Processing zone — Phase 06)
  - `discovery_reason` (which Phase 09 query template yielded this document)
  - `created_at`, `updated_at`
- **`match_records` table:**
  - `id` (UUID v7), `organization_id`, `business_id`
  - `transaction_id` (FK), `document_id` (FK)
  - `match_level` (`EXACT`, `STRONG_PROBABLE`, `WEAK_POSSIBLE`)
  - `match_method` (`DETERMINISTIC_RULE`, `AI_FALLBACK`)
  - `match_score` (numeric)
  - `match_signals` (JSONB — structured score breakdown per Block 10's signal list)
  - `match_reason_plain_language` (text — generated by Block 06 End-Scan / explanation pipeline)
  - `match_status` (`MATCHED_CONFIRMED`, `MATCHED_AUTO_HIGH_CONFIDENCE`, `MATCHED_NEEDS_CONFIRMATION`, `POSSIBLE_MATCH`, `NO_MATCH`, `REJECTED_MATCH`)
  - `split_payment_flag` (boolean), `split_payment_group_id` (nullable; ties multiple match rows together for split-payment cases per Stage 1's combinatorial detection)
  - `matched_by` (user_id or system principal)
  - `requires_user_confirmation`, `user_confirmation_status`, `confirmed_by`, `confirmed_at`
  - `created_at`, `updated_at`
- **RLS policies** on both tables.
- **Indexes:**
  - `documents(business_id, supplier_vat_number)` for VAT-relevance queries.
  - `documents(business_id, invoice_date)` for cross-period look-back per Stage 1's 1–2 month window.
  - `documents(document_hash)` for cross-source duplicate detection.
  - `match_records(transaction_id)`, `match_records(document_id)`.
  - `match_records(business_id, match_status)` for review-queue filtering.
- **Constraints:**
  - Unique on `(transaction_id, document_id)` where `match_status != 'REJECTED_MATCH'` — the rejection-memory rule from Stage 1 is enforced at row level.

## Definition of Done

- Both tables exist with correct columns, FKs, and constraints.
- RLS policies in place; cross-tenant invariant tests extended.
- A document can be inserted, linked to a transaction via a match record, and the duplicate constraint blocks a second non-rejected link of the same `(transaction, document)` pair.
- `match_status = REJECTED_MATCH` permits the same pair to exist in history without preventing future fresh evaluations of unrelated pairs (per Stage 1 rejection-memory: pair-scoped, not global).
- Indexes verified for the cross-period look-back and the review-status filter.
