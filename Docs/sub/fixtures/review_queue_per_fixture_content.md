# Review Queue Per-Fixture Content

**Category:** Fixtures · **Owning block:** 14 — Review Queue · **Block reference:** Block 14 § Phase 01 (Issue Registry), Phase 02 (Issue Lifecycle) · **Stage:** 4 sub-doc (Layer 2 fixture corpus)

**Purpose:** Defines one canonical fixture issue per issue group (8 groups), plus two additional fixtures for snooze and escalation testing. This corpus is used by `review_queue_live_integration_runbook.md` and as the development seed data for local environment setup. All fixture IDs are stable identifiers; the actual UUID values in the database use the placeholder pattern described below.

---

## Format conventions

Each fixture record is shown in table and detail form. Fields:

| Field | Description |
| --- | --- |
| `fixture_id` | Stable short identifier for this corpus. Used in runbook step references. |
| `issue_type` | Value from `issue_type_registry.issue_type`. Must match exactly. |
| `issue_group` | Value from `issue_type_registry.issue_group`. The routing assertion target. |
| `default_severity` | Enum: `LOW`, `MEDIUM`, `HIGH`, `BLOCKING`. Taken from `issue_type_registry`. |
| `workflow_run_id` | Placeholder: `'00000000-0000-7000-0000-00000000000f'` — a deterministic UUID v7-format placeholder for the fixture run. Replaced with the actual run ID at seed time. |
| `affected_record_type` | The entity type the issue is associated with: `TRANSACTION`, `DOCUMENT`, `INVOICE`, or `WORKFLOW_RUN`. |
| `affected_record_id` | Placeholder: `'00000000-0000-7000-0000-0000000000XX'` where `XX` is the two-digit fixture index. Replaced at seed time. |
| `expected_routing_group` | The `issue_group` the issue must be routed to. Must equal `issue_group`. Listed separately for explicit assertion in tests. |

**UUID note:** Primary keys on `review_issues` use `gen_uuid_v7()`. The placeholders above are v7-format (version nibble = 7) for consistency with the production schema, but they are not generated by `gen_uuid_v7()` — they are deterministic test values.

## Seed script notes

The seed script must: (1) insert all 10 fixture rows in a single transaction; (2) set `workflow_run_id` to the actual fixture run ID; (3) replace placeholder `affected_record_id` values with actual IDs from the fixture transaction/document/invoice seed data; (4) emit `REVIEW_ISSUE_CREATED` for each row via `emitAudit()`; (5) call `review_queue.snooze_issue` for `rq_fix_snoozed_a` to set `snoozed_until` and emit `REVIEW_QUEUE_ISSUE_SNOOZED`.

Teardown reverses Step 5 first (un-snooze), then deletes all 10 rows, then deletes the fixture run.
