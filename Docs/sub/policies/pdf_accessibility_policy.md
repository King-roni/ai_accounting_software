# PDF Accessibility Policy

**Category:** Policies · **Owning block:** 16 — Dashboard & Reporting · **Stage:** 4 sub-doc (Layer 2)

This policy defines the accessibility and archival compliance standards for all customer-facing PDF output produced by the platform. The governing standard is PDF/A-2a (ISO 19005-2). Compliance is verified at CI time using automated tooling; non-compliant PDFs block the build.

---

## Purpose

PDF/A-2a serves two distinct goals that reinforce each other. As an archival standard, it guarantees that a document can be rendered without external resources (fonts, colour profiles, JavaScript) at any future point — a regulatory requirement for accounting records. As an accessibility standard, it requires a tagged content structure that enables screen readers and assistive technology to interpret the document.

These two goals are not in tension. Meeting PDF/A-2a satisfies both simultaneously, and the CI validation step verifies both in a single pass.

---

## Scope

This policy applies to:

- **Invoice PDFs** — generated by `in_workflow.render_invoice_pdf` for all ISSUED and archived invoices.
- **Report PDFs** — generated by `report.generate_period_report` for the period report and accountant pack.
- **Evidence PDFs** — bank statement evidence PDFs generated during the ingestion pipeline.

Internally generated diagnostic exports (not delivered to customers or included in the archive bundle) are **exempt** from PDF/A-2a compliance but must still embed fonts (see Font embedding below).

Credit note PDFs follow the same rules as invoice PDFs.

---

## Standard: PDF/A-2a (ISO 19005-2)

PDF/A-2a is the conformance level that requires both archival constraints (Part 2 of the standard: based on PDF 1.7) and accessibility tagging (the `a` conformance level within Part 2). The `b` conformance level (visual reproduction only, no accessibility tagging) is insufficient. Only `a` is accepted.

Validation uses `veraPDF` with the `PDFA_2_A` validation profile. No other validator is accepted as authoritative for CI purposes.

---

## Accessibility tagging requirements

All PDFs in scope must include a complete tag tree (`MarkInfo/Marked = true`). The following structural elements must be tagged:

- **Headings** — tagged as `H1`, `H2`, or `H3` as appropriate to the document hierarchy.
- **Paragraphs** — tagged as `P`.
- **Tables** — tagged as `Table`, with `TR`, `TH`, and `TD` child tags. Column header cells use `TH` with a `Scope` attribute (`Column` or `Row`).
- **Figures** — tagged as `Figure` with an `Alt` attribute providing a text description. This applies to logos and any chart images embedded in report PDFs.

Reading order must match the visual order of the document. Tagged content that appears in a different logical order from the visual layout is non-compliant. The PDF rendering library's layout engine must be configured to emit tags in visual top-to-bottom, left-to-right order.

The document `Lang` attribute must be set to `en`. This is set in the PDF document-level metadata, not per-element. The `Lang` value must be an IETF BCP 47 language tag. `en` (not `en-US`, not `en-GB`) is the project standard for all customer-facing documents.

---

## Font embedding

All fonts used in PDFs must be fully embedded in the PDF file. No font substitution is permitted. If a font is not embedded, a future renderer may substitute a system font, altering the document's visual appearance and breaking archival integrity.

Subset embedding is permitted: only the glyphs used in the document need be included. Full embedding (all glyphs in the font file) is also acceptable but not required.

Font files are pinned per `zip_bundle_determinism_policy.md`. The embedding call in the PDF rendering module must reference the pinned font path, not a system font path. This is enforced by the startup assertion described in `zip_bundle_determinism_policy.md`.

Exempt documents (diagnostic exports) must still embed fonts. Embedding is not optional for exempt documents — only the PDF/A-2a tag-tree and metadata requirements are relaxed for exempt documents.

---

## Colour contrast

Text in PDFs must meet WCAG 2.1 AA contrast requirements:

- Normal text (below 18pt or below 14pt bold): minimum contrast ratio 4.5:1 against background.
- Large text (18pt or above, or 14pt bold or above): minimum contrast ratio 3:1 against background.

Contrast is measured between the text colour and the background colour at the point of rendering (accounting for any background fill on the containing element).

This requirement is enforced by a CI step that extracts colour values from generated PDFs and computes contrast ratios using the WCAG relative luminance formula. The check runs on the same fixture set as the determinism check. A contrast failure blocks the build.

The design token colours used for PDF generation are defined in `design_tokens.css` and `design_tokens.ts`. When the colour tokens change, the CI contrast check must be re-run before the change is merged. Token changes that cause a contrast failure are blocking.

## CI validation

A `pdfvalidate` CI step runs on every PR that touches:

- Any file under `src/pdf/` (PDF rendering templates, layout, metadata).
- `pdf_font_version_manifest.json`.
- Any file that changes the colour token values used in PDF generation.

The step:

1. Generates a sample PDF from the test fixture set for each PDF type in scope (invoice, report, evidence).
2. Runs `veraPDF` with the `PDFA_2_A` profile against each generated PDF.
3. Runs the WCAG contrast check against each generated PDF.
4. Fails the CI build if any check does not pass.

The `veraPDF` version used in CI is pinned in the CI configuration file. Upgrading `veraPDF` follows the same approval path as upgrading a pinned library (see `zip_bundle_determinism_policy.md`).

Validation failures emit `REPORT_PDF_ACCESSIBILITY_VALIDATION_FAILED` in the CI audit context. This event is not emitted at runtime during normal PDF generation — CI is the enforcement gate.

---

## Runtime enforcement

At runtime, the PDF rendering pipeline does not re-run `veraPDF` on every generated PDF (that would be prohibitively slow). The assumption is that CI has validated the rendering logic against the standard. Runtime enforcement consists of:

- The startup font-embedding assertion (per `zip_bundle_determinism_policy.md`).
- The `Lang` attribute and `MarkInfo/Marked` fields being set by the rendering template, not at call-site discretion.

If a customer reports a PDF accessibility issue that was not caught by CI, the investigation should start with whether the fixture set covers the specific document variant that failed.

---

## Audit event

| Event | Severity | Emission point |
|---|---|---|
| `REPORT_PDF_ACCESSIBILITY_VALIDATION_FAILED` | BLOCKING | CI `pdfvalidate` step, on any `veraPDF` or contrast check failure |

`REPORT_PDF_ACCESSIBILITY_VALIDATION_FAILED` (BLOCKING) — emitted by the CI `pdfvalidate` step when any PDF in the fixture set fails `veraPDF` PDF/A-2a validation or fails the WCAG contrast check. BLOCKING because non-compliant PDF output would enter the archive bundle and fail post-seal verification. Payload (CI audit context): `failing_pdf_type` (`INVOICE`, `REPORT`, or `EVIDENCE`), `validation_rule_id` (the `veraPDF` rule identifier or `WCAG_CONTRAST`), `failure_detail`, `git_commit_sha`.

This event is emitted in the CI test audit context, not in the runtime audit chain.

---

## Failure handling outside CI

If a PDF/A-2a compliance gap is discovered in production (e.g., a customer reports a PDF that fails their own archival validator), the investigation procedure is:

1. Identify the PDF type (invoice, report, evidence) and the specific document that failed.
2. Reproduce the failure using the PDF rendering pipeline in isolation against the relevant input data.
3. Determine whether the failure is covered by the existing CI fixture set. If not, add a fixture that covers it before fixing the rendering code.
4. Fix the rendering code. The CI `pdfvalidate` step must pass before the fix is merged.
5. If the failed PDF was already included in a sealed archive bundle, raise a `REVIEW_HOLD` on that period and follow the bundle re-generation procedure in `archive_verification_policy.md`.

This procedure ensures that production compliance gaps are captured in the fixture set and cannot regress.
