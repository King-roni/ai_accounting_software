# Archive Access Control Policy

**Category:** Policies · **Owning block:** 15 — Finalization & Secure Archive · **Block reference:** Block 15 § Phase 05 (Archive Access & Verification) · **Stage:** 4 sub-doc (Layer 2 policy)

**Purpose:** Defines the role matrix, step-up requirements, signed URL generation contract, audit obligations, and revocation procedure for access to sealed archive bundles. This policy governs all reads of `archive.archive_packages` and `archive.archive_manifests` and all downloads of sealed ZIP bundles from Object Storage. Implementation binds to this document; any deviation requires a `Docs/decisions_log.md` amendment.

---

## Role matrix

| Role | Download bundle | Verify bundle | View manifest | Revoke access grants | Delete bundle |
|---|---|---|---|---|---|
| `OWNER` | Yes | Yes | Yes | No | No |
| `ACCOUNTANT` | Yes | Yes | Yes | No | No |
| `ADMIN` | Yes | Yes | Yes | Yes | No |
| `VIEWER` | No | No | Yes | No | No |
| `BOOKKEEPER` | No | Yes | Yes | No | No |

**No role has DELETE access.** Object Lock (COMPLIANCE mode, Cyprus 6-year retention) prohibits deletion regardless of role. The `retention_engine` database role is the only principal authorized to delete archive objects after the retention window expires, and it does not map to any user-facing role.

Role evaluation uses `auth.role_on_business(business_id)` per Block 02 Phase 04. The archive access check runs after mobile policy and after tenant isolation; it is the third gate in the authorization chain.

---

## Step-up requirement

Downloading a sealed archive bundle requires a valid step-up MFA challenge. The step-up token must be:

1. Present in the download request (passed as a header or in the signed URL generation call).
2. Valid at download time — within the validity window defined in `step_up_validity_window_policy.md`.
3. Not already consumed for a different action (`consumed_for_action_id` must be null or must match this download request).

The step-up token is consumed by the download. A second download of the same bundle requires a new step-up challenge.

Step-up tokens use `gen_random_uuid()` (UUID v4) per `data_layer_conventions_policy` — these are short-lived unpredictable security tokens where the time-ordered UUID v7 prefix would be information leakage.

The step-up challenge UI follows the pattern defined in `step_up_ui_spec.md`. The archive-specific rules (which roles require step-up, which actions qualify) are defined in `archive_step_up_policy.md`.

## Signed URL generation

Signed URLs are generated by `archive.generate_download_url`. The call is authorized only after step-up verification.

**Tool:** `archive.generate_download_url`
**Side-effect class:** `WRITES_ARCHIVE | WRITES_AUDIT | EXTERNAL_CALL`

**Call signature:**

```ts
archive.generate_download_url({
  bundle_id: uuid,          // archive.archive_packages.id
  step_up_token_id: uuid,   // the consumed step-up token
})
→ {
  signed_url: string,       // pre-signed Object Storage URL
  expires_at: string,       // ISO 8601 — 15 minutes from generation
  download_id: uuid,        // opaque identifier for audit correlation
}
```

**URL lifetime:** 15 minutes from generation time.

**Single-use:** The signed URL is invalidated after its first download. If the download fails after the URL is served (e.g., the user's connection drops), a new step-up challenge is required to generate a replacement URL. This prevents URL sharing or replay.

**Single-use enforcement:** The Object Storage bucket is configured to track download state per pre-signed URL. Alternatively, a `archive_download_tokens` table row is marked `consumed = true` on first use and the download middleware rejects the URL if `consumed = true`. The implementation approach is deferred to Block 15 Phase 05; this policy commits to the single-use property.

## Audit event

Every completed bundle download emits:

**Event:** `ARCHIVE_BUNDLE_DOWNLOADED` (LOW)
**Domain:** `ARCHIVE` per `audit_log_policies`
**Payload:**

| Field | Type | Description |
|---|---|---|
| `user_id` | uuid | The user who initiated the download |
| `bundle_id` | uuid | `archive.archive_packages.id` |
| `workflow_run_id` | uuid | The finalized run the bundle belongs to |
| `ip_address` | text | SHA-256 hex of the raw IP address string — stored hashed per GDPR data minimisation; the original IP is not retained |
| `download_id` | uuid | Correlation ID from `archive.generate_download_url` response |
| `step_up_token_id` | uuid | The step-up token consumed for this download |

Severity: LOW — a download by an authorized role is the expected outcome. A download by an unexpected role or an anomalous volume of downloads triggers a separate `SECURITY_ALERT` via the Block 05 alerting rules.

## Manifest-only access

Viewing the manifest (without downloading the bundle) does not require step-up. Role check (`VIEWER` and above) is sufficient. `ARCHIVE_DATA_READ_SESSION_SUMMARY` aggregates per-session manifest reads per the Block 15 scan amendment.

## Access revocation

If a security incident is suspected — credential compromise, unauthorized access, or anomalous download volume detected — an `ADMIN` can revoke all outstanding signed URLs for a business.

**Tool:** `archive.revoke_all_urls`
**Authorized roles:** `ADMIN` only
**Side-effect:** Marks all active `archive_download_tokens` rows for the `business_id` as `revoked = true`. Any in-flight download using a revoked URL receives an HTTP 403 response from the Object Storage middleware.

Object Lock is not affected by revocation. Revocation operates on application-layer signed URLs only; the underlying objects in Object Storage remain intact and subject to their original retention terms.

Revocation emits `SECURITY_ALERT` with `alert_type = ARCHIVE_URL_REVOCATION` and `business_id`. A follow-on investigation using the `tamper_detection_forensic_runbook.md` procedure is recommended after any revocation event.

## RLS implementation

Archive tables use dedicated RLS policies in the `archive` Postgres schema. The policies enforce both role-level access and the download step-up requirement:

```sql
-- Example: archive.archive_packages read policy
CREATE POLICY archive_packages_read
  ON archive.archive_packages
  FOR SELECT
  USING (
    business_id = current_setting('app.current_business_id')::uuid
    AND auth.role_on_business(business_id) IN ('OWNER', 'ADMIN', 'ACCOUNTANT', 'BOOKKEEPER', 'VIEWER')
  );

-- No INSERT / UPDATE / DELETE policy for application roles
-- Writes are only permitted under the archive_writer role (Block 15 lock sequence)
```

The `archive_reader` role is granted `SELECT` on all `archive.*` tables. It cannot write. Application sessions assume `archive_reader` for read operations; the `archive_writer` role is assumed only during the lock sequence.

There is no cross-business read path. The `current_setting('app.current_business_id')` binding ensures a session scoped to business A cannot read business B's archive rows, even if both businesses share the same Postgres cluster.

## Audit log access rules

Per `audit_log_policies` Section 2, the `ARCHIVE` domain events are visible to roles with the operational surface (`ACCOUNTANT`, `OWNER`, `ADMIN`). `VIEWER` and `BOOKKEEPER` see a subset:

- `VIEWER`: sees `ARCHIVE_PROMOTION_COMPLETED` and `ARCHIVE_VERIFIED` only (high-level lifecycle events; no per-download detail).
- `BOOKKEEPER`: sees all `ARCHIVE` domain events except those with PII in the payload (the `ip_address` field is hashed, so `ARCHIVE_BUNDLE_DOWNLOADED` is visible but the raw IP is not recoverable).

The `ip_address` in `ARCHIVE_BUNDLE_DOWNLOADED` is stored as a SHA-256 hex hash of the raw IP string. The hash is one-way; the original IP cannot be recovered from the audit log. Operators requiring the original IP for a forensic investigation must use the Object Storage access logs (retained separately per `storage_bucket_configuration.md`).
