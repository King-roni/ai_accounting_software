# Session Lifetime Policy

**Category:** Policies · **Owning block:** 02 — Tenancy & Access · **Stage:** 4 sub-doc (Layer 2)

This policy defines session expiry rules, concurrency limits, and termination semantics for all user sessions. It is binding for the session creation handler, the session validation middleware, and the session expiry background job. Step-up sessions are not governed here — they are a separate token type covered by `step_up_validity_window_policy`.

---

## Purpose

Session tokens are the primary credential for authenticated API calls. Bounding their lifetime limits the window of exposure for a compromised token. Bounding concurrency limits the number of simultaneously active sessions that could be exfiltrated. Both constraints are user-visible UX considerations; the values chosen here balance security with practical multi-device use.

---

## Session token

Session tokens are UUID v4 generated by `gen_random_uuid()`. UUID v4 is required here — the session token is a security-sensitive value where temporal ordering (UUID v7's property) would leak the approximate session creation time to any party that can observe the token. See `data_layer_conventions_policy` for the full v4 exception list.

The token is stored as a SHA-256 hash in `sessions.token_hash`. The plaintext token is returned to the client at session creation only and never stored anywhere in the system. The audit log records `session_id` (the UUID v7 PK of the `sessions` row), never the token itself.

---

## Idle timeout

**30 minutes of inactivity.**

Activity is defined as any authenticated API call that passes session validation. Calls that fail authentication (including calls made after expiry) do not extend the session.

Implementation: each session validation in the middleware updates `sessions.last_active_at = now()`. The expiry check is:

```sql
last_active_at < now() - interval '30 minutes'
```

This check runs on every session validation, not in a background job. A session that would have expired is expired in-band during the next request that presents its token. The request that triggers expiry receives a 401 with `error: "SESSION_EXPIRED"`.

The background job `job_session_expiry_gc` runs every 5 minutes and hard-deletes rows where `last_active_at < now() - interval '35 minutes'` (5-minute grace window for in-flight requests). Deletion is physical — sessions table rows are not soft-deleted. The audit event `SESSION_TERMINATED` with `reason: IDLE_TIMEOUT` is emitted before deletion.

---

## Absolute timeout

Session lifetime is capped regardless of activity:

| Role | Absolute timeout |
| --- | --- |
| Standard (Owner, Admin, Bookkeeper, Reviewer, Read-only) | 8 hours from `sessions.created_at` |
| Accountant | 12 hours from `sessions.created_at` |

The accountant role receives a longer window because accountants commonly work across extended multi-hour sessions reviewing and reconciling periods, and their access is constrained to read-heavy surfaces (see `permission_matrix.md`).

The absolute timeout check:

```sql
-- Standard roles
created_at < now() - interval '8 hours'

-- Accountant role
created_at < now() - interval '12 hours'
```

The session middleware evaluates both the idle timeout and the absolute timeout on every request. The shorter of the two conditions takes effect first.

Absolute expiry triggers `SESSION_TERMINATED` with `reason: ABSOLUTE_TIMEOUT`.

---

## Step-up sessions

Step-up tokens are a separate token type from session tokens. They represent a temporary elevation of authorization for a specific action (e.g., viewing an encrypted field, approving a finalization). Step-up tokens are generated by `gen_random_uuid()` and stored in `step_up_tokens`.

Step-up token validity is governed by `step_up_validity_window_policy`. The validity window for a step-up token is independent of the session lifetime — a step-up token can expire while the session is still active, or a session can expire while a step-up token is still technically within its window. In both cases, the more restrictive condition takes effect: if the session is expired, step-up tokens for that session are also treated as expired regardless of their own TTL.

---

## Multi-device concurrency

**Maximum 5 concurrent active sessions per user, across all businesses and devices.**

"Active" means a session row where `is_revoked = false` AND the session has not been expired by idle or absolute timeout.

When a 6th session is created (a new login on a 6th device or browser):

1. The session with the oldest `last_active_at` among the current 5 active sessions is identified.
2. That session's `is_revoked` is set to `true` and `revoked_reason = 'MAX_CONCURRENCY_EVICTION'`.
3. `SESSION_EVICTED_MAX_CONCURRENCY` is emitted.
4. The new session is created and the new token is returned.

Eviction happens atomically within the same transaction as the new session creation. If the eviction write fails, the new session creation is also rolled back and the client receives an error.

The evicted user receives no real-time push notification. If the evicted device presents its token again, it receives a 401 with `error: "SESSION_REVOKED"` and must re-authenticate.

### Audit event: `SESSION_EVICTED_MAX_CONCURRENCY`

**Severity:** MEDIUM

MEDIUM because unexpected session eviction on another device is a meaningful security signal — it indicates the user has exceeded concurrent session limits, which may or may not be expected.

**Payload:**

| Field | Type | Description |
| --- | --- | --- |
| `evicted_session_id` | uuid | The `sessions.id` of the session that was evicted |
| `user_id` | uuid | The user whose session was evicted |
| `new_session_id` | uuid | The `sessions.id` of the newly created session |
| `eviction_reason` | text | Always `MAX_CONCURRENCY_EVICTION` |
| `session_count_before` | integer | Always 5 at eviction time |
| `last_active_at_evicted` | timestamptz | The `last_active_at` of the evicted session |

## Session termination

Sessions are terminated by three mechanisms:

| Mechanism | `revoked_reason` value | Triggered by |
| --- | --- | --- |
| Explicit logout | `USER_LOGOUT` | User action via the logout endpoint |
| Idle timeout expiry | `IDLE_TIMEOUT` | Middleware check on next request, or GC job |
| Absolute timeout expiry | `ABSOLUTE_TIMEOUT` | Middleware check on next request, or GC job |
| Concurrency eviction | `MAX_CONCURRENCY_EVICTION` | 6th concurrent session creation |
| Admin-forced revocation | `ADMIN_FORCED_REVOCATION` | Platform admin or Owner-initiated forced logout |

### Audit event: `SESSION_TERMINATED`

**Severity:** LOW for `USER_LOGOUT`, `IDLE_TIMEOUT`, `ABSOLUTE_TIMEOUT`; MEDIUM for `MAX_CONCURRENCY_EVICTION` and `ADMIN_FORCED_REVOCATION`.

**Payload:**

| Field | Type | Description |
| --- | --- | --- |
| `session_id` | uuid | The `sessions.id` being terminated |
| `user_id` | uuid | The user whose session is terminated |
| `business_id` | uuid (nullable) | The business context of the session; null for org-level sessions |
| `reason` | text | One of the `revoked_reason` values above |
| `terminated_by_user_id` | uuid (nullable) | Present for `ADMIN_FORCED_REVOCATION`; the admin who initiated the revocation |
| `session_age_seconds` | integer | `now() - sessions.created_at` in seconds |

The session token is never included in the payload. The `session_id` (the UUID v7 primary key of the `sessions` row) is the traceable identifier for forensic queries.

---

## Implementation constraints

- Session token plaintext must not appear in application logs, error messages, or API responses after initial issuance.
- The `sessions` table carries RLS: a user can only SELECT their own sessions; admin roles can SELECT sessions for their business.
- `SESSION_TERMINATED` emission happens before physical deletion or `is_revoked` write, inside the same short transaction as the termination. If the audit write fails, the termination write is not retried silently — the GC job will catch expired sessions on its next pass.
- The session validation middleware must not short-circuit the absolute timeout check for the Accountant role even if idle timeout has not been reached.
