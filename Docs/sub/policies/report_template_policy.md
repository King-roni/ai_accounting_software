# Report Template Policy

**Category:** Policies · **Owning block:** 16 — Dashboard & Reporting · **Stage:** 4 sub-doc (Layer 2)

Rules governing the structure, determinism, and versioning of reports generated by Block 16 tools. A report is a formatted, user-facing document produced from finalized financial data. This policy covers period reports, accountant packs, and VIES XML exports — any output produced by the `report.*` toolset that is intended for external consumption or long-term archival.

---

## 1. Determinism requirement

All reports generated by Block 16 tools must be **deterministic**: given identical inputs, two separate invocations produce byte-for-byte identical output. This is a binding architectural requirement with three justifications:

1. **Regulatory reproducibility** — an accountant or regulator must be able to request the same report twice and receive the same document. Any difference between two instances of the same report for the same period raises an integrity question.
2. **Tamper detection** — the report job stores a hash of each generated output. If a re-generation for the same period produces a different hash, that is evidence of data mutation or template drift.
3. **Archive consistency** — reports embedded in accountant packs or archive bundles must match the independently generated version for the same inputs.

Determinism is not a best-effort property. A report generator that is non-deterministic on any code path is a code-review-blocking violation.

---

## 2. Determinism mechanism

Determinism is achieved by the conjunction of three invariants:

### Invariant A — Fixed input snapshot

All data consumed by a report tool is sourced from the finalized `analytics_snapshots` row for the period (per `analytics_snapshot_schema`), not from live queries against `transactions`, `ledger_entries`, or `invoices`. For a `FINALIZED` period, the snapshot is immutable (`is_current = true` row does not change after finalization). A report for a finalized period reads from a fixed, immutable data source.

Reports for non-finalized periods are not supported in MVP. Any request to generate a period report for a period without a finalized snapshot returns an error.

### Invariant B — Fixed template version

Each report is generated using a specific, versioned template. The template version used is stored in `report_jobs.parameters_json` at the time the job is queued. The `report.generate_period_report` tool resolves the template version from `parameters_json` before rendering; it never resolves "latest template at render time". A report job queued with template version `1.3` will always produce output against template `1.3`, even if template `1.4` has been deployed by the time the job executes.

Template version `1.0` is the MVP baseline. All templates are stored in the `report-templates` bucket keyed by `{template_name}/{version}/`.

### Invariant C — No locale-dependent formatting beyond the configured timezone

Report output uses deterministic number formatting: currency amounts are serialized as decimal strings with exactly 2 decimal places and no thousands separator (e.g., `"12345.67"`). Dates are formatted as `YYYY-MM-DD`. Timestamps are formatted as ISO 8601 in the business's configured timezone — a single, stable configuration field, not the requesting user's browser locale or OS locale. No locale library is invoked with dynamic locale detection. This ensures identical output on any server, in any region.

## 3. Template versioning

A template version bump is required when any of the following changes:

- The rendered output format or layout changes (including visual rearrangement of sections).
- The set of data fields rendered in the report changes (fields added or removed).
- The formula used to compute any derived figure in the report changes.
- The number formatting or date formatting convention changes.

A template version bump **requires a `Docs/decisions_log.md` amendment** documenting the rationale, the old and new version numbers, and the backwards-compatibility implications (e.g., whether existing queued jobs should be completed with the old version or re-queued against the new version).

A version bump does not invalidate previously generated reports for the same period — those are stored in `export-temp` keyed by their job ID and are accessible until their TTL expires.

### Version format

Template versions follow `major.minor`:

- **Major bump** — layout or formula change that produces visually different output for the same input data.
- **Minor bump** — cosmetic fixes, label corrections, or performance optimisations that produce output indistinguishable to a human reader.

Both major and minor bumps require the `decisions_log.md` amendment. The distinction is informational for operators assessing the impact of template changes.

---

## 4. Template version in output

Every generated report embeds the template version in two places:

1. **Footer of the rendered document** — visible to human readers; formatted as e.g. `Report version 1.3 · Generated by [Platform Name]`.
2. **`report_jobs.parameters_json`** — the `template_version` key records the version used to generate the report. This is the machine-readable record consumed by re-generation, comparison, and forensic audits.

---

## 5. Tool side-effect classification

The tools covered by this policy are classified as follows:

| Tool | Side-effect class | Notes |
|---|---|---|
| `report.generate_period_report` | `READ_ONLY` | Reads from `analytics_snapshots`; writes only to `export-temp` bucket (transient storage zone, not an operational-DB table) |
| `report.generate_accountant_pack` | `READ_ONLY` | Same reasoning; also reads from `archive-bundles` |
| `report.generate_vies_xml` | `READ_ONLY` | Reads from `vies_submissions` and `analytics_snapshots`; writes XML to `export-temp` |

All three tools carry `WRITES_AUDIT` in addition to the above because every report generation is audit-logged via `emitAudit()`.

---

## 6. Asynchronous execution

Reports are generated asynchronously via the `report_job_schema` queue. A client requests a report by calling `report.queue_report_job`, which writes a `report_jobs` row and returns a `job_id`. A worker picks up the job and calls the relevant generator tool. The client polls or subscribes for job completion. This pattern decouples the client request from the potentially latency-variable rendering step.

The `report.queue_report_job` tool resolves the current template version at queue time and stores it in `parameters_json`. The generator tool receives the fully resolved parameters and does not perform any dynamic version resolution.

---

## 7. FINALIZED period report sourcing

For a `FINALIZED` period, the report sources its financial data exclusively from the archived snapshot, not from a live recompute of `locked_ledger_entries`. The snapshot was computed at finalization time against `locked_ledger_entries`; re-querying those entries would be equivalent for a correctly immutable archive, but the snapshot is faster and avoids any risk of reading entries modified by a subsequent adjustment run before the snapshot was superseded.

If the period has been re-finalized via an adjustment run, the report uses the `is_current = true` snapshot for the period (the latest snapshot version), unless `parameters_json` explicitly specifies a `snapshot_version` — which is supported for forensic and comparison use cases.

---

## 8. Mobile rejection

`report.queue_report_job` (the client-facing write that queues a report) is listed in `mobile_write_rejection_endpoints.md`. Mobile clients cannot queue new report generation jobs. Previously generated reports can be downloaded on mobile via the read-only signed-URL endpoint.

---

## 9. Audit events

Report generation events are in the `REPORT` domain of `audit_event_taxonomy`:

| Event | Severity | When |
|---|---|---|
| `REPORT_JOB_QUEUED` | LOW | `report.queue_report_job` writes a new `report_jobs` row |
| `REPORT_JOB_COMPLETED` | LOW | Report job transitions to `COMPLETED`; output is accessible |
| `REPORT_JOB_FAILED` | MEDIUM | Report job transitions to `FAILED`; user action required |

All events are emitted on the business-scoped hash chain per `audit_log_policies`.
