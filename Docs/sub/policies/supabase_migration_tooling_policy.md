# Supabase Migration Tooling Policy

**Category:** Policies · **Owning block:** 02 — Tenancy & Access · **Stage:** 4 sub-doc (Layer 2)

Single binding reference for how database migrations are authored, reviewed, applied, and rolled back in this project. Every developer creating schema changes, every CI job gating a merge, and every operator executing a production push binds to the rules in this document.

Migration tooling is Supabase CLI only. No Prisma Migrate, no raw `psql` scripts pushed outside this flow, no ad-hoc `ALTER TABLE` in a production console.

---

## 1. Tooling baseline

**Supabase CLI commands (binding):**

| Operation | Command |
| --- | --- |
| Create a new migration file | `supabase migration new <descriptive_name>` |
| Apply to local dev | `supabase db reset` (full replay) or `supabase db push --local` |
| Apply to staging branch | `supabase db push --linked` (staging project linked) |
| Apply to production | `supabase db push --linked` (production project linked, after staging passes CI) |
| Diff the local schema against a live branch | `supabase db diff` |

No other mechanism may be used to apply schema changes. Platform console DDL edits are forbidden.

## 2. Migration file rules

### 2.1 Filename format

Every migration file is a `.sql` file whose name is generated by `supabase migration new`. The CLI prefix is a 14-digit UTC timestamp (`YYYYMMDDHHMMSS`), followed by an underscore and a descriptive slug in `snake_case`.

```
20260512143000_add_business_memberships_capacity_columns.sql
```

The CLI-generated timestamp is the canonical name. Renaming the timestamp prefix is forbidden; it breaks the sequential replay order.

### 2.2 One file, one logical change

Each migration file contains the complete DDL for one logical change. Mixing unrelated changes in one file is forbidden. If a feature requires multiple independent schema changes, they are separate migration files.

### 2.3 Anonymous migrations are forbidden

Every migration file must begin with a comment block identifying the ticket reference and the owning block:

```sql
-- Migration: 20260512143000_add_business_memberships_capacity_columns
-- Ticket: PROJECT-1234
-- Block: 02 — Tenancy & Access
-- Author: <engineer handle>
-- Description: Adds soft-limit warning threshold columns to business_memberships.
```

A migration file without this header block fails the migration lint check.

### 2.4 Table creation completeness

Every migration that creates a new table must also include, in the same file:

1. **RLS enablement:** `ALTER TABLE <table> ENABLE ROW LEVEL SECURITY;`
2. **At least one RLS policy** using the helper functions from `rls_helper_functions` — a `SELECT` policy at minimum.
3. **All indexes** required by the table's known query patterns (as declared in the relevant Schema sub-doc).

A `CREATE TABLE` statement without accompanying RLS and indexes in the same migration file fails the migration lint check.

### 2.5 Migration lint checks (CI-enforced)

The CI pipeline runs `supabase db lint` plus a project-specific lint script before allowing a PR to merge. The following conditions block the merge:

| Lint rule | Violation |
| --- | --- |
| Missing header comment block | Merge blocked |
| `DROP TABLE` without a ticket reference in the same file's header | Merge blocked |
| `ALTER COLUMN` type change without an adjacent `-- BACKFILL NOTE:` comment | Merge blocked |
| `ADD COLUMN … NOT NULL` without a `DEFAULT` clause | Merge blocked |
| `CREATE TABLE` without `ENABLE ROW LEVEL SECURITY` in the same file | Merge blocked |
| `CREATE TABLE` without at least one index | Merge blocked |
| Schema version mismatch (out-of-order file timestamp vs. `schema_migrations` tracking table) | Merge blocked |

## 3. Staging-first application order

The application sequence is fixed:

```
Local dev → Staging branch → Production
```

No migration may be applied to production before it has been applied to the staging branch and the staging CI suite has passed. The staging CI suite includes:

1. Full migration replay on a clean Supabase staging project.
2. The RLS invariant test suite (Block 02 Phase 10).
3. Schema lint.
4. Application-layer integration tests that exercise the migrated surface.

Production push is gated by a manual approval step in CI for migrations that carry `DROP TABLE`, `DROP COLUMN`, or `ALTER COLUMN … TYPE` statements.

---

## 4. Rollback scripts for destructive changes

A destructive migration is any file containing one or more of:

- `DROP TABLE`
- `DROP COLUMN`
- `ALTER COLUMN … TYPE` (type coercion)
- `DROP INDEX` (on a column flagged in a Schema sub-doc as load-bearing)
- `TRUNCATE`

For every destructive migration, a companion rollback script must exist in `supabase/migrations/rollbacks/` with the same filename prefix:

```
supabase/migrations/rollbacks/20260512143000_add_business_memberships_capacity_columns_rollback.sql
```

The rollback script must reverse the destructive change such that applying it after the migration restores the prior state. The rollback script is tested in CI on the staging branch before the forward migration is promoted to production.

Rollback scripts are not applied automatically; they require a manual operator action. Their existence is the insurance policy.

## 5. Naming conventions for migration objects

All database object names created in migrations follow `snake_case` and the prefix/suffix conventions defined in `data_layer_conventions_policy`. Table names are plural nouns (`business_memberships`, not `business_membership`). Index names use the pattern `idx_<table>_<column(s)>`. Constraint names use the pattern `<table>_<column(s)>_<type>` (e.g., `business_memberships_user_id_fk`).

---

## 6. Mobile write surface note

Mobile clients are rejected at all write surfaces in this project. This policy governs schema-layer changes only; the enforcement mechanism for rejecting mobile writes is documented in `mobile_write_rejection_endpoints`. Migrations that add new write endpoints must not add mobile-accessible paths without a corresponding `mobile_write_rejection_endpoints` amendment.

---

## 7. Audit events

Migrations themselves are infrastructure operations and do not emit runtime audit events. Ops-level migration activity is tracked via the Supabase platform audit log (project-level, not the application's `audit_events` table). Schema-level changes that affect tenant data boundaries must be recorded in `Docs/decisions_log.md` at time of application.
