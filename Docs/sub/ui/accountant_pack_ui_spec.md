# Accountant Pack — UI Specification

**Category:** UI · **Owning block:** 16 — Dashboard & Reporting · **Stage:** 4 sub-doc (Layer 2)

**Block reference:** Block 16, Phase 06 (accountant pack configuration and scheduled delivery).

**Purpose:** Specifies the accountant pack configuration panel, preview behaviour, delivery status tracking, manual trigger, and mobile constraints. The accountant pack is the primary mechanism for periodic export delivery to external accountants and must be configured accurately to avoid delivery failures or premature sends. This spec is binding for the frontend implementation.

---

## What the accountant pack is

The accountant pack is a scheduled bundle of pre-defined exports sent to a recipient email address at a configured cadence. It is distinct from ad-hoc exports (see `export_pipeline_ui_spec.md`) in that delivery is automated, time-triggered or finalization-triggered, and the bundle is a ZIP containing multiple formats selected by the business OWNER.

The pack is generated by `report.generate_accountant_pack` and delivered by `out_workflow.send_accountant_pack` (email delivery via the platform email pipeline). The underlying job is tracked in `report_jobs` with `report_type = 'ACCOUNTANT_PACK'`.

Each delivery attempt is logged with an audit event: `ACCOUNTANT_PACK_SENT` (success) or `ACCOUNTANT_PACK_DELIVERY_FAILED` (failure).

## Pack configuration panel

**Location:** Settings → Accountant Pack.

**Access:** OWNER role required to edit. ADMIN can view the configuration but cannot save changes. ACCOUNTANT and below see only the delivery history table (read-only).

**Fields:**

**Recipient email**

Pre-filled from the business contact marked as accountant in the business profile. The field is editable; changes take effect on the next delivery. The email is validated on save (format check; no live MX lookup). Changing the recipient email emits `ACCOUNTANT_PACK_CONFIG_UPDATED` with `changed_field = 'recipient_email'`.

Only one recipient address is supported per business in the current configuration schema. Multiple-recipient support is deferred to Stage 2+; the field does not accept comma-separated values and will show a validation error if attempted.

**Delivery cadence**

Single-select with three options:

| Option | Label | Trigger condition |
|---|---|---|
| `MONTHLY` | Monthly (after each finalized month) | Fired when `ARCHIVE_PROMOTION_COMPLETED` is emitted for a monthly run |
| `QUARTERLY` | Quarterly (after Q1, Q2, Q3, Q4 finalization) | Fired when `ARCHIVE_PROMOTION_COMPLETED` is emitted for a run whose `period_end` falls on a quarter boundary |
| `ON_FINALIZATION` | On every finalization | Fired on every `ARCHIVE_PROMOTION_COMPLETED` regardless of period type |

The delivery job checks for a finalized period before dispatching. If the selected cadence window contains no finalized period, the delivery is `SKIPPED` (see Delivery history below).

**Included formats**

Multi-select from the formats in `export_definitions_catalog.md`. Maximum 6 formats per pack. The selector displays all 13 formats grouped by category (Accounting, Tax, Bank, Archive). Formats for which the OWNER's role does not meet `min_role` are excluded from the selector — they are not shown, not greyed out.

The `archive_bundle_zip` format, if selected, includes the Object-Locked archive bundle. Note: `archive_bundle_zip` has permanent retention but its inclusion in the accountant pack ZIP does NOT change the pack ZIP's own 24-hour TTL — the pack ZIP is a transient delivery vehicle, not a permanent archive.

The selector enforces the 6-format maximum client-side and server-side. Attempting to save with > 6 formats returns a validation error: "Select a maximum of 6 formats."

**Password protection**

Toggle: off (default) / on. When enabled:

- The system generates a random 16-character alphanumeric password on save.
- The password is stored encrypted in `accountant_pack_config` under the field `zip_password_enc`.
- The password is displayed to the OWNER once immediately after save, in a modal with a "Copy password" button. It is not retrievable after the modal is dismissed. If the OWNER loses the password, they must reset it by toggling password protection off and back on, which generates a new password.
- The password is included in the delivery email body alongside the ZIP attachment, formatted as: "Archive password: `<password>`". This is intentional — the pack is sent to the accountant who needs to open it immediately.
- Changing the password protection setting emits `ACCOUNTANT_PACK_CONFIG_UPDATED`.

**Save button**

On save, `accountant_pack_config` is updated and `ACCOUNTANT_PACK_CONFIG_UPDATED` is emitted. The form shows a success confirmation inline ("Settings saved"). No toast is shown for configuration saves (only for delivery actions).

## Preview — dry-run pack generation

The "Preview pack" button is accessible to OWNER and ADMIN roles in the configuration panel.

**Behaviour:**

1. Triggers `report.generate_accountant_pack` with `dry_run = true`.
2. Does NOT send an email.
3. Does NOT create an `accountant_pack_deliveries` row.
4. Uses the most recent finalized period's data. If no finalized period exists, the preview button is disabled with tooltip: "No finalized period available for preview."
5. Returns within 60 seconds for most configurations. Packs including `archive_bundle_zip` may take longer.

**Preview result modal:**

- Title: "Pack preview"
- Content: a list of each included format with its estimated file size (e.g., "Ledger Entries (CSV) — 42 KB") and a total estimated pack size.
- Action: "Close" only. The preview is not downloadable; it is a size/format inspection tool.
- Error: if the dry-run fails for any format, the modal shows the affected format with "Unable to generate — check configuration" and the error class.

**Note on data freshness:** The preview uses data from the last finalized period. It will not reflect in-progress period data. This is by design — the pack should only contain finalized data.

## Delivery history

The delivery history table is displayed below the configuration panel for OWNER and ADMIN roles, and as the sole visible element for ACCOUNTANT and below.

The table shows the last 12 delivery attempts with the following columns:

| Column | Content |
|---|---|
| Period | The finalized period covered by the pack (e.g., "January 2026") |
| Sent at | UTC timestamp, displayed in local timezone |
| Recipient | The email address the pack was sent to at time of delivery |
| Status | `DELIVERED` / `FAILED` / `SKIPPED` |
| Detail | Error message (FAILED) or skip reason (SKIPPED); empty for DELIVERED |

**Status values:**

- `DELIVERED` — `ACCOUNTANT_PACK_SENT` emitted; email accepted by the delivery provider.
- `FAILED` — `ACCOUNTANT_PACK_DELIVERY_FAILED` emitted; the email was not delivered. Common causes: invalid recipient address, delivery provider bounce, or pack generation failure. The Detail column shows the error class.
- `SKIPPED` — No finalized period existed within the cadence window. For example, a QUARTERLY pack fires after Q1 ends; if Q1 has no finalized run, the delivery is `SKIPPED` and logged. The Detail column shows "No finalized period in window."

**Retry for FAILED deliveries:** An OWNER can retry a failed delivery via the "Retry" button that appears on FAILED rows. This re-queues the pack generation and delivery with the same parameters as the original attempt. Retrying does NOT create a new cadence-window entry; it is logged as a retry against the original delivery attempt.

The table is read-only. No bulk actions are available. Delivery history is not exportable from this panel (the audit log export covers the underlying `ACCOUNTANT_PACK_*` events for compliance purposes).

## Manual trigger — "Send now"

The "Send now" button is visible to OWNER role only. It is not shown for ADMIN or below.

**Behaviour:**

1. Triggers an immediate pack delivery outside the scheduled cadence.
2. Requires step-up MFA confirmation (same pattern as finalization). The step-up challenge appears in a modal before dispatch.
3. Calls `report.generate_accountant_pack` with `cadence_override = true` and the current `accountant_pack_config`.
4. A `report_jobs` row is created with `report_type = 'ACCOUNTANT_PACK'` and `metadata.trigger = 'MANUAL'`.
5. On completion, `ACCOUNTANT_PACK_SENT` is emitted and a delivery history row is added with `trigger_kind = 'MANUAL'`.

**Constraints:**

- If a pack generation or delivery job is already `RUNNING` for the business, "Send now" is disabled with tooltip: "A delivery is already in progress."
- Manual sends count against the delivery history table (they appear as a row with Period set to the most recent finalized period and a "Manual" label in the detail column).
- If no finalized period exists, "Send now" is disabled with tooltip: "Finalize a period before sending."

## Mobile behaviour

Mobile clients are identified by `client_form_factor = MOBILE`.

**Pack configuration panel on mobile:** Read-only. All fields are displayed but no input is editable. The "Save" button is not shown. The "Preview pack" button is not shown.

**"Send now" on mobile:** Blocked. The button is not rendered on mobile clients. If the endpoint is called directly (e.g., via API client), it is rejected with `MOBILE_WRITE_REJECTED` per `mobile_write_rejection_endpoints.md`.

**Delivery history on mobile:** Fully accessible. The table is displayed in a scrollable list format on narrow viewports.

The mobile write rejection for "Send now" is consistent with the platform-wide policy: write surfaces that trigger delivery (an external side effect) are blocked on mobile. The pack configuration editor follows the same policy.
