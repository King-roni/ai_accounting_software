## DDL

```sql
CREATE TYPE match_status_enum AS ENUM (
    'PROPOSED',
    'CONFIRMED',
    'REJECTED',
    'AUTO_CONFIRMED',
    'SUPERSEDED',
    'EXCEPTION_DOCUMENTED'
);
```

## Value Definitions

### PROPOSED

A match candidate has been generated by `tool_match_propose` but has not yet been reviewed or actioned by a human or the auto-confirm engine. This is the initial status for all match rows inserted by the matching engine.

A PROPOSED match does not affect ledger postings or run phase gates. It exists only in the `income_matches` table waiting for action.

### CONFIRMED

The match has been manually confirmed by an accountant via `tool_match_confirm`. A CONFIRMED match authorises the corresponding ledger posting to proceed. CONFIRMED is a stable status; a match in CONFIRMED status can only be displaced by a SUPERSEDED transition if a new match replaces it.

### REJECTED

The match has been manually rejected by an accountant. A REJECTED match signals that the proposed pairing between a transaction and an income record was incorrect. Rejection triggers the matching engine to re-propose for the same transaction (if additional candidates exist) or to create a `MATCHING_NO_MATCH` review queue issue.

REJECTED is a terminal status for the individual match row. The same pairing cannot be re-proposed; a new `income_matches` row must be inserted if a retry is warranted.

### AUTO_CONFIRMED

The match was automatically confirmed by the engine without human review, based on the match level reaching `EXACT` (all scoring dimensions at maximum confidence). AUTO_CONFIRMED carries the same ledger-authorisation weight as CONFIRMED.

AUTO_CONFIRMED matches are eligible for post-hoc review by accountants. An accountant who disagrees with an AUTO_CONFIRMED match can reject it, which transitions the status to REJECTED and creates a new PROPOSED match if an alternative candidate exists.

### SUPERSEDED

A previously CONFIRMED or AUTO_CONFIRMED match has been replaced by a newer match for the same transaction. SUPERSEDED occurs when:
- An accountant confirms a different match candidate for the same transaction.
- The engine proposes and auto-confirms a new EXACT match after an earlier match was confirmed with lower confidence.

SUPERSEDED is a terminal status. The superseded match row is retained for audit trail purposes.

### EXCEPTION_DOCUMENTED

The match has known anomalies (e.g., amount discrepancy within tolerance, date mismatch within acceptable window) that were explicitly acknowledged by an accountant. The accountant provided a documented exception reason. EXCEPTION_DOCUMENTED carries the same ledger-authorisation weight as CONFIRMED.

EXCEPTION_DOCUMENTED is a terminal status. It cannot be transitioned to any other status except SUPERSEDED if a corrective match is later confirmed.

## Transition Rules

| From | To | Actor | Condition |
|---|---|---|---|
| PROPOSED | CONFIRMED | Accountant (ACCOUNTANT, ADMIN, OWNER role) | Manual confirm action via `tool_match_confirm` |
| PROPOSED | REJECTED | Accountant (ACCOUNTANT, ADMIN, OWNER role) | Manual reject action |
| PROPOSED | AUTO_CONFIRMED | Engine (system) | Match level is EXACT; auto-confirm policy enabled for business |
| CONFIRMED | SUPERSEDED | System (on new confirm for same transaction) | New match confirmed for the same transaction |
| AUTO_CONFIRMED | SUPERSEDED | System (on new confirm for same transaction) | New match confirmed for the same transaction |
| AUTO_CONFIRMED | REJECTED | Accountant (ACCOUNTANT, ADMIN, OWNER role) | Manual override of auto-confirmed match |
| CONFIRMED | EXCEPTION_DOCUMENTED | Accountant (ADMIN, OWNER role) | Exception documented with reason text |
| EXCEPTION_DOCUMENTED | SUPERSEDED | System (on corrective match confirm) | Corrective match confirmed by accountant |

## Audit Events on Status Change

| Status change | Audit event | Severity |
|---|---|---|
| PROPOSED → CONFIRMED | `MATCHING_MATCH_CONFIRMED` | LOW |
| PROPOSED → REJECTED | `MATCHING_MATCH_REJECTED` | LOW |
| PROPOSED → AUTO_CONFIRMED | `MATCHING_MATCH_AUTO_CONFIRMED` | LOW |
| Any → SUPERSEDED | `MATCHING_MATCH_SUPERSEDED` | LOW |
| CONFIRMED/AUTO_CONFIRMED → REJECTED | `MATCHING_MATCH_REJECTED` | LOW |
| CONFIRMED → EXCEPTION_DOCUMENTED | `MATCHING_EXCEPTION_DOCUMENTED` | LOW |

All events carry `match_id`, `transaction_id`, `income_record_id`, `from_status`, `to_status`, `actor_id` (null for engine transitions), and `business_id`.

## Usage in income_matching_schema.md

The `income_matches` table has a `status` column of type `match_status_enum` with a default of `PROPOSED`. All status transitions are performed via the matching tool layer; direct SQL updates to `income_matches.status` from application code outside the tool layer are not permitted.

```sql
-- income_matches table (excerpt)
status  match_status_enum  NOT NULL  DEFAULT 'PROPOSED'
```
