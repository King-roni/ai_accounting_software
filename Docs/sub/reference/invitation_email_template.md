# invitation_email_template

**Category:** Reference data · **Owning block:** 02 — Tenancy & Access · **Co-owner:** 14 — Review Queue (consumes email-delivery) · **Stage:** 4 sub-doc (Layer 2)

The canonical content + structural specification for the invitation email dispatched when an OWNER/ADMIN invites a new user. Consumed by Block 02 Phase 07 invitation flow and rendered by the template runtime documented in `email_delivery_integration.md`.

Adds template #9 to the email-delivery integration's 8-template inventory and adds the `invitation` row to its From-address-by-category table.

---

## 1. What it is

The user-facing artefact that completes the invitation lifecycle. Per BOOK-203 `invitation_token_schema.md`, an `invitation_tokens` row INSERT triggers email dispatch; this doc defines what that email contains.

The template follows `email_delivery_integration.md`'s architecture:

- File path: `email-templates/invitation.html`.
- Token-substitution `{{variable_name}}` engine.
- Shared `base_layout.html` for logo + footer.
- Plain-text fallback auto-generated by stripping HTML.

---

## 2. Subject line

```
{{inviter_display_name}} invited you to {{business_legal_name}} on Cyprus Bookkeeping
```

Variables (resolved at send time):

| Variable | Source |
|---|---|
| `inviter_display_name` | `users.display_name` of the inviter |
| `business_legal_name` | `business_entities.legal_name` |

Subject is plain text. Maximum 100 characters after substitution (long business names truncated mid-line in the email client; plain text in the raw email keeps the full value for filing/searching).

## 4. Body structure

Five sections in order:

### 4.1 Greeting

```
Hi {{invitee_name_or_email}},
```

`invitee_name_or_email` falls back to the email address when no display name is known (the common case — the platform doesn't know the invitee's name pre-acceptance).

### 4.2 Context paragraph

**First-time-invitee path:**

```
{{inviter_display_name}} has invited you to join {{business_legal_name}}
on Cyprus Bookkeeping as a {{role_display_name}}.
```

**Pre-existing-account path** (when `invitee_email` matches an existing `users` row):

```
{{inviter_display_name}} has added you as a {{role_display_name}}
on {{business_legal_name}}, in addition to your existing business memberships.
```

The backend detects the path at email-build time via a SELECT on `users.email = invitee_email`.

### 4.3 Role description block

Per-role text substituted via `{{role_description_html}}`. See §5 for the per-role table.

### 4.4 CTA button

**First-time-invitee path:**

```
[ Accept invitation ]
```

**Pre-existing-account path:**

```
[ View {{business_legal_name}} ]
```

Both link to `https://{{platform-domain}}/invite/{{token_id}}`. The backend redirect at that URL routes pre-existing-account users to the business switcher; first-time invitees go to the new-account flow.

Button styling: primary brand colour via the email-safe design-token output (resolved hex at template-build time, NOT a CSS variable — email clients don't support custom properties). Min hit area 44×44 px per WCAG.

### 4.5 Footer

See §8.

## 5. Per-role description block

Substituted via `{{role_description_html}}` based on `invitation_tokens.assigned_role`. Six descriptions (one per BOOK-179 `permission_matrix` role); writing all six even though BOOK-203's schema currently shows a 4-value enum drift — this template is forward-compatible:

| Role | Description block (target ≤ 200 chars) |
|---|---|
| Owner | "Owners have full access. They can configure the business, manage team members, finalize periods, and access all financial data and reports." |
| Admin | "Admins can run workflows, manage team members, edit settings, and finalize periods. They cannot transfer ownership." |
| Bookkeeper | "Bookkeepers handle day-to-day operations: triggering workflow runs, resolving review queue issues, and applying transaction tags." |
| Accountant | "Accountants review and reconcile data. They generate reports, view ledger entries, and download regulator-grade exports." |
| Reviewer | "Reviewers can read transactional data and inspect the review queue. They cannot make changes to settings or workflows." |
| Read-only | "Read-only users can view dashboards and audit logs only. They have no access to operational or workflow surfaces." |

When the `role_enum` drift surfaced in BOOK-203 is reconciled (current schema shows 4 values: OWNER/ADMIN/ACCOUNTANT/VIEWER; canonical model from BOOK-179 is 6 values), the role-description table above already covers both possibilities.

## 7. Localisation

English-only in MVP per the platform's EN-only commitment. Greek variant (`email-templates/invitation_el.html`) is post-MVP work; same variable set; role-description table localised; subject line + CTA strings translated.

The platform doesn't currently detect the invitee's locale at invitation time (the invitee has no account yet). Stage 2+ may add an explicit `locale` field to the invitation row (defaulting to inviter's locale or business default).

## 8. Footer + security note + unsubscribe handling

Three blocks in order:

### 8.1 Expiry notice

```
This invitation expires {{expires_at_friendly}} (in {{days_until_expiry}} days).
```

| Variable | Derived as |
|---|---|
| `expires_at_friendly` | `invitation_tokens.expires_at` formatted as "12 Jun 2026" |
| `days_until_expiry` | `(expires_at - now())` floored to integer days at send time |

Per BOOK-203 the lifetime is 7 days from creation; at first send `days_until_expiry = 7`. Re-sends (§10) re-compute against current time so the recipient always sees the accurate remaining-window.

### 8.2 Security note

```
If you didn't expect this invitation, ignore this email. The invitation will
expire automatically. If you have concerns about your account or this invitation,
contact {{inviter_email}}.
```

`inviter_email` provides a direct reply-path for security concerns; the invitee doesn't need to navigate to the platform to flag an unexpected invitation.

### 8.3 Unsubscribe handling

**Invitation emails are system-critical and cannot be unsubscribed.** Per `email_delivery_integration.md` §"Unsubscribe Handling" — there's no recipient account to opt out of yet, and the email represents a one-time action by another authenticated user.

No unsubscribe link is shown in the footer. Footer instead carries: "This is a one-time invitation, not a marketing email. You will not receive further emails from this address unless you accept the invitation and configure notifications."

## 9. CTA link path

The link `https://{{platform-domain}}/invite/{{token_id}}` resolves on click as follows:

1. Backend looks up `invitation_tokens` row by `id = token_id`.
2. Per BOOK-203 redemption flow §"Redemption", validates: row exists, not revoked, not expired, not consumed.
3. If `invitee_email` matches an existing `users` row → redirect to `/businesses/{{business_id}}` (business switcher) after attaching the new role.
4. Else → redirect to `/accept-invitation?token={{token_id}}` (new-account flow with email pre-filled and unchangeable).
5. On failure validation (revoked/expired/consumed) → redirect to `/invitation-error?reason={{reason}}` with the structured error context.

The token UUID is exposed in the URL. Per BOOK-203 it's NOT hashed in the database (subject to Stage-6 reconciliation per the BOOK-203 drift note); the URL design assumes the token IS the credential.

## 10. Re-send rule

When an OWNER/ADMIN clicks "Resend invitation" on an existing pending invitation in the UI:

1. The platform dispatches the same email again with the same `token_id` (no new `invitation_tokens` row created).
2. A new column `last_sent_at timestamptz` is bumped (column to be added at B02·P07 implementation — currently BOOK-203's schema doesn't include it).
3. The expiry notice §8.1 re-renders against the original `expires_at` — re-sends do NOT extend the expiry.
4. Audit emits `AUTH_INVITATION_RESENT` (LOW) with payload `{ token_id, business_id, resent_by_user_id, original_sent_at, last_sent_at_now }`.

Rate-limit on re-sends: at most 3 re-sends per `token_id` per 24-hour window (prevents spam against the invitee's inbox). Beyond 3 attempts → UI shows "Already resent 3 times today. Cancel and re-invite if needed."

## 11. Audit and send-side behaviour

Two distinct audit events fire:

| Event | Severity | When | Source |
|---|---|---|---|
| `AUTH_INVITATION_SENT` | LOW | `invitation_tokens` row INSERT (per BOOK-203) | Invitation creation RPC |
| `EMAIL_DISPATCHED` (or `EMAIL_SENT` per BOOK-164 audit-name drift) | LOW | Email provider accepts the send | `email_delivery_integration.md` send path |

The two are distinct because the invitation is "sent" the moment the row exists (the token can be redeemed if the invitee somehow has the URL by other means); the email is a best-effort delivery channel. Failed email delivery does NOT roll back the invitation row.

Per `email_delivery_integration.md` §"Bounce Handling": if the invitation email hard-bounces (`contact.email_status = INVALID`), the UI surfaces a warning on the invitation row showing "Email could not be delivered. Update the email address or share the link directly."

## 12. Test fixtures (CI requirements)

Three fixture variants must render in CI without errors and pass HTML lint + accessibility check:

1. **First-time invitee, OWNER role** — exercises the first-time-invitee body path + the OWNER role-description block.
2. **First-time invitee, BOOKKEEPER role** — exercises the lower-privileged role-description block (verifies the description fits within 200-char target).
3. **Pre-existing-account invitee, ACCOUNTANT role** — exercises the pre-existing-account body path (§4.2 alternate) + the ACCOUNTANT role-description block.

Each fixture validates:

- HTML lint passes (no orphan tags, all attributes quoted).
- Inline styles only (no `<style>` blocks).
- Logo URL resolves (HEAD request returns 200 at fixture-build time).
- CTA link URL pattern matches `https://[domain]/invite/[uuid-v4-regex]`.
- Plain-text fallback rendered correctly (URLs preserved inline, no orphan markup).
- Subject line ≤ 100 chars after substitution.
- Role-description block ≤ 200 chars.

resolving review queue issues, and applying transaction tags.
