# VIES Record Format

**Category:** Reference data ¬∑ **Owning block:** 11 ‚Äî Ledger & Cyprus VAT Engine ¬∑ **Co-owner:** 16 ‚Äî Dashboard & Reporting ¬∑ **Stage:** 4 sub-doc (Layer 1 reference)

The canonical VIES record shape ‚Äî used in two distinct output formats per the 2026-05-08 amendment: a **CSV** form inside the Block 15 archive bundle (for audit and accountant access), and an **XML** form generated by Block 16 for regulator filing. Both forms carry the same record fields; the serialization differs.

Per Stage 1: full VIES file export to the Cyprus regulator specification is in MVP scope.

---

## Record fields

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `client_country_iso` | text(2) | ‚úì | ISO 3166-1 alpha-2 (e.g., `DE`, `FR`, `NL`); must be an EU member state per `eu_member_state_set` |
| `client_vat_number` | text | ‚úì | Normalised per `vat_number_format_catalog` ‚Äî country prefix + digits; e.g., `DE123456789` |
| `value_eur_cents` | bigint | ‚úì | Total transaction value for the period, EUR minor units (integer); never negative |
| `transaction_count` | integer | ‚úì | Number of transactions consolidated into this record (typically 1 per invoice for IN side, varies for OUT) |
| `period_year` | integer | ‚úì | Cyprus VIES reporting period ‚Äî year (e.g., `2026`) |
| `period_quarter` | integer | ‚úì | Cyprus VIES reporting period ‚Äî quarter (1‚Äì4) |
| `transaction_type` | enum | ‚úì | One of: `SERVICE`, `GOODS_SUPPLY`, `GOODS_ACQUISITION` |
| `source_run_id` | uuid | ‚úì | The workflow run that finalized these transactions; lets a reader trace back to the period in the archive |
| `source_period_start` | date | ‚úì | Operational period start |
| `source_period_end` | date | ‚úì | Operational period end |

## Per-transaction-type semantics

| Transaction type | Routes via | VAT treatment |
| --- | --- | --- |
| `SERVICE` | IN-side B2B EU service supplied by Cyprus business | `EU_REVERSE_CHARGE` (`NON_EU_SERVICE` is NOT VIES-reportable ‚Äî see `vat_treatment_enum`) |
| `GOODS_SUPPLY` | IN-side EU goods supplied by Cyprus business | `EU_REVERSE_CHARGE` |
| `GOODS_ACQUISITION` | OUT-side intra-EU goods acquired by Cyprus business | `IMPORT_OR_ACQUISITION` with `import_acquisition_subtype = INTRA_EU_ACQUISITION` |

## Eligibility (which transactions become VIES records)

Per `vies_quarterly_eligibility_policy` (Block 11) and the `vies_relevant` projection (per `vat_treatment_enum`):

- Source: `archive.locked_ledger_entries` for finalized periods
- Filter: `vies_relevant = true`
- Additional filter for IN side: `EU_REVERSE_CHARGE` requires validated `counterparty_vat_number` per Block 11 Phase 06; UNRESOLVED counterparties never produce VIES records (they were caught at finalization preconditions)
- Aggregation: one record per `(client_country_iso, client_vat_number, transaction_type, period_quarter)` tuple ‚Äî multiple invoices to the same client in the same quarter sum into one record (per Cyprus VIES filing convention)

## CSV format (archive bundle)

File: `vies_export.csv` inside the Block 15 archive bundle.

```
client_country_iso,client_vat_number,value_eur_cents,transaction_count,period_year,period_quarter,transaction_type,source_run_id,source_period_start,source_period_end
DE,DE123456789,154200,3,2026,1,SERVICE,...,2026-01-01,2026-01-31
NL,NL987654321,89000,1,2026,1,GOODS_SUPPLY,...,2026-01-01,2026-01-31
```

- UTF-8 encoding, LF line endings, no BOM
- Header row required (exact field names above)
- Quoting: RFC 4180 ‚Äî fields with commas or quotes are quoted; quote escape doubles the quote
- Numeric fields without quotes; text fields without quotes unless they contain commas (won't happen for VAT numbers / country codes)
- Stable sort: `(client_country_iso ASC, client_vat_number ASC, transaction_type ASC)` ‚Äî deterministic for archive integrity

## XML format (regulator filing ‚Äî Block 16)

File generated by Block 16 Phase 11. Conforms to `cyprus_vies_xsd_schema` (Block 16 Reference data, sibling to this sub-doc).

Indicative shape (final XSD pinned per Cyprus tax authority specification):

```xml
<vies-return xmlns="..." schema-version="..." period-year="2026" period-quarter="1">
  <header>
    <filer-vat-number>CY12345678X</filer-vat-number>
    <filer-name>...</filer-name>
    <submission-timestamp>2026-04-10T09:00:00Z</submission-timestamp>
  </header>
  <records>
    <record>
      <client-country-iso>DE</client-country-iso>
      <client-vat-number>DE123456789</client-vat-number>
      <value-eur>1542.00</value-eur>
      <transaction-count>3</transaction-count>
      <transaction-type>SERVICE</transaction-type>
    </record>
    ...
  </records>
</vies-return>
```

XML carries `value-eur` as a decimal string (e.g., `1542.00`) per Cyprus tax authority convention; CSV carries `value_eur_cents` as an integer for determinism. Block 16 Phase 11 owns the cents ‚Üí decimal conversion at XML render time.

## Determinism

Both formats commit to byte-identical output for the same input:

- Field ordering ‚Äî fixed (CSV header order; XML element order)
- Record ordering ‚Äî `(client_country_iso, client_vat_number, transaction_type)` ascending
- Number formatting ‚Äî CSV uses integer cents (no formatting), XML uses 2-decimal-place string in en-US locale (always `.` separator)
- Whitespace ‚Äî XML pretty-printing pinned to 2-space indent + LF line endings
- No timestamps in the body (the submission-timestamp is in the header but its value is recorded once per submission and stable for re-renders)

## Online validation

VIES VAT numbers can be validated against the EU VIES service via `vies_online_validation_integration` (Block 11 Integrations, Layer 2). The validation is per-client at counterparty resolution time (Block 11 Phase 04 / 06), not per-record at export time. Invalid VAT numbers route the transaction to `UNKNOWN` per `vat_treatment_enum` and never produce a VIES record.

## Quarterly eligibility edge cases

- A period that straddles a quarter boundary is split ‚Äî transactions before the boundary route to one VIES quarter, after to the next
- A quarterly VIES return covers transactions whose `accounting_impact_date` falls in the quarter (per the Cyprus VAT-period assignment rule pinned in Block 13 Phase 11 for credit notes)
- A retroactive credit note from a later period adjusts the earlier quarter's record via the manifest-versioning chain (Block 15 Phase 06)

## VIES response field semantics

When the system validates a client VAT number against the EU VIES online validation service (via `vies_online_validation_integration`), the response carries these fields:

| Response field | Type | Nullable | Meaning |
| --- | --- | --- | --- |
| `countryCode` | string(2) | No | ISO 3166-1 alpha-2 of the member state where the VAT number is registered |
| `vatNumber` | string | No | The VAT number as registered (may differ from the normalized form sent in the query ‚Äî e.g., the service strips the country prefix) |
| `requestDate` | date | No | The date the validation request was processed by the VIES service |
| `valid` | boolean | No | `true` if the VAT number is registered and active; `false` if invalid or deregistered |
| `name` | string | Yes | Legal name of the VAT-registered entity. Nullable ‚Äî VIES returns this only when the member state's tax authority exposes it (not all do) |
| `address` | string | Yes | Registered address. Same nullability as `name` ‚Äî member-state-dependent |
| `consultationNumber` | string | No | A unique VIES consultation reference number for this validation request. Used for audit documentation ("validated on VIES, consultation #XYZ") |

The `name` and `address` fields being nullable is significant: the system cannot rely on them for counterparty resolution. They are stored when present (for auditor convenience) but Block 11 Phase 06's counterparty resolution logic uses `countryCode` + `vatNumber` + `valid` only.

## VIES error codes and their handling

The VIES service returns SOAP faults for error conditions. The system maps them to internal review issues:

| VIES error code | Meaning | System behavior |
| --- | --- | --- |
| `INVALID_INPUT` | The VAT number format is malformed | Block 11 raises `Possible Tax-VAT Issue` at HIGH severity; the counterparty remains `UNRESOLVED`; the classifier sets `vat_treatment = UNKNOWN` for the affected transaction |
| `INVALID_REQUESTER_INFO` | The requesting country code is not accepted | System-level configuration error; escalated to operator; emits `VIES_VALIDATION_SYSTEM_ERROR` audit event |
| `SERVICE_UNAVAILABLE` | The VIES service is temporarily down | Retry up to 3 times with exponential backoff (2s, 4s, 8s); if all retries fail, cache the `SERVICE_UNAVAILABLE` state for 15 minutes; raise a LOW-severity `Needs Confirmation` issue noting VIES is unavailable |
| `MS_UNAVAILABLE` | The specific member state's system is unavailable | Same retry and cache behavior as `SERVICE_UNAVAILABLE` |
| `TIMEOUT` | The VIES request timed out | Same retry and cache behavior |
| `VAT_BLOCKED` | The VAT number exists but is blocked (under investigation, suspended) | Block 11 raises `Possible Tax-VAT Issue` at HIGH severity; counterparty flagged as `VALIDATION_BLOCKED`; a VIES record is NOT generated for this counterparty until cleared |
| `GLOBAL_MAX_CONCURRENT_REQ` | Rate limit hit at the VIES gateway | Retry after 60 seconds; if persistent, queue the validation for the next run |

Successful validation with `valid = false` (the number is formatted correctly but is not registered or has been deregistered) is treated equivalently to `INVALID_INPUT` for the purposes of VIES record eligibility ‚Äî a deregistered VAT number cannot be a VIES reportable counterparty.
