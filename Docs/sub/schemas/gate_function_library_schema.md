# gate_function_library_schema

**Category:** Schemas · **Owning block:** 12 — OUT Workflow · **Stage:** 4 sub-doc (Layer 2)

The schema for the phase gate function registry. A gate function is a named, pure predicate evaluated at each phase boundary to determine whether the workflow run should `ADVANCE`, `HOLD`, or `FAIL` (with the third variant being `ROUTE_TO_SIDE_PHASE` for non-terminal routing decisions, per Block 03 Phase 05's return-type contract). Gate functions are registered at engine boot and are immutable during a run's lifetime.

Gate functions are a sub-class of the tool registry: they follow the naming convention from `tool_naming_convention_policy` and are catalogued separately from operational tools because their side-effect class is always `READ_ONLY` (pure predicates; no writes).

---

## Table definition

```sql
CREATE TYPE gate_outcome_enum AS ENUM (
  'ADVANCE',
  'HOLD',
  'ROUTE_TO_SIDE_PHASE',
  'FAIL'
);

CREATE TYPE hold_severity_enum AS ENUM (
  'LOW',
  'MEDIUM',
  'HIGH',
  'BLOCKING'
);

CREATE TABLE gate_function_registry (
  gate_id                  uuid PRIMARY KEY DEFAULT gen_uuid_v7(),
  gate_name                text NOT NULL UNIQUE,
  owning_block             integer NOT NULL,
  phase_name               text NOT NULL,
  gate_version             text NOT NULL,
  evaluation_logic_ref     text NOT NULL,
  hold_severity_on_fail    hold_severity_enum NOT NULL DEFAULT 'HIGH',
  retry_allowed            boolean NOT NULL DEFAULT true,
  registered_at            timestamptz NOT NULL DEFAULT now(),

  CONSTRAINT gate_name_format
    CHECK (gate_name ~ '^[a-z][a-z0-9_]*\.[a-z][a-z0-9_]*$'),
  CONSTRAINT gate_version_format
    CHECK (gate_version ~ '^\d+\.\d+$'),
  CONSTRAINT evaluation_logic_ref_nonempty
    CHECK (char_length(evaluation_logic_ref) > 0)
);
```

## Column rationale

### `gate_id` (UUID v7)

Primary key generated by `gen_uuid_v7()` per `data_layer_conventions_policy`. UUID v7 is correct here — the gate registry is a business-data table and the time-prefix is not security-sensitive.

### `gate_name` (text, UNIQUE)

Follows the two-part naming convention `engine.gate_<phase_descriptor>`, where `engine` is the block short-name for Block 03 (Workflow Engine) drawn from the allowlist in `tool_naming_convention_policy`, and the action segment is `gate_` followed by a snake_case phase descriptor. Gate registration names fall under the `engine` block short-name and are subject to the same lint rules as tool names.

Examples (canonical, registered at boot):

| gate_name | Phase | Block |
| --- | --- | --- |
| `engine.gate_ingestion_complete` | `INGESTION` | 12 |
| `engine.gate_classification_complete` | `CLASSIFICATION` | 12 |
| `engine.gate_out_filter_complete` | `OUT_FILTER` | 12 |
| `engine.gate_evidence_discovery_email_complete` | `EVIDENCE_DISCOVERY_EMAIL` | 12 |
| `engine.gate_evidence_discovery_drive_complete` | `EVIDENCE_DISCOVERY_DRIVE` | 12 |
| `engine.gate_matching_complete` | `MATCHING` | 12 |
| `engine.gate_manual_upload_hold_clear` | `MANUAL_UPLOAD_HOLD` | 12 |
| `engine.gate_ledger_preparation_complete` | `LEDGER_PREPARATION` | 12 |
| `engine.gate_ai_end_scan_complete` | `AI_END_SCAN` | 12 |
| `engine.gate_human_review_hold_clear` | `HUMAN_REVIEW_HOLD` | 12 |
| `engine.gate_finalization_complete` | `FINALIZATION` | 12 |

The `UNIQUE` constraint on `gate_name` is intentional and fatal-on-collision at boot — identical to the tool registry's name-collision behaviour. Two gate functions with the same name cannot coexist.

The format regex `^[a-z][a-z0-9_]*\.[a-z][a-z0-9_]*$` enforces lowercase snake_case with exactly two dot-separated segments. The first segment must be in the `tool_naming_convention_policy` block short-name allowlist (enforced by boot-time registration, not the CHECK constraint alone).

### `owning_block` (integer)

The block number (12 for OUT Workflow gates, 13 for IN Workflow gates, 03 for engine-internal gates). This column is informational; the authoritative binding is the `gate_name` prefix.

### `phase_name` (text)

The phase this gate is attached to. Must be a valid phase name in the registered workflow type's phase sequence. The engine resolves the gate at phase-boundary evaluation time by looking up `(phase_name, owning_block)` in this table.

### `gate_version` (text — semver `major.minor`)

Version of the gate function's evaluation logic. Bump rules mirror `tool_naming_convention_policy`:
- **Major bump** — evaluation logic changes in a way that could produce a different outcome for the same inputs (predicate condition added/removed, severity threshold changed). Old version remains registered for one full workflow-run cycle before removal; `WORKFLOW_TOOL_REGISTRATION_DEPRECATED` is emitted.
- **Minor bump** — implementation change with identical output behaviour (query optimisation, index hint). No deprecation period.

### `evaluation_logic_ref` (text)

File system path to the TypeScript implementation file for this gate, relative to the project root (e.g., `src/blocks/12_out_workflow/gates/gate_out_ingestion_complete.ts`). The CI lint rule for tool description references (`description_ref` in `tool_naming_convention_policy`) applies here: the path must resolve to an existing file.

### `hold_severity_on_fail` (hold_severity_enum)

The severity of the review issue raised when this gate returns `FAIL` (unrecoverable error path, distinct from a `HOLD` outcome). Default `HIGH`. Most gates use `HIGH`; a gate whose failure is ambiguous (e.g., the finalization gate on partial write) uses `BLOCKING`.

The `hold_severity_enum` is the project-wide `{LOW, MEDIUM, HIGH, BLOCKING}` set. `CRITICAL` is not a valid value (removed in Stage 2 audit taxonomy cleanup).

### `retry_allowed` (boolean, default true)

When `true`, the engine applies Block 03 Phase 08's bounded retry policy before treating an error as a `FAIL`. When `false`, the first error is immediately treated as `FAIL` (used for gates where retrying is known to be futile, e.g., a missing required configuration).

### `registered_at` (timestamptz)

Boot-time registration timestamp. Used for gate-version history auditing. Not updated on minor-version bumps.

## Side-effect class constraint

Gate functions must carry exactly one side-effect class: `READ_ONLY`. The engine rejects any gate registration that declares `WRITES_*` or `EXTERNAL_CALL` classes. This constraint is architectural — gates are pure predicates. Any side effect required as part of a gate's evaluation must be extracted into a separate operational tool that runs before the gate is called.

This is a binding rule. CI blocks any gate implementation that calls a write-path function directly.

## Gate outcome to run-state transition mapping

| Gate outcome | Run-level state | Notes |
| --- | --- | --- |
| `ADVANCE` | `RUNNING` | Engine immediately begins next phase in sequence |
| `HOLD` | `REVIEW_HOLD` | System-initiated hold; run re-evaluates gate when an unblocking event arrives |
| `ROUTE_TO_SIDE_PHASE` | `REVIEW_HOLD` (for `MANUAL_UPLOAD_HOLD`) or `AWAITING_APPROVAL` (for `HUMAN_REVIEW_HOLD`) | Side-phase entered; main-sequence advance deferred until side-phase clears |
| `FAIL` (unrecoverable) | `FAILED` | Terminal after bounded retries exhaust; creates a review issue at `hold_severity_on_fail` |

The canonical 10-value run-state enum is in `workflow_state_enum`. Transitions are executed by `transitionRun()` in Block 03 Phase 04; the gate function returns an outcome and the engine drives the transition — the gate itself makes no state writes.

## `WORKFLOW_GATE_EVALUATED` audit event

Every gate evaluation emits `WORKFLOW_GATE_EVALUATED` (domain `WORKFLOW_GATE`, severity `LOW`). The event is emitted regardless of outcome.

Payload (canonical JSON per `data_layer_conventions_policy`):

```json
{
  "gate_name": "engine.gate_matching_complete",
  "gate_version": "1.0",
  "run_id": "<uuid>",
  "business_id": "<uuid>",
  "phase_name": "MATCHING",
  "phase_index": 5,
  "outcome": "ROUTE_TO_SIDE_PHASE",
  "side_phase_target": "MANUAL_UPLOAD_HOLD",
  "evaluation_duration_ms": 42,
  "evaluated_at": "2026-01-15T09:01:30Z"
}
```

`side_phase_target` is present only when `outcome = ROUTE_TO_SIDE_PHASE`. For `HOLD` outcomes, an additional `hold_reason` field carries the first blocking predicate that failed (implementation detail; sub-doc owns the exact shape).

In addition to `WORKFLOW_GATE_EVALUATED`:
- `WORKFLOW_GATE_HOLD` is emitted when outcome is `HOLD`.
- `WORKFLOW_GATE_ROUTED_TO_SIDE_PHASE` is emitted when outcome is `ROUTE_TO_SIDE_PHASE`.
- `WORKFLOW_GATE_TIMEOUT` is emitted if gate evaluation exceeds its configured timeout budget (Block 03 Phase 05 owns the timeout).

---

## RLS

Gate function registry rows are system-level metadata, not tenant data. They carry no `business_id` and are not subject to per-tenant RLS. Read access is unrestricted for all authenticated sessions; write access is restricted to the `app.migration_role` and `app.boot_role` — normal application code does not INSERT or UPDATE gate registrations at runtime.

---

## Mobile write rejection

Gate function registry rows are written only at engine boot by the migration/boot roles. No end-user write path exists. The `mobile_write_rejection_endpoints` policy does not apply to this table because it has no user-facing write surface. Read access is not restricted for mobile clients.

## Indexes

```sql
CREATE INDEX idx_gate_function_registry_phase
  ON gate_function_registry(phase_name, owning_block);

CREATE INDEX idx_gate_function_registry_block
  ON gate_function_registry(owning_block);
```

The `UNIQUE` constraint on `gate_name` doubles as the primary lookup index for the engine's phase-boundary evaluation.
