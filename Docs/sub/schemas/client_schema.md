# client_schema

**Category:** Schemas · **Owning block:** 13 — IN Workflow + Invoice Generator · **Stage:** 4 sub-doc (Layer 2)

The `clients` table. Clients are the counterparty records against which incoming bank transactions are matched in the income-matching phase (Block 10 Phase 08) and to which pro-forma and tax invoices are issued by the Invoice Generator (Block 13 Phases 01–05). A client record is created once per business–counterparty pair; the same legal entity appearing across multiple businesses of the same tenant is stored as separate client rows (one per `business_id`) to preserve tenant isolation.

## Column rationale

### `client_id` (UUID v7)

Primary key generated by `gen_uuid_v7()` per `data_layer_conventions_policy`. UUID v7 is correct — client records are business-data rows.

### `business_id`

Standard tenant isolation column. All RLS policies and index prefixes use this column. Deriving tenancy from `client_id` is explicitly prohibited per `data_layer_conventions_policy`.

### `display_name` (text NOT NULL)

The human-readable name shown in the UI, on invoices, and in matching reason text. Not used on formal tax documents (use `legal_name` for those). Minimum length 1; blank strings are rejected at the application layer before INSERT.

### `legal_name` (text, nullable)

The full legal company name used on tax invoices, VAT submissions, and credit notes. When null, the PDF renderer falls back to `display_name`. This field is nullable because many clients (individual freelancers, small businesses) do not have a distinct legal name.

### `vat_number` (text, nullable)

The client's VAT registration number. Format and checksum validation rules are country-specific; the application layer performs format validation before INSERT. VIES online verification is attempted asynchronously after INSERT via `in_workflow.verify_client_vat_number`; the result is written to `vat_number_verified_at`. A null `vat_number` is valid — not all clients are VAT-registered.

### `vat_number_verified_at` (timestamptz, nullable)

Timestamp of the most recent successful VIES verification. Null means either the VAT number was never submitted for verification, or the last verification attempt failed (the failure is audit-logged separately as part of the VIES verification flow in Block 11). VIES reverification is triggered by Block 11's ledger phase when the VAT number is first used on a ledger entry.

### `country_code` (char(2) NOT NULL)

ISO 3166-1 alpha-2 country code. Required — used in VAT treatment decisions (e.g., distinguishing Cyprus domestic from EU B2B from non-EU). Validated against the ISO 3166 list at the application layer.

### `address_json` (JSONB, nullable)

Structured postal address. Shape:

```json
{
  "line1": "123 Main Street",
  "line2": null,
  "city": "Nicosia",
  "postal_code": "1010",
  "country_code": "CY"
}
```

Numeric amounts are never stored in `address_json`. The shape is validated by the application layer against a JSON Schema before INSERT. Null is permitted when only email delivery is configured for the client.

### `email` (text, nullable)

Primary billing contact email. Used as the delivery address for `in_workflow.send_invoice` when `send_channel = EMAIL`. The `DRAFT → SENT` guard in `invoice_lifecycle_policy` requires a non-null email (or postal address) before sending.

### `currency` (char(3), default `EUR`)

ISO 4217 currency code. Default EUR per Cyprus operational context. Stored on the client record as a default for new invoices issued to this client; individual invoice rows carry their own `currency` column (immutable after creation per `decisions_log.md` multi-currency lock rule). Changing a client's `currency` does not retroactively affect existing invoices.

### `payment_terms_days` (integer, default 30)

Number of days from invoice issue date to due date, applied as the default when creating new invoices for this client. May be overridden per-invoice. Range 0–365.

### `is_active` (boolean, default true)

Soft-delete flag per `soft_delete_vs_status_policy`. Setting `is_active = false` deactivates the client — the record is retained for audit and historical invoice purposes, but the client no longer appears in active client lists, can no longer receive new invoices, and is excluded from matching candidates in Block 10 Phase 08. Hard-delete of client records is not permitted; the `is_active` flag is the only supported deactivation path.

Active clients with open invoices (any non-terminal invoice status) cannot be deactivated until those invoices are resolved. This guard is enforced at the application layer by `in_workflow.deactivate_client` before setting `is_active = false`.

## Unique constraint on VAT number

`UNIQUE NULLS NOT DISTINCT (business_id, vat_number)` where `vat_number IS NOT NULL`:
- One business cannot have two active client rows for the same VAT number. The constraint fires regardless of `is_active` — a deactivated client's VAT number is still reserved (prevents silent re-registration of a deactivated client under a different `client_id`).
- Two clients in the same business may both have `vat_number = NULL` — this is expected for unverified or non-VAT-registered counterparties.

## Indexes

```sql
CREATE INDEX idx_clients_business_active
  ON clients(business_id, is_active);

CREATE INDEX idx_clients_business_email
  ON clients(business_id, email)
  WHERE email IS NOT NULL;

CREATE INDEX idx_clients_business_name
  ON clients(business_id, display_name);
```

The VAT-number unique constraint (or partial unique index on older Postgres) covers `(business_id, vat_number)` equality lookups.

## RLS

Standard tenant isolation:

```sql
CREATE POLICY clients_business_isolation ON clients
  FOR ALL
  USING (business_id = ANY(auth.business_ids_for_session()));
```

## Permission gate

| Operation | Minimum role |
| --- | --- |
| SELECT | Any role with business access |
| INSERT (create client) | Bookkeeper, Admin, Owner |
| UPDATE (edit client) | Bookkeeper, Admin, Owner |
| Deactivate (`is_active = false`) | Admin, Owner |

Accountant, Reviewer, and Read-only roles receive 403 on any mutation attempt. Permission surfaces are defined in Block 02 Phase 04's permission matrix.

---

## Mobile write rejection

All write operations (INSERT, UPDATE, deactivation) are rejected for sessions where `client_form_factor = MOBILE`. The rejection fires before the permission check and emits `MOBILE_WRITE_REJECTED`. Mobile clients may read client records. Reference: `mobile_write_rejection_endpoints.md`.

## Audit events

| Event | Severity | Trigger |
| --- | --- | --- |
| `CLIENT_CREATED` | LOW | New client row inserted |
| `CLIENT_UPDATED` | LOW | Any field updated on an existing client row; payload carries field-level before/after diff |
| `CLIENT_DEACTIVATED` | MEDIUM | `is_active` set to `false`; payload carries `deactivated_by_user_id` and timestamp |

All events use domain `CLIENT` per `audit_log_policies`. Payloads are canonical JSON per `data_layer_conventions_policy`.
