# invoice_line_item_schema

**Category:** Schemas · **Owning block:** 13 — IN Workflow + Invoice Generator · **Stage:** 4 sub-doc (Layer 2)

The `invoice_line_items` table records individual line items on an invoice: services rendered, goods delivered, fees charged, or any other discrete billable component. This table is the normalized complement to the `invoices.lines_payload` JSONB column, which carries a denormalized snapshot of the same data for read performance and PDF rendering. The normalized table is the write target; the JSONB column is maintained as a derived representation.

Per `data_layer_conventions_policy`: primary key is UUID v7, numeric amounts use `numeric` types (not floats), and canonical JSON serialization applies to any JSON serialized form of these rows.

---

## Table definition

```sql
CREATE TABLE invoice_line_items (
  line_item_id        uuid PRIMARY KEY DEFAULT gen_uuid_v7(),
  invoice_id          uuid NOT NULL REFERENCES invoices(invoice_id) ON DELETE CASCADE,
  business_id         uuid NOT NULL REFERENCES business_entities(id),

  -- Ordering
  line_index          integer NOT NULL CHECK (line_index >= 0),

  -- Description
  description         text NOT NULL CHECK (char_length(description) >= 1),

  -- Quantity and price
  quantity            numeric(10,4) NOT NULL DEFAULT 1
                        CHECK (quantity > 0),
  unit_price_eur      numeric(15,2) NOT NULL,

  -- VAT
  vat_treatment       text NOT NULL,   -- value from vat_treatment_enum
  vat_rate            numeric(5,4),    -- e.g. 0.1900 for 19%; NULL when vat_treatment is zero-rate or exempt

  -- Computed totals (denormalized for integrity enforcement; see constraint below)
  line_total_net_eur  numeric(15,2) NOT NULL,
  line_total_vat_eur  numeric(15,2) NOT NULL DEFAULT 0,
  line_total_gross_eur numeric(15,2) NOT NULL,

  -- Authorship
  created_at          timestamptz NOT NULL DEFAULT now(),
  updated_at          timestamptz NOT NULL DEFAULT now(),

  -- Constraints
  CONSTRAINT line_item_unique_index
    UNIQUE (invoice_id, line_index),

  CONSTRAINT line_total_net_check
    CHECK (line_total_net_eur = ROUND(quantity * unit_price_eur, 2)),

  CONSTRAINT line_total_gross_check
    CHECK (line_total_gross_eur = line_total_net_eur + line_total_vat_eur),

  CONSTRAINT vat_rate_present_when_nonzero
    CHECK (
      vat_rate IS NOT NULL OR vat_treatment IN (
        'STANDARD_EXEMPT', 'INTRAEU_SUPPLY_ZERO', 'OUTSIDE_SCOPE', 'REVERSE_CHARGE_EU'
      )
    )
);
```

## Column rationale

### `line_item_id` (UUID v7)

Primary key generated by `gen_uuid_v7()` per `data_layer_conventions_policy`. UUID v7 is used because line items are business-data rows and the time-prefix is not security-sensitive.

### `invoice_id` (FK to `invoices`, CASCADE DELETE)

Foreign key to the parent invoice. `ON DELETE CASCADE` means that deleting an invoice cascades to all its line items. In practice, invoice deletion only applies to `DRAFT` invoices per `invoice_lifecycle_policy`; non-DRAFT invoices cannot be deleted. The cascade is a safety net for the DRAFT deletion path.

### `business_id`

Redundant tenant isolation column. This column is denormalized from `invoices.business_id` to allow the RLS policy to filter directly on this table without a join to `invoices`. Deriving tenancy from `line_item_id` is explicitly prohibited per `data_layer_conventions_policy`.

### `line_index` (integer, 0-based)

Zero-based integer defining the display and calculation order of line items on the invoice. The `UNIQUE (invoice_id, line_index)` constraint ensures no two line items on the same invoice share a position. Line items are displayed in ascending `line_index` order by the PDF renderer and the API response.

Gaps in `line_index` values are permitted (e.g., indices 0, 2, 5 are valid). The renderer sorts by index and does not require contiguous values.

### `description` (text NOT NULL)

Free-text description of the service or good. Minimum length 1; blank strings are rejected. This field appears on the rendered PDF invoice. There is no maximum length enforced at the DB level; the PDF renderer may truncate long descriptions to fit the layout (renderer sub-doc owns truncation rules).

### `quantity` (numeric(10,4), default 1)

The quantity of units billed. Four decimal places accommodate fractional units (e.g., 1.5 hours, 0.25 day). Must be strictly positive; zero or negative quantities are rejected by the CHECK constraint. When billing a fixed amount (e.g., a flat service fee), `quantity = 1` and `unit_price_eur` carries the full amount.

### `unit_price_eur` (numeric(15,2))

The price per unit in EUR. Negative values are permitted to represent discounts or adjustments as a line item. The convention for credit lines is a negative `unit_price_eur` with a positive `quantity`. Two decimal places are sufficient for EUR amounts at the unit level; the `line_total` columns carry the rounded product.

Currency is fixed to EUR at the line-item level. The parent `invoices.currency` column carries the invoice-level currency, which must be consistent with the line items. Multi-currency line items on the same invoice are not supported in MVP.

### `vat_treatment` (text NOT NULL)

The VAT treatment applied to this line item. Values are drawn from `vat_treatment_enum`. The treatment determines whether `vat_rate` is applicable and how the VAT amount is computed. Validation against the enum is performed at the application layer before INSERT.

### `vat_rate` (numeric(5,4), nullable)

The VAT rate as a decimal fraction (e.g., `0.1900` for 19%, `0.0500` for 5%). Present when the `vat_treatment` implies a positive rate; null when the treatment is zero-rate, exempt, or outside scope. The `vat_rate_present_when_nonzero` CHECK constraint enforces that `vat_rate` is non-null for treatments that require it, and permits null for the named zero/exempt/outside-scope treatments.

The Cyprus standard VAT rate is 19% (`0.1900`). Reduced rates (5%, 9%) apply per `vat_rate_table_reference`.

### `line_total_net_eur` (numeric(15,2))

Net amount for this line item: `ROUND(quantity × unit_price_eur, 2)`. The CHECK constraint `line_total_net_check` enforces this relationship at the database level. The application layer must compute and pass the correct value; the DB constraint is an integrity backstop.

### `line_total_vat_eur` (numeric(15,2), default 0)

VAT amount for this line item. Computed as `ROUND(line_total_net_eur × vat_rate, 2)` for treatments with a positive rate; `0` for zero-rate, exempt, and outside-scope treatments. Default is `0`.

### `line_total_gross_eur` (numeric(15,2))

Gross amount for this line item: `line_total_net_eur + line_total_vat_eur`. The CHECK constraint `line_total_gross_check` enforces this relationship. There is no separate rounding step for the gross; it is the arithmetic sum of net and VAT.

## Immutability after invoice SENT

Line items are immutable once the parent invoice transitions out of `DRAFT` (i.e., once the invoice reaches `SENT` status and a sequence number is allocated). Any attempt to INSERT, UPDATE, or DELETE a line item for a non-DRAFT invoice is rejected before the permission check and emits `INVOICE_LIFECYCLE_TRANSITION_FAILED`.

Amendment of line items on an ISSUED invoice follows `invoice_amendment_policy`: the original invoice must be voided, and the replacement invoice is created with the corrected line items.

---

## Audit coverage

There is no separate `LINE_ITEM_AMENDED` audit event. Line item changes on a `DRAFT` invoice are covered by the `INVOICE_AMENDED` event emitted on the parent invoice. The `INVOICE_AMENDED` payload includes a `changed_sections` field listing `"line_items"` when line items are the changed component.

Voiding the parent invoice (which cascades to line items via the `ON DELETE CASCADE` path) is covered by `INVOICE_VOIDED` on the parent.

---

## Relationship to `invoices.lines_payload`

The `invoices.lines_payload` JSONB column carries a denormalized snapshot of all line items. The canonical write path is:

1. INSERT/UPDATE/DELETE in `invoice_line_items`.
2. The application layer (or a Postgres trigger, as determined by Block 13 Phase 01's implementation) refreshes `invoices.lines_payload` with the canonical JSON serialization of the current line items, per `data_layer_conventions_policy` canonical JSON rules.

The `lines_payload` is the source of truth for PDF rendering and archive export. The `invoice_line_items` table is the source of truth for relational queries (total recomputation, VAT breakdown aggregation, per-line audit).

## RLS

Tenant isolation is enforced via the `business_id` column:

```sql
CREATE POLICY invoice_line_items_isolation ON invoice_line_items
  FOR ALL
  USING (business_id = ANY(auth.business_ids_for_session()));
```

---

## Mobile write rejection

All write operations on `invoice_line_items` (INSERT, UPDATE, DELETE) are rejected for sessions where `client_form_factor = MOBILE`. Rejection fires before the permission check and emits `MOBILE_WRITE_REJECTED` per `mobile_write_rejection_endpoints.md`. Mobile clients may read line items.

---

## Indexes

```sql
CREATE INDEX idx_invoice_line_items_invoice
  ON invoice_line_items(invoice_id);

CREATE INDEX idx_invoice_line_items_business
  ON invoice_line_items(business_id);
```

The `UNIQUE (invoice_id, line_index)` constraint doubles as the primary lookup index for ordered line item retrieval per invoice.
