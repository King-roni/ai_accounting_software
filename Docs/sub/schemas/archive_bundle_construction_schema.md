# Archive Bundle Construction Schema

**Category:** Schemas · **Owning block:** 15 — Finalization & Secure Archive · **Stage:** 4 sub-doc (Layer 2)

This document specifies how the archive bundle is constructed during the finalization sequence. It defines the two-pass manifest construction protocol, the bundle file-naming convention, evidence cross-version deduplication, pass failure handling, and manifest version increment rules. All archive tooling in the `archive` namespace binds to this specification.

---

## Purpose

The archive bundle is the immutable, sealed evidence record for a finalized accounting period. The two-pass construction protocol guarantees that the bundle never seals over a missing or corrupted file. Any deviation — a missing object, a hash mismatch, a partial write — is caught before Object Lock is applied. Once sealed, the bundle cannot be modified; errors discovered after sealing require a new manifest version following the adjustment workflow.

---

## Bundle file-naming convention

The Object Storage key for a sealed bundle follows a fixed pattern:

```
<business_id>/<period_year>-<period_month>/<workflow_run_id>/bundle-v<manifest_version>.zip
```

Rules:
- All path components are lowercase.
- Separators within the date segment use a hyphen, not an underscore.
- `<period_year>` is a four-digit calendar year (e.g., `2025`).
- `<period_month>` is a zero-padded two-digit month (e.g., `03`).
- `<workflow_run_id>` is the UUID v7 of the workflow run, formatted in standard 8-4-4-4-12 hyphenated notation, lowercase.
- `<manifest_version>` is a monotonically increasing integer, starting at `1` for the first attempt on a given `workflow_run_id`. It is never zero-padded (e.g., `bundle-v1.zip`, not `bundle-v01.zip`).

Example:

```
3f2504e0-4f89-11d3-9a0c-0305e82c3301/2025-03/019186f4-7c3a-7000-8000-000000000001/bundle-v1.zip
```

If a hold is encountered and a re-attempt creates a new manifest version, the path increments only the `bundle-v<n>` segment. All other components remain identical for the same `workflow_run_id`.

## Two-pass manifest construction

### Overview

Bundle construction is a two-pass sequential process. Pass 2 may not begin until Pass 1 reports success. The bundle does not proceed to sealing until Pass 2 reports success.

```
Pass 1: Collect references → generate preliminary manifest
Pass 2: Verify presence + compute SHA-256 hashes → finalize manifest
           ↓
        Sealing (lock sequence)
```

### Pass 1 — Reference collection

Pass 1 collects all document, evidence, and ledger export references that belong to the current `workflow_run_id` and `manifest_version`. Sources:

- All `documents` rows where `workflow_run_id` matches and `document_state = PROCESSED`.
- All `evidence_pdfs` rows linked to transactions in the run's scope.
- The period ledger export (generated by `report.generate_period_report` before bundle construction begins).
- The audit log slice for the run (extracted by `archive.extract_audit_slice`).

For each reference, Pass 1 writes a preliminary manifest entry containing:

| Field | Value at Pass 1 |
|---|---|
| `object_key` | Object Storage key of the file |
| `file_path_in_bundle` | Relative path within the ZIP |
| `sha256_hex` | `null` — not yet computed |
| `size_bytes` | `null` — not yet verified |
| `evidence_inherited_from_version` | Populated if applicable (see dedup section) |

On completion, `ARCHIVE_BUNDLE_PASS1_COMPLETED` is emitted. The preliminary manifest is persisted to `archive.archive_manifests` with `pass_status = PASS1_COMPLETE`.

If Pass 1 finds any document reference where the Object Storage object is absent (key returns 404 or equivalent), the run transitions to `REVIEW_HOLD` before Pass 2 is attempted. Pass 2 does not begin while any reference is unresolved. A `REVIEW_HOLD` at this stage creates a new review issue of type `ARCHIVE_MISSING_OBJECT` with severity HIGH.

### Pass 2 — Verification and hash computation

Pass 2 iterates over every entry in the preliminary manifest. For each entry:

1. Retrieve the object from Object Storage using the `object_key`.
2. Stream the bytes and compute SHA-256.
3. Compare the computed hash against the hash stored on the source row (`documents.evidence_hash`, `evidence_pdfs.sha256_hex`, etc.).
4. Record `sha256_hex` and `size_bytes` on the manifest entry.

If any hash comparison fails — the computed value does not match the stored value on the source row — `ARCHIVE_BUNDLE_INTEGRITY_FAILED` is emitted with the `object_key`, `expected_sha256_hex`, and `computed_sha256_hex` in the payload. The bundle construction is aborted. The manifest row is updated to `pass_status = PASS2_FAILED`. A `REVIEW_HOLD` is raised.

If all hashes pass, `ARCHIVE_BUNDLE_PASS2_COMPLETED` is emitted. The manifest row is updated to `pass_status = PASS2_COMPLETE`. Bundle construction proceeds to ZIP assembly and sealing.

### Pass failure handling summary

| Failure point | Action | Run status |
|---|---|---|
| Pass 1: missing Object Storage object | Emit nothing; raise `ARCHIVE_MISSING_OBJECT` review issue; block Pass 2 | `REVIEW_HOLD` |
| Pass 2: SHA-256 mismatch | Emit `ARCHIVE_BUNDLE_INTEGRITY_FAILED`; abort bundle; set `pass_status = PASS2_FAILED` | `REVIEW_HOLD` |

## Evidence cross-version deduplication

A document uploaded in period N may be referenced again in period N+1 (for example, a supplier contract that spans multiple periods). In this case:

- The manifest entry for period N+1 references the document by its **original** `object_key` in Object Storage.
- The file is **not** copied or re-uploaded. No second object is created.
- The manifest entry carries `evidence_inherited_from_version: <original_manifest_version_id>`, where `original_manifest_version_id` is the UUID of the manifest version in which the document was first included.
- Pass 2 still verifies the object's presence and recomputes its SHA-256 for the current pass. Deduplication does not exempt an entry from verification.

The `evidence_inherited_from_version` field is `null` for documents first introduced in the current manifest version.

Downstream bundle readers must treat an inherited reference as equivalent to a direct reference — the ZIP includes the file, sourced from Object Storage, regardless of which manifest version first recorded it.

## Manifest version increment

Each re-attempt after a `REVIEW_HOLD` creates a new manifest version. Rules:

- Manifest versions are integers, monotonically increasing, scoped to a `workflow_run_id`.
- The first attempt is always version `1`.
- Version numbers are never reused or skipped.
- A new version is created by inserting a new `archive.archive_manifests` row; the previous version row is not modified.
- The event `FINALIZATION_MANIFEST_VERSION_INCREMENTED` is emitted on each version creation after version 1. This event already exists in the taxonomy.
- The bundle file-naming `bundle-v<n>` segment reflects the manifest version of the successful (sealed) bundle.
- If version 3 is the first to reach sealing, the stored bundle key ends in `bundle-v3.zip`; versions 1 and 2 have no corresponding sealed ZIP.

## Audit events

| Event | Severity | Emission point |
|---|---|---|
| `ARCHIVE_BUNDLE_PASS1_COMPLETED` | LOW | End of Pass 1, all references collected, no missing objects |
| `ARCHIVE_BUNDLE_PASS2_COMPLETED` | LOW | End of Pass 2, all hashes verified |
| `ARCHIVE_BUNDLE_INTEGRITY_FAILED` | BLOCKING | Pass 2 detects a SHA-256 mismatch on any entry |

`ARCHIVE_BUNDLE_INTEGRITY_FAILED` (BLOCKING) — emitted by `archive.construct_bundle` when the SHA-256 computed from the retrieved object bytes does not match the hash stored on the source row. BLOCKING because a hash mismatch indicates either object corruption or a tampered source record, and neither scenario is safe to finalize. Payload: `archive_manifest_id`, `workflow_run_id`, `business_id`, `object_key`, `expected_sha256_hex`, `computed_sha256_hex`, `manifest_version`.

`ARCHIVE_BUNDLE_PASS1_COMPLETED` (LOW) — emitted when all document references have been collected and no missing objects were detected. Payload: `archive_manifest_id`, `workflow_run_id`, `business_id`, `manifest_version`, `entry_count`.

`ARCHIVE_BUNDLE_PASS2_COMPLETED` (LOW) — emitted when all entries have been verified and hash computation is complete. Payload: `archive_manifest_id`, `workflow_run_id`, `business_id`, `manifest_version`, `entry_count`, `total_size_bytes`.

---

## Relationship to the lock sequence

Bundle construction (two-pass) is Steps 1–2 of the 5-step lock sequence defined in `lock_sequence_policies.md`. The lock sequence does not advance to Step 3 (ZIP sealing) until `ARCHIVE_BUNDLE_PASS2_COMPLETED` has been emitted for the current manifest version. Steps 3–5 (seal, RFC 3161 timestamp, Object Lock) proceed only after a valid pass-2-complete manifest exists.
