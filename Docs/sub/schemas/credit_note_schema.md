# credit_note_schema

**Category:** Schemas · **Owning block:** 13 — IN Workflow + Invoice Generator · **Stage:** 4 sub-doc (Layer 2)

The `credit_notes` table is the dedicated operational record for credit notes issued against tax invoices. It supplements the `invoices` table (which stores `CREDIT_NOTE`-typed rows for the invoice lifecycle) with structured fields specific to credit note management: gap-free sequence numbers, voiding, and cap enforcement. All conventions in this document bind to `data_layer_conventions_policy` (UUID v7, SHA-256 hashing, canonical JSON serialization, decimal-precise currency).

---

## Table definition

```sql
CREATE TYPE credit_note_status_enum AS ENUM (
  'DRAFT',
  'ISSUED',
  'VOIDED'
);

CREATE TABLE credit_notes (
  credit_note_id         uuid PRIMARY KEY DEFAULT gen_uuid_v7(),
  business_id            uuid NOT NULL REFERENCES business_entities(id),

  -- Source invoice
  invoice_id             uuid NOT NULL REFERENCES invoices(invoice_id),

  -- Number (allocated at ISSUED time; NULL while DRAFT)
  credit_note_number     text,    -- CN-YYYY-NNNN; gap-free per business per year

  -- Dates
  issue_date             date NOT NULL,

  -- Amounts (decimal-precise; positive value representing amount being credited)
  amount_eur             numeric(15,2) NOT NULL
    CHECK (amount_eur > 0),
  vat_amount_eur         numeric(15,2) NOT NULL DEFAULT 0
    CHECK (vat_amount_eur >= 0),

  -- Reason (mandatory)
  reason                 text NOT NULL,

  -- Lifecycle
  status                 credit_note_status_enum NOT NULL DEFAULT 'DRAFT',
  issued_at              timestamptz,
  voided_at              timestamptz,
  voided_by_user_id      uuid REFERENCES users(id),

  -- Authorship
  created_at             timestamptz NOT NULL DEFAULT now(),
  created_by_user_id     uuid REFERENCES users(id),

  -- Constraints
  CONSTRAINT issued_fields_require_issued_status
    CHECK (
      (issued_at IS NULL) OR (status IN ('ISSUED', 'VOIDED'))
    ),
  CONSTRAINT voided_fields_require_voided_status
    CHECK (
      (voided_at IS NULL AND voided_by_user_id IS NULL)
      OR status = 'VOIDED'
    ),
  CONSTRAINT credit_note_number_set_when_issued
    CHECK (
      credit_note_number IS NULL OR status IN ('ISSUED', 'VOIDED')
    )
);
```

## Column rationale

### `credit_note_id` (UUID v7)

Primary key generated by `gen_uuid_v7()` (Block 04 Phase 01 helper). This is a business-data table; UUID v7 is correct per `data_layer_conventions_policy`. UUID v4 is not used here.

### `invoice_id` (FK to `invoices`)

References the tax invoice this credit note offsets. A credit note cannot be issued against a `PRO_FORMA` invoice — that is enforced at the application layer by `in_workflow.issue_credit_note`, which validates `invoices.invoice_type = 'TAX_INVOICE'` before creating the credit note row. A credit note cannot be issued against a `DRAFT` invoice — enforced by the same tool checking that the source invoice has exited `DRAFT` status.

### `credit_note_number` (text, nullable)

Follows the `CN-YYYY-NNNN` format. NULL while in `DRAFT` status. The sequence number is allocated at `ISSUED` time via the write path described in Section 3. Sequence numbers are gap-free and per-business per-year per `invoice_sequence_schema`. Once allocated, the number is immutable.

### `amount_eur` (numeric(15,2))

Positive value representing the amount being credited. The amount does not include VAT; `vat_amount_eur` is declared separately. `amount_eur` must not exceed the `total_amount` of the source invoice (`invoices.invoice_id`). This cumulative cap is enforced as described in Section 4. Currency amounts are stored as `numeric` (never float) per `data_layer_conventions_policy`.

### `reason` (text, NOT NULL)

Mandatory free-text description of why the credit note was issued. Blank strings are rejected at the application layer. This field is included in the `CREDIT_NOTE_ISSUED` audit payload.

### `status` (credit_note_status_enum)

Three-value enum:

| Status | Semantics |
| --- | --- |
| `DRAFT` | Credit note created but not yet issued; no sequence number allocated |
| `ISSUED` | Credit note issued; `CN-YYYY-NNNN` number allocated; sent or recorded against the source invoice |
| `VOIDED` | Credit note voided after issuance; requires Owner/Admin and step-up MFA |

`VOIDED` is terminal for the credit note row itself. Voiding does not retroactively change the source invoice's payment state; if the source invoice was moved to `CREDITED`, that transition must be addressed via an adjustment run if required.

### `voided_by_user_id`

Populated only when `status = VOIDED`. The voiding user must hold Owner or Admin role. See Section 5.

## Sequence number allocation

The `credit_note_number` (format `CN-YYYY-NNNN`) is allocated via the `in_workflow.next_invoice_number` tool at the moment the credit note transitions from `DRAFT` to `ISSUED`. The write path:

1. `in_workflow.next_invoice_number` calls `SELECT ... FOR UPDATE` on the `invoice_sequences` row for `(business_id, 'CREDIT_NOTE', year)`.
2. Increments `last_sequence_number`.
3. Formats the number as `CN-<year>-<zero-padded-4-digit-sequence>`.
4. Writes the number to `credit_notes.credit_note_number` in the same transaction as the `ISSUED` status update.

This is the **only** permitted write path for sequence allocation. Direct UPDATE of `credit_note_number` from application code is forbidden. See `invoice_sequence_schema` for the full sequence table definition.

Sequence numbers are never reused or decremented. Voiding an issued credit note does not release or decrement the sequence number — the `CN-YYYY-NNNN` number is retired but not recycled. This is a Cyprus VAT compliance requirement.

---

## Cumulative cap enforcement

The total `amount_eur` across all non-voided credit notes issued against a single source invoice must not exceed the source invoice's `total_amount`. This invariant is enforced via row-level locking:

1. `in_workflow.issue_credit_note` acquires `SELECT ... FOR UPDATE` on the source `invoices` row.
2. Reads `SUM(amount_eur) WHERE invoice_id = $source AND status != 'VOIDED'` from `credit_notes`.
3. Validates that `existing_sum + new_amount_eur <= invoices.total_amount`.
4. On violation, returns `INVOICE_CREDIT_NOTE_CAP_REJECTED` and no credit note row is written.

A database-level `CHECK` constraint alone cannot enforce this cross-row aggregate; the row-lock pattern is required.

For the cap enforcement schema detail, reference `credit_note_cumulative_cap_schema` (Layer 1).

---

## Voiding rules

1. Only Owner or Admin may void an issued credit note.
2. Step-up MFA (`WORKFLOW_APPROVE` surface equivalent for invoice management — `INVOICE_MANAGE` with step-up) is required.
3. A `DRAFT` credit note may be deleted (not voided) by Owner, Admin, or Bookkeeper without step-up.
4. A `VOIDED` credit note cannot be reinstated; the voiding is terminal.
5. Voiding emits `CREDIT_NOTE_VOIDED` (MEDIUM severity).

CREATE POLICY credit_notes_isolation ON credit_notes
  FOR ALL
  USING (business_id = ANY(auth.business_ids_for_session()));

## Permission gate

Read: any role with business access may SELECT credit note rows.
Write (create, issue): Owner, Admin, Bookkeeper via `INVOICE_MANAGE` surface.
Write (void): Owner, Admin only.

Mobile write rejection: all write operations are rejected from `client_form_factor = MOBILE` before the permission check, emitting `MOBILE_WRITE_REJECTED`. See `mobile_write_rejection_endpoints`.

## Audit events

| Event | Severity | Trigger |
| --- | --- | --- |
| `CREDIT_NOTE_ISSUED` | LOW | Credit note transitions to `ISSUED`; sequence number allocated |
| `CREDIT_NOTE_VOIDED` | MEDIUM | Credit note transitions to `VOIDED` by Owner/Admin |
| `INVOICE_CREDIT_NOTE_CAP_REJECTED` | MEDIUM | Cap enforcement blocks issuance |

`CREDIT_NOTE_ISSUED` payload (canonical JSON per `data_layer_conventions_policy`): `credit_note_id`, `business_id`, `invoice_id`, `credit_note_number`, `amount_eur` (decimal string), `reason`, `issued_at`.
