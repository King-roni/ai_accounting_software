# out_config_schema

**Category:** Schemas Â· **Owning block:** 12 â€” OUT Workflow Â· **Stage:** 4 sub-doc (Layer 2)

The `out_workflow_configs` table holds one configuration row per business for both `OUT_MONTHLY` and `OUT_ADJUSTMENT` runs. It is the authoritative record of per-business toggles that govern which phases run, how long hold reminders fire, and how far back adjustments may reach. All columns carry explicit defaults so that a freshly provisioned business is fully operational without a manual configuration step.

---

## Table definition

```sql
CREATE TABLE out_workflow_configs (
  config_id                         uuid PRIMARY KEY DEFAULT gen_uuid_v7(),
  business_id                       uuid NOT NULL UNIQUE REFERENCES business_entities(id),

  -- Hold reminder
  manual_upload_hold_reminder_days  integer NOT NULL DEFAULT 7
    CHECK (manual_upload_hold_reminder_days >= 1),

  -- Adjustment lookback cap (Cyprus VAT retention window = 6 years)
  out_adjustment_max_lookback_years integer NOT NULL DEFAULT 6
    CHECK (out_adjustment_max_lookback_years >= 1 AND out_adjustment_max_lookback_years <= 6),

  -- Event-driven trigger
  auto_trigger_on_statement_upload  boolean NOT NULL DEFAULT true,

  -- Phase subset (null = all phases enabled; explicit array = named subset only)
  enabled_phases                    jsonb DEFAULT NULL,

  created_at                        timestamptz NOT NULL DEFAULT now(),
  updated_at                        timestamptz NOT NULL DEFAULT now(),
  updated_by_user_id                uuid REFERENCES users(id)
);
```

## Column rationale

### `config_id` (UUID v7)

Primary key generated by `gen_uuid_v7()` (Block 04 Phase 01 helper). UUID v7 is used here because this is a business-data table. Session or token IDs would use UUID v4 per `data_layer_conventions_policy`.

### `business_id` (UNIQUE)

The `UNIQUE` constraint enforces the invariant that exactly one config row exists per business. The bootstrap loader (Block 12 Phase 01's `loadOutWorkflowConfigForBusiness`) inserts this row at business-provisioning time and is idempotent â€” a second call is a no-op.

### `manual_upload_hold_reminder_days` (int, default 7)

Controls how many calendar days after a manual-upload hold begins before a reminder notification is dispatched to the business owner. Minimum 1. Stage 1 default 7 matches the `decisions_log.md` entry: "MANUAL_UPLOAD_HOLD timeout: reminder after 7 days, no auto-action."

The run sits indefinitely until the user uploads or documents an exception; this column sets only the first-reminder cadence.

### `out_adjustment_max_lookback_years` (int, default 6)

The maximum number of calendar years prior to today that an `OUT_ADJUSTMENT` run may target. Default 6 matches the Cyprus VAT statutory retention requirement (6-year books retention). The engine enforces this cap at `OUT_ADJUSTMENT` run creation time; any attempt to target a period older than `now() - interval '? years'` where `?` is this value is rejected with a structured error before a run row is written.

The CHECK constraint prevents a value above 6 â€” a config row cannot extend an adjustment lookback beyond the statutory retention window that the rest of the system is designed around.

### `auto_trigger_on_statement_upload` (boolean, default true)

When `true`, the event handler for `STATEMENT_UPLOAD_COMPLETED` automatically creates an `OUT_MONTHLY` run. When `false`, the event arrives but is swallowed silently (audit event `OUT_WORKFLOW_AUTO_START_SUPPRESSED` is still emitted). The user must then trigger the run manually via `out_workflow.start_run_manually`.

This flag is symmetric with `in_workflow_business_config.auto_start_on_statement_upload` on the IN side (Block 13 Phase 07).

### `enabled_phases` (JSONB, nullable)

Null means all phases in the registered `OUT_MONTHLY` phase sequence are active for this business. A non-null value is a JSON array of phase-name strings (e.g., `["INGESTION","CLASSIFICATION","OUT_FILTER","LEDGER_PREPARATION","AI_END_SCAN","FINALIZATION"]`). Phases not in the array are skipped at runtime; the engine emits `OUT_WORKFLOW_PHASE_SKIPPED_BY_CONFIG` for each skipped phase.

Validation: the engine asserts at run-start that every element in `enabled_phases` is a known registered phase name. An invalid phase name raises a `BLOCKING` review issue and halts the run before the first phase executes.

### `updated_by_user_id`

Nullable FK to `users`. Set on every `out_workflow_configs` UPDATE via the settings API. The combination of `updated_at` + `updated_by_user_id` is sufficient for the audit trail inside the `out_workflow_configs` table itself; the full before/after diff is preserved in the business-scoped audit log via `OUT_WORKFLOW_CONFIG_UPDATED`.

## Indexes

```sql
CREATE INDEX idx_out_workflow_configs_business
  ON out_workflow_configs(business_id);
```

The unique constraint on `business_id` already functions as a unique index for equality lookups. The explicit index is redundant for the primary lookup but makes intent clear for range scans during analytics.

## Permission gate

Read: any role with business access may SELECT the config row.
Write: Owner and Admin only, via the `BUSINESS_SETTINGS_EDIT` permission surface (Block 02 Phase 04 / 2026-05-09 amendment). Bookkeeper, Accountant, Reviewer, and Read-only roles receive a 403 on any mutation attempt.

Mobile write rejection: write surfaces for this table are in `mobile_write_rejection_endpoints`. Any mutation arriving from a `client_form_factor = MOBILE` session is rejected before the permission check with audit event `MOBILE_WRITE_REJECTED`. Mobile clients may read the config.

## Audit events

| Event | Trigger |
| --- | --- |
| `OUT_WORKFLOW_CONFIG_INITIALIZED` | Bootstrap loader creates the initial row at business provisioning |
| `OUT_WORKFLOW_CONFIG_UPDATED` | Any field update via the settings API; payload carries field-level before/after diff |

Both events use domain `OUT_WORKFLOW` per `audit_log_policies`. Payloads are canonical JSON per `data_layer_conventions_policy`.

## Evolution rules

1. New config columns must carry a NOT NULL DEFAULT equal to the MVP default value so that existing rows remain valid without a backfill migration.
2. A change to the type, constraints, or semantics of an existing column is a backward-incompatible migration and requires: (a) a migration file, (b) a version bump in the `effective_phase_sequence_json` snapshot shape if the column affects phase selection, and (c) an amendment in `Docs/decisions_log.md` if it alters the Stage 1 default.
3. The `out_adjustment_max_lookback_years` cap must never exceed 6 without a Cyprus VAT retention-policy change recorded in `decisions_log.md`.

## Bootstrap contract

`loadOutWorkflowConfigForBusiness(business_id)` is called by Block 02 Phase 01's business-provisioning flow. It is idempotent: if a row already exists for `business_id`, the function returns without modification and without emitting `OUT_WORKFLOW_CONFIG_INITIALIZED` a second time.

## Settings API shape

The settings API follows the standard pattern from Block 02 Phase 11:

```ts
outConfig.get({ business_id: string }) â†’ OutWorkflowConfig
outConfig.update({ business_id: string, patch: Partial<OutWorkflowConfig> }) â†’ OutWorkflowConfig
```

`patch` accepts any subset of the mutable columns: `manual_upload_hold_reminder_days`, `auto_trigger_on_statement_upload`, `enabled_phases`. The columns `config_id`, `business_id`, and `created_at` are immutable and are rejected if present in `patch`.

Every `update` call sets `updated_at = now()` and `updated_by_user_id` to the calling user's ID, then emits `OUT_WORKFLOW_CONFIG_UPDATED` with the field-level diff in the audit payload.
