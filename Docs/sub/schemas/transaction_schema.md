# Transaction Schema

**Category:** Schemas · **Owning block:** 07 — Bank Statement Pipeline / 08 — Transaction Classification · **Stage:** 4 sub-doc (Layer 2)

Canonical DDL and supporting conventions for the `transactions` table. This table is the primary operational record for every bank statement row that has passed parsing, normalisation, deduplication, and zone promotion. Classification, matching, and ledger tooling all write to and read from this table.

---

## Purpose

The `transactions` table is the central data surface shared across Blocks 07, 08, 10, and 11. It carries the normalised representation of a parsed bank statement row, enriched progressively as each workflow phase completes:

- Block 07 ingestion populates the base row fields through zone promotion.
- Block 08 classification writes `category`, `classification_confidence`, `classification_tier_used`.
- Block 10 matching writes `is_reconciled` and `effective_match_status`.
- Block 11 ledger reads the row for VAT and ledger entry preparation.

---

## Table DDL

```sql
CREATE TABLE transactions (
  id                          UUID            NOT NULL DEFAULT gen_uuid_v7(),
  business_id                 UUID            NOT NULL REFERENCES business_entities(id),
  workflow_run_id             UUID            NOT NULL REFERENCES workflow_runs(id),
  bank_upload_id              UUID            NOT NULL REFERENCES bank_uploads(id),

  -- Core transaction fields
  value_date                  DATE            NOT NULL,
  booking_date                DATE,
  amount_eur                  NUMERIC(15, 2)  NOT NULL,
  original_amount             NUMERIC(15, 2),
  original_currency           CHAR(3),
  fx_rate_used                NUMERIC(18, 8),

  -- Counterparty fields
  counterparty_name           TEXT,
  counterparty_iban           TEXT,

  -- Raw description
  description_raw             TEXT,

  -- Transaction type enum
  transaction_type            transaction_type_enum  NOT NULL,

  -- Classification fields (populated by Block 08)
  category                    TEXT,
  classification_confidence   NUMERIC(5, 4),
  classification_tier_used    TEXT,

  -- Deduplication and match status
  effective_match_status      dedup_status_enum  NOT NULL,
  prior_match_status          dedup_status_enum,
  dedup_fingerprint           TEXT            NOT NULL,

  -- Reconciliation flag
  is_reconciled               BOOLEAN         NOT NULL DEFAULT FALSE,

  -- Timestamps
  created_at                  TIMESTAMPTZ     NOT NULL DEFAULT now(),
  updated_at                  TIMESTAMPTZ     NOT NULL DEFAULT now(),

  PRIMARY KEY (id)
);
```

### Enum: `transaction_type_enum`

```sql
CREATE TYPE transaction_type_enum AS ENUM (
  'CREDIT_TRANSFER',
  'DIRECT_DEBIT',
  'FEE',
  'INTEREST',
  'REVERSAL',
  'INTERNAL_TRANSFER',
  'UNKNOWN'
);
```

### Enum: `dedup_status_enum`

```sql
CREATE TYPE dedup_status_enum AS ENUM (
  'NEW',
  'DUPLICATE_EXACT',
  'DUPLICATE_PROBABLE',
  'NEEDS_REVIEW'
);
```

## Field notes

### `id`

Primary key. Generated by `gen_uuid_v7()` (Postgres-side, per `data_layer_conventions_policy`). Monotonically increasing within ~1 ms precision; suitable for time-range scans without a secondary index on `created_at`.

### `business_id`

Foreign key to `business_entities.id`. Not nullable. All RLS policies use this column as the tenant discriminator. Tenant isolation is enforced exclusively here; no tenancy information is derived from the `id` column.

### `workflow_run_id`

Foreign key to `workflow_runs.id`. Identifies the workflow run that produced this row. Used for run-scoped queries and for zone promotion eligibility checks.

### `bank_upload_id`

Foreign key to `bank_uploads.id`. Identifies the specific bank upload file that contained this row. Multiple uploads within the same run may contribute rows; this field distinguishes which upload each row came from.

### `value_date`

The date on which the transaction was valued by the bank. Not nullable. Used as the primary date for period assignment, deduplication, and ledger entry date.

### `booking_date`

The date on which the transaction was booked by the bank. Nullable; not all SEPA CSV formats include a separate booking date. When absent, consumers fall back to `value_date`.

### `amount_eur`

The transaction amount in EUR, normalised from the source row. Two decimal places. Positive = inflow (credit); negative = outflow (debit). FX conversion is applied before this field is set; see `fx_rate_used`.

### `original_amount` / `original_currency` / `fx_rate_used`

Populated only when the source transaction was denominated in a non-EUR currency. `original_currency` is the ISO 4217 three-character code. `fx_rate_used` is the ECB rate applied at conversion time, stored to 8 decimal places for auditability. When the source is already EUR, all three fields are `NULL`.

### `counterparty_name`

Raw counterparty name from the bank statement row, after whitespace normalisation but before any further standardisation. Nullable because some bank formats omit the name field. Used as a fuzzy-match input by the Matching Engine and as a lookup key by the Vendor Memory classifier.

### `counterparty_iban`

The IBAN of the counterparty, if present in the source row. Nullable. Used in deduplication fingerprint derivation and counterparty resolution.

### `description_raw`

The raw remittance information or free-text description from the source row. Nullable. May contain invoice reference numbers (e.g. `INV-2026-0042`) that the Matching Engine extracts as `reference_hint` values. Not modified after initial write.

### `transaction_type`

Set during row normalisation (Block 07 Phase 04). Mapped from the bank format's transaction type field per `bank_format_sepa_spec.md`. Default `UNKNOWN` when the source value cannot be mapped.

### `category`

Set by Block 08 classification. The human-readable category label assigned by the winning classification layer (L1 rule, L2 vendor memory, or L3 AI). Nullable before classification runs.

### `classification_confidence`

Numeric confidence in the `category` assignment, in the range `[0.0000, 1.0000]`. Four decimal places. Nullable before classification. Set even when a `confidence_override` is applied by an L1 rule — the override value is used.

### `classification_tier_used`

The classification tier that produced the final `category` value. Values: `LAYER_1`, `LAYER_2`, `LAYER_3`. Nullable before classification. Used for analytics and calibration.

### `effective_match_status`

Current match/dedup status. Set to `NEW` on initial insert for rows that passed the hard-duplicate check. Updated by the deduplication engine and the Matching Engine.

### `prior_match_status`

The previous value of `effective_match_status` before the most recent update. Nullable. Retained for audit trail purposes; allows queries to see transitions without reading the full audit log.

### `dedup_fingerprint`

SHA-256 hex string (64 chars, lowercase) used for deduplication. Derivation rules are in the section below. Computed once at insert time and never updated.

### `is_reconciled`

Set to `TRUE` by the Matching Engine when the transaction has been matched and the match confirmed (auto or human). Used by the ledger phase as a precondition check.

## `dedup_fingerprint` derivation

The fingerprint is a SHA-256 hex digest of the concatenation of the following fields:

```
SHA-256(
  business_id          ||
  value_date::text     ||          -- ISO 8601 format: YYYY-MM-DD
  amount_eur::text     ||          -- two decimal places, e.g. "1200.00"
  (counterparty_iban ?? '')  ||
  normalise(description_raw ?? '')
)
```

`normalise()` is defined as: lowercase, collapse all whitespace sequences to a single space, strip all characters outside `[a-z0-9 ]`. This is deterministic across runs, replicas, and library versions per `data_layer_conventions_policy`.

The fingerprint is unique per `business_id`. The UNIQUE index `(business_id, dedup_fingerprint)` enforces this at the database layer; a second insert with an identical fingerprint for the same business is rejected. See `deduplication_policy.md` for the HARD vs. SOFT duplicate distinction.

## Indexes

```sql

CREATE INDEX idx_transactions_business_run
  ON transactions (business_id, workflow_run_id);

CREATE UNIQUE INDEX idx_transactions_dedup_fingerprint
  ON transactions (business_id, dedup_fingerprint);

CREATE INDEX idx_transactions_value_date
  ON transactions (value_date);

-- Match status queries (Matching Engine, Review Queue)
CREATE INDEX idx_transactions_effective_match_status
  ON transactions (business_id, effective_match_status);
```

## Row-level security

RLS is enabled on `transactions`. Policies enforce tenant isolation by `business_id`.

```sql
ALTER TABLE transactions ENABLE ROW LEVEL SECURITY;

-- Authenticated users may SELECT and UPDATE their own business's rows.
CREATE POLICY transactions_select
  ON transactions FOR SELECT
  USING (business_id = auth.current_business_id());

CREATE POLICY transactions_update
  ON transactions FOR UPDATE
  USING (business_id = auth.current_business_id());

-- INSERT is performed only by service-role functions (intake pipeline).
-- DELETE is not permitted at any role level.
```

No `DELETE` policy exists. Rows are never hard-deleted from `transactions`. Soft-deletion patterns are not used; rows remain for the full data zone retention period. Regulatory retention requirements (7-year Operational zone) prohibit deletion.

---

## Data zone lifecycle

Rows are created in the **Processing zone** during the active workflow run (7-day TTL post-run completion, per `data_layer_conventions_policy`). On `FINALIZATION`, the zone promotion pipeline transitions rows to the **Operational zone** (7-year retention post-business-deactivation). After period finalization, rows are included in the **Archive zone** bundle (permanent Object Lock). Rows are never written to the Export temp zone.

Zone tracking is managed by the `data_zone` column on the parent `workflow_runs` row and the zone promotion pipeline (Block 04 Phase 08). The `transactions` table itself does not carry an explicit zone column; zone membership is derived from the run status.
