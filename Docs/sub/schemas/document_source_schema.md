# document_source_schema

**Category:** Schemas · **Owning block:** 09 — Document Intake · **Stage:** 4 sub-doc (Layer 2)

This sub-doc defines the `document_sources` table, which records the origin of every document discovered by the intake pipeline. Each row represents one source path through which a document entered the system — email, Google Drive, or manual upload. A single document may have multiple source rows when the same file hash arrives through more than one path (cross-source deduplication per Block 09 Phase 08). The Gmail query template JSONB schema and sender-allowlist matching rules are defined here as the canonical source.

---

## Table definition

```sql
CREATE TYPE document_source_type_enum AS ENUM (
  'EMAIL',
  'DRIVE',
  'MANUAL_UPLOAD'
);

CREATE TABLE document_sources (
  source_id                 uuid        PRIMARY KEY DEFAULT gen_uuid_v7(),
  document_id               uuid        NOT NULL REFERENCES documents(document_id),
  business_id               uuid        NOT NULL REFERENCES business_entities(id),
  source_type               document_source_type_enum NOT NULL,
  source_metadata           jsonb       NOT NULL,
  discovered_at             timestamptz NOT NULL DEFAULT now(),

  -- Prevent recording the same source twice for the same document
  CONSTRAINT uq_document_source_external
    UNIQUE (document_id, source_type, (source_metadata->>'external_id'))
);
```

### Column notes

- `source_id` — UUID v7 per `data_layer_conventions_policy §2`.
- `document_id` — FK to `documents.document_id` (Block 04 Phase 03). A `document_sources` row is always created in the same transaction as or after the `documents` row it references.
- `source_type` — the intake channel; drives the expected `source_metadata` shape.
- `source_metadata` — JSONB; shape validated at write time by `intake.record_document_source` per the per-`source_type` schemas below.
- `discovered_at` — timestamp when the intake pipeline first observed this source. For email: when the Gmail finder ran. For Drive: when the Drive finder ran. For manual: when the upload was received.
- Uniqueness constraint uses `source_metadata->>'external_id'` — a derived virtual column expression. This catches re-recording the same Gmail message, Drive file, or upload session for the same document.

## `source_metadata` shapes per `source_type`

All metadata objects use canonical JSON key ordering (RFC 8785 lexical sort) per `data_layer_conventions_policy §3` when stored. The validation layer normalizes on write.

### `EMAIL`

```json
{
  "external_id": "<gmail_message_id>",
  "gmail_message_id": "<string>",
  "thread_id": "<string>",
  "sender_address": "<lowercase email address>",
  "subject": "<string>",
  "received_at": "<iso8601 timestamp>",
  "attachment_filename": "<string>",
  "query_template_used": {
    "template_name": "<string>",
    "raw_query": "<string>",
    "supplier_type": "<string | null>",
    "transaction_context": "<string | null>"
  }
}
```

**Field notes:**
- `external_id` must equal `gmail_message_id` (the unique constraint is on `external_id`; the explicit `gmail_message_id` field exists for readability in queries).
- `sender_address` is stored in lowercase (see sender-allowlist matching below).
- `query_template_used` captures the fixed-library query pattern that surfaced this document, per the Stage 1 decision: "Fixed library of query patterns per supplier type and transaction context. No per-call generative queries in MVP." The `raw_query` field is the instantiated query string (parameters substituted); `template_name` identifies which template from `gmail_search_query_templates` produced it.

### `DRIVE`

```json
{
  "external_id": "<drive_file_id>",
  "drive_file_id": "<string>",
  "file_name": "<string>",
  "parent_folder_id": "<string>",
  "folder_path": "<string>",
  "modified_at": "<iso8601 timestamp>"
}
```

**Field notes:**
- `external_id` equals `drive_file_id`.
- `folder_path` is the human-readable path string from Drive root to the parent folder (e.g., `Invoices/2026-05`). This is stored for audit readability; the authoritative reference is `parent_folder_id`.
- `modified_at` is the Drive file's last-modified timestamp as returned by the Drive API at discovery time. It does not reflect subsequent modifications (the document is immutable once ingested).

### `MANUAL_UPLOAD`

```json
{
  "external_id": "<upload_session_id>",
  "uploaded_by_user_id": "<uuid>",
  "original_filename": "<string>",
  "upload_session_id": "<string>"
}
```

**Field notes:**
- `external_id` equals `upload_session_id`.
- `upload_session_id` is generated by the upload endpoint at request time; it is the deduplication key for manual uploads. Two separate manual uploads of the same file by the same user produce two distinct session IDs, but the `documents` layer deduplicates by content hash (Block 09 Phase 08), so only one `documents` row exists; both `document_sources` rows may exist (recording both upload events).

## Gmail query template JSONB schema

Defined on `gmail_search_query_templates` (Block 09 Phase 01), the `pattern_jsonb` column holds a parameterized template. The `query_template_used` field in `EMAIL` source metadata is an instantiated copy. The fixed-library constraint (Stage 1 decision) means no template is generated at runtime; templates are seeded at deployment time.

The canonical template object shape:

```json
{
  "template_name": "invoice_by_supplier_domain",
  "raw_query": "from:billing@acme.com has:attachment newer_than:30d",
  "supplier_type": "SAAS_VENDOR",
  "transaction_context": "OUT_EXPENSE"
}
```

- `template_name` — unique identifier for the template within the template library.
- `raw_query` — the fully-instantiated Gmail search query. For global templates, parameters like `{supplier_domain}` are substituted at runtime using the counterparty's domain; the instantiated string is what appears in the source metadata.
- `supplier_type` — optional classification hint (e.g., `SAAS_VENDOR`, `UTILITY`, `PROFESSIONAL_SERVICES`). Used by the matching engine as a signal.
- `transaction_context` — optional transaction-type hint aligning with `transaction_type_enum` values. Used as a matching signal; not binding.

## Sender allowlist matching rules

The `business_sender_allowlist` table (Block 09 Phase 01) controls which email senders reach the intake pipeline. The matching logic is:

1. **Case-folding.** The incoming `sender_address` (from the Gmail message header) is folded to lowercase before matching. All allowlist entries are stored and compared lowercase (`UNIQUE (business_id, entry_kind, lower(value))` per Block 09 Phase 01).

2. **Match precedence — domain over exact address.** When a sender address matches both an `EMAIL_DOMAIN` entry (domain of the address) and an `EMAIL_ADDRESS` entry (full address), the `EMAIL_DOMAIN` match takes precedence. Rationale: domain entries represent a broad decision ("trust all invoices from google.com"); an exact-address entry in the same allowlist would be redundant. Domain-match-takes-precedence means that adding `EMAIL_DOMAIN = google.com` covers `billing@google.com` without needing an explicit address entry.

3. **Subdomain handling.** An `EMAIL_DOMAIN` entry of `acme.com` matches `billing@acme.com` and `invoices@mail.acme.com` (subdomains are included). The match is a suffix check on the domain part: `lower(sender_domain) = entry_value OR lower(sender_domain) LIKE ('%.' || entry_value)`.

4. **No allowlist entry — spam path.** If the sender matches neither an allowlist domain nor an exact address and is not already a known supplier in `recurring_vendor_memory`, the document is held at `INTAKE_PENDING_ALLOWLIST_REVIEW` status and a review issue is raised. The document does not reach the matching engine until a reviewer approves the sender or adds it to the allowlist.

## Indexes

```sql
-- Primary lookup: all sources for a document
CREATE INDEX idx_document_sources_document
  ON document_sources (document_id);

-- Business-scoped source lookup
CREATE INDEX idx_document_sources_business_type
  ON document_sources (business_id, source_type, discovered_at);

-- External ID lookup (for re-ingestion prevention)
CREATE INDEX idx_document_sources_external_id
  ON document_sources (source_type, (source_metadata->>'external_id'));
```

## RLS

```sql
CREATE POLICY document_sources_isolation ON document_sources
  FOR ALL
  USING (business_id = ANY (auth.business_ids_for_session()));
```

## Audit events

| Event | When | Severity |
|---|---|---|
| `DOCUMENT_SOURCE_INDEXED` | New `document_sources` row created via email or Drive finder | LOW |
| `DOCUMENT_MANUAL_UPLOAD_RECEIVED` | New `document_sources` row created via manual upload path | LOW |

Both events emitted via `emitAudit()` per `audit_log_policies` and exist in `audit_event_taxonomy`.

Note: `DOCUMENT_EMAIL_FINDER_RAN` and `DOCUMENT_DRIVE_FINDER_RAN` (existing taxonomy entries from Block 09) cover the finder-run aggregate events. The events defined here cover the per-document source-indexing outcome within a finder run.
