# evidence_pdf_schema

**Category:** Schemas · **Owning block:** 07 — Bank Statement Pipeline · **Stage:** 4 sub-doc (Layer 2)

This sub-doc defines the `evidence_pdfs` table, which records every PDF artefact generated from an accepted bank-statement transaction. Evidence PDFs are generated from structured `transactions` rows (Block 01 Principle 2 — structured data is the source of truth; the PDF is never the source). Each PDF is written to the `raw-uploads` storage bucket, hashed via SHA-256, and linked back to its originating transaction. The schema enforces immutability: a PDF row is never updated; re-generation produces a new versioned row with the old row retained for audit traceability.

---

## Table definition

```sql
CREATE TYPE ocr_status_enum AS ENUM (
  'PENDING',
  'RUNNING',
  'COMPLETED',
  'FAILED'
);

CREATE TABLE evidence_pdfs (
  evidence_id               uuid        PRIMARY KEY DEFAULT gen_uuid_v7(),
  business_id               uuid        NOT NULL REFERENCES business_entities(id),
  upload_id                 uuid        NOT NULL REFERENCES bank_statement_uploads(upload_id),
  storage_key               text        NOT NULL,
  original_filename         text        NOT NULL,
  content_type              text        NOT NULL,
  file_size_bytes           bigint      NOT NULL,
  sha256_hex                text        NOT NULL,
  ocr_status                ocr_status_enum NOT NULL DEFAULT 'PENDING',
  ocr_completed_at          timestamptz,
  page_count                integer,
  created_at                timestamptz NOT NULL DEFAULT now(),
  created_by_user_id        uuid        REFERENCES users(id),

  CONSTRAINT chk_storage_key_prefix
    CHECK (storage_key LIKE (business_id::text || '/' || evidence_id::text || '%')),
  CONSTRAINT chk_sha256_hex_length
    CHECK (length(sha256_hex) = 64),
  CONSTRAINT chk_file_size_positive
    CHECK (file_size_bytes > 0),
  CONSTRAINT chk_page_count_positive
    CHECK (page_count IS NULL OR page_count > 0),
  CONSTRAINT chk_ocr_completed_at_requires_terminal
    CHECK (
      ocr_completed_at IS NULL
      OR ocr_status IN ('COMPLETED', 'FAILED')
    )
);
```

## Column notes

- `evidence_id` — UUID v7 per `data_layer_conventions_policy §2`. Monotonically increasing; drives sort-order in evidence-history queries.
- `business_id` — tenant isolation anchor. Every RLS policy and index includes this column. Derived from the `upload_id`'s parent `statement_upload`; stored redundantly here to allow direct tenant-scoped queries without joining through the uploads table.
- `upload_id` — FK to `bank_statement_uploads.upload_id`. Records which statement upload triggered this evidence PDF's generation. A single upload produces multiple `evidence_pdfs` rows (one per accepted transaction row).
- `storage_key` — path in the `raw-uploads` bucket under the convention `{business_id}/{evidence_id}/{filename}`. The check constraint enforces the prefix, preventing cross-tenant key collisions. The actual file is written by `intake.generate_evidence_pdf` using the signed-URL upload pattern from Block 04 Phase 05.
- `original_filename` — the filename used when the PDF was written to storage (typically `evidence_{transaction_id}_{date}.pdf`). Stored for human-readable retrieval; the authoritative reference is `storage_key`.
- `content_type` — MIME type. Constrained to the allowed types declared in `upload_content_sniff_policy`. For evidence PDFs generated by the pipeline, the value is always `application/pdf`. For uploaded bank statement PDFs, the allowed set includes `image/png`, `image/jpeg`, and `image/tiff` in addition to `application/pdf` — the content-sniff validation step (Block 07 Phase 01) enforces membership before the row is written.
- `file_size_bytes` — byte count of the stored file. Must be positive; zero-byte files are rejected before the row is inserted.
- `sha256_hex` — SHA-256 hex digest of the file bytes, computed after content-sniff validation and before the upload to storage. Lowercase, 64 characters per `data_layer_conventions_policy §1`. Used for tamper-detection and cross-reference in `deduplication_fingerprint_schema`.
- `ocr_status` — lifecycle of the OCR pass run over this evidence file. Valid values: `PENDING | RUNNING | COMPLETED | FAILED`. OCR is run only for uploaded bank statement PDFs (Block 07 Phase 03 path); evidence PDFs generated programmatically by the pipeline start at `PENDING` and may be promoted to `COMPLETED` after a lightweight text-layer verification pass, or left `PENDING` if no OCR is required.
- `ocr_completed_at` — timestamptz populated when `ocr_status` transitions to `COMPLETED` or `FAILED`. Null for rows that have not yet reached a terminal OCR state.
- `page_count` — number of pages in the PDF, populated after OCR or PDF inspection. Null until the inspection step completes.
- `created_at` — insertion timestamp; immutable.
- `created_by_user_id` — nullable. Null for pipeline-generated evidence PDFs (no user actor). Populated for manually-uploaded PDFs. References `users.id`.

## Immutability rule

Once a row is inserted into `evidence_pdfs`, it is never updated or deleted. Corrections are never applied via `UPDATE`. If a PDF must be regenerated (e.g., after a transaction field is manually corrected by a user), a new `evidence_pdfs` row is inserted with an incremented logical version tracked in the `generated_from_transaction_version` field on the application layer. Both rows remain for audit traceability. The `sha256_hex` uniqueness is not DB-enforced across the table (different transactions can produce files with the same hash in edge cases); uniqueness is enforced at the `(upload_id, sha256_hex)` level by application logic in `intake.generate_evidence_pdf`.

---

## `ocr_status` transitions

```
PENDING → RUNNING → COMPLETED
                  → FAILED
```

The transition from `PENDING` to `RUNNING` is recorded when the OCR tool invocation starts. The transition to `COMPLETED` or `FAILED` is recorded when the invocation completes. Transitions are driven by `intake.ocr_and_extract`; no other tool may advance `ocr_status` directly.

---

## Allowed MIME types

Constrained by `upload_content_sniff_policy` (Block 07 Phase 01). Permitted values for `content_type`:

| `content_type` | Context |
|---|---|
| `application/pdf` | Generated evidence PDFs; uploaded PDF bank statements |
| `image/png` | Uploaded bank statement images |
| `image/jpeg` | Uploaded bank statement images |
| `image/tiff` | Uploaded bank statement images (multi-page TIFF) |

Any other MIME type causes the upload to be rejected before the `evidence_pdfs` row is written. The content-sniff validation step reads the file's magic bytes, not the declared `Content-Type` header.

---

## Mobile write rejection

Write operations to this table (via `intake.generate_evidence_pdf` or the manual upload path) are not available to mobile clients. Mobile write surfaces are rejected per `mobile_write_rejection_endpoints.md`. Read-only access to evidence PDF metadata is permitted on mobile; the signed download URL endpoint is read-only.

## Indexes

```sql
-- Tenant-scoped lookup by upload
CREATE INDEX idx_evidence_pdfs_business_upload
  ON evidence_pdfs (business_id, upload_id);

-- OCR work queue: pending/running items per business
CREATE INDEX idx_evidence_pdfs_business_ocr_status
  ON evidence_pdfs (business_id, ocr_status)
  WHERE ocr_status IN ('PENDING', 'RUNNING');

-- SHA-256 lookup for tamper-detection
CREATE INDEX idx_evidence_pdfs_sha256
  ON evidence_pdfs (sha256_hex);
```

## RLS

```sql
CREATE POLICY evidence_pdfs_isolation ON evidence_pdfs
  FOR ALL
  USING (business_id = ANY (auth.business_ids_for_session()));
```

Tenant isolation is enforced exclusively via `business_id`. No cross-tenant read is possible regardless of role.

## Audit events

| Event | When | Severity |
|---|---|---|
| `EVIDENCE_PDF_UPLOADED` | Row inserted via upload path or pipeline generation | LOW |
| `EVIDENCE_PDF_OCR_COMPLETED` | `ocr_status` transitions to `COMPLETED` | LOW |
| `EVIDENCE_PDF_OCR_FAILED` | `ocr_status` transitions to `FAILED` | MEDIUM |

All events emitted via `emitAudit()` per `audit_log_policies` and catalogued in `audit_event_taxonomy` under the EVIDENCE domain.
