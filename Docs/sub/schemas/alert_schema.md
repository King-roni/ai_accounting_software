# Alert Schema

**Category:** Schemas · **Owning block:** 05 — Security & Audit · **Stage:** 4 sub-doc (Layer 2)

Canonical definition of the `security_alerts` table. This table stores system-generated security and integrity alerts surfaced by the audit log monitoring pipeline, the hash-chain verification job, the access-control runtime, and the workflow watchdog. Each row represents a discrete alert instance. Alert lifecycle management (creation, acknowledgement, resolution) is governed by `security_alert_routing_policy`.

---

## ENUM definitions

```sql
CREATE TYPE alert_severity_enum AS ENUM ('MEDIUM', 'HIGH', 'BLOCKING');

CREATE TYPE alert_status_enum AS ENUM ('OPEN', 'ACKNOWLEDGED', 'RESOLVED');
```

Severity values are drawn from the project-wide closed set `{LOW, MEDIUM, HIGH, BLOCKING}` per `audit_log_policies`. LOW-severity conditions do not generate alert rows — they are recorded in the audit log only. MEDIUM is the minimum severity for alert creation.

## Table: `security_alerts`

```sql
CREATE TABLE security_alerts (
  alert_id                 uuid                NOT NULL DEFAULT gen_uuid_v7(),
  business_id              uuid,
  alert_type               text                NOT NULL,
  severity                 alert_severity_enum NOT NULL,
  source_event_id          uuid,
  workflow_run_id          uuid,
  description              text                NOT NULL,
  status                   alert_status_enum   NOT NULL DEFAULT 'OPEN',
  acknowledged_by_user_id  uuid,
  acknowledged_at          timestamptz,
  resolved_at              timestamptz,
  resolution_note          text,
  duplicate_count          integer             NOT NULL DEFAULT 0
    CHECK (duplicate_count >= 0),
  created_at               timestamptz         NOT NULL DEFAULT now(),
  updated_at               timestamptz         NOT NULL DEFAULT now(),

  CONSTRAINT security_alerts_pkey              PRIMARY KEY (alert_id),
  CONSTRAINT security_alerts_business_fk       FOREIGN KEY (business_id)
    REFERENCES business_entities(id) ON DELETE RESTRICT,
  CONSTRAINT security_alerts_run_fk            FOREIGN KEY (workflow_run_id)
    REFERENCES workflow_runs(id) ON DELETE RESTRICT,
  CONSTRAINT security_alerts_ack_by_fk         FOREIGN KEY (acknowledged_by_user_id)
    REFERENCES users(id) ON DELETE RESTRICT,
  CONSTRAINT security_alerts_ack_check
    CHECK (
      (status IN ('ACKNOWLEDGED', 'RESOLVED') AND acknowledged_by_user_id IS NOT NULL AND acknowledged_at IS NOT NULL)
      OR (status = 'OPEN' AND acknowledged_by_user_id IS NULL AND acknowledged_at IS NULL)
    ),
  CONSTRAINT security_alerts_resolved_check
    CHECK (
      (status = 'RESOLVED' AND resolved_at IS NOT NULL AND resolution_note IS NOT NULL)
      OR status IN ('OPEN', 'ACKNOWLEDGED')
    )
);
```

### Column notes

| Column | Notes |
| --- | --- |
| `alert_id` | UUID v7 PK via `gen_uuid_v7()` per `data_layer_conventions_policy`. Monotonically increasing; useful for ordered alert-queue queries. |
| `business_id` | FK to `business_entities.id`, nullable. NULL for platform-level alerts (cross-tenant, backup, infrastructure). Business-scoped alerts carry the `business_id` of the affected tenant. |
| `alert_type` | Text identifier for the alert rule that fired. Examples: `CROSS_TENANT_ACCESS_ATTEMPT`, `AUDIT_HASH_CHAIN_MISMATCH`, `WORKFLOW_RUN_STALLED`, `GATEWAY_BYPASS_DETECTED`. Not a database-level enum — new alert types are added without a migration. |
| `severity` | Enum: `MEDIUM`, `HIGH`, or `BLOCKING`. Determines notification routing per `security_alert_routing_policy`. |
| `source_event_id` | UUID, nullable. The `audit_log.id` of the event that triggered the alert, if one exists. NULL for alerts triggered by background jobs (e.g., stall watchdog) with no single triggering audit event. |
| `workflow_run_id` | FK to `workflow_runs.id`, nullable. Populated when the alert is associated with a specific workflow run (e.g., `WORKFLOW_RUN_STALLED`). |
| `description` | Human-readable description of the alert condition. Generated by the `security.raise_alert` tool. Not empty; bounded in practice to a few hundred characters. |
| `status` | Lifecycle state: `OPEN` (unacknowledged), `ACKNOWLEDGED` (reviewed by Owner/Admin, not yet closed), `RESOLVED` (closed with a resolution note). |
| `acknowledged_by_user_id` | FK to `users.id`, nullable. Populated when status transitions to `ACKNOWLEDGED`. The CHECK constraint requires this to be set when the alert is acknowledged. |
| `acknowledged_at` | Timestamptz, nullable. Set atomically with `acknowledged_by_user_id`. NULL until acknowledged. |
| `resolved_at` | Timestamptz, nullable. Set when status transitions to `RESOLVED`. NULL until resolution. |
| `resolution_note` | Text, nullable. Required when `status = RESOLVED`; the CHECK constraint enforces this. Provides a human-authored explanation of how the alert was investigated and closed. |
| `duplicate_count` | Integer, default 0. Incremented by the deduplication path in `security.raise_alert` when an identical alert tuple `(business_id, alert_type, workflow_run_id)` arrives within the 1-hour deduplication window. No new row is created for duplicates; the existing row's `duplicate_count` is incremented instead. |
| `created_at` | Set on INSERT; immutable. |
| `updated_at` | Maintained by `set_updated_at` trigger on every mutation. |

## Indexes

```sql
-- Business dashboard lookup: open and acknowledged alerts for a business.
CREATE INDEX idx_security_alerts_business_status
  ON security_alerts (business_id, status)
  WHERE status IN ('OPEN', 'ACKNOWLEDGED');

-- Alert type time-series: used by the alert rules engine for deduplication and trend queries.
CREATE INDEX idx_security_alerts_type_created
  ON security_alerts (alert_type, created_at DESC);
```

## RLS

Row-level security is enabled on this table.

```sql
ALTER TABLE security_alerts ENABLE ROW LEVEL SECURITY;

-- Owner and Admin may SELECT alerts for their business.
CREATE POLICY security_alerts_select_business
  ON security_alerts FOR SELECT
  USING (
    business_id IS NOT NULL
    AND business_id = ANY(current_user_businesses())
    AND current_user_role() IN ('OWNER', 'ADMIN')
  );
```

Platform-level alerts (`business_id IS NULL`) have no SELECT policy for authenticated roles — they are accessible to platform admin via the service role only. Cross-business read is impossible: the `current_user_businesses()` helper is scoped to the authenticated user's accessible businesses per `rls_helper_functions`.

Bookkeeper, Accountant, Reviewer, and Read-only roles do not have SELECT on this table. Security alert visibility is restricted to Owner and Admin.

### Mobile

Mobile clients may read security alerts (SELECT is permitted from mobile for dashboard display). Write operations (acknowledgement, resolution) on this table are performed through the API, which enforces `mobile_write_rejection_endpoints.md` — write requests from mobile clients return HTTP 405 with audit event `MOBILE_WRITE_REJECTED`.

## Audit events

| Event | Trigger | Severity |
| --- | --- | --- |
| `SECURITY_ALERT_CREATED` | A new `security_alerts` row is inserted by `security.raise_alert` | HIGH |
| `SECURITY_ALERT_RESOLVED` | `status` transitions to `RESOLVED` | MEDIUM |

`SECURITY_ALERT_CREATED` is HIGH because the act of raising an alert is itself a security-relevant signal that operators must be aware of. `SECURITY_ALERT_RESOLVED` is MEDIUM because resolution closes a known issue; MEDIUM ensures it is visible in the audit log without the urgency of a new alert.
