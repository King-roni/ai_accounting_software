# User Profile Schema

**Category:** Schemas · **Owning block:** 02 — Tenancy & Access · **Stage:** 4 sub-doc (Layer 2)

This document defines the `user_profiles` table, which extends Supabase `auth.users` with
application-level user data. `user_profiles` is a 1-to-1 extension; every `auth.users` row has
exactly one `user_profiles` row, created automatically by the trigger defined in Section 4.

---

## 1. Key convention exception

`user_profiles.id` references `auth.users(id)` and therefore uses the UUID generated by
Supabase Auth, which is `gen_random_uuid()`. This is a documented exception to the platform PK
convention. The platform's standard PK pattern (`gen_uuid_v7()`) applies to all business PKs.
`user_profiles.id` is not a business PK — it is an auth identity mirror. Do not change it to
`gen_uuid_v7()`; doing so would break the FK reference to `auth.users(id)`.

---

## 2. Table definition

```sql
CREATE TABLE user_profiles (
    -- PK mirrors auth.users(id); NOT gen_uuid_v7() — see Section 1.
    id                          UUID            PRIMARY KEY
                                                REFERENCES auth.users(id)
                                                ON DELETE CASCADE,

    display_name                TEXT            NOT NULL
                                                CHECK (char_length(display_name) BETWEEN 1 AND 120),

    avatar_url                  TEXT            NULL,
    -- Nullable. Points to a Supabase Storage object or an external URL.
    -- Must be a valid HTTPS URL when present. Validated at the application layer.

    preferred_locale            TEXT            NOT NULL
                                                DEFAULT 'el-CY',
    -- BCP 47 language tag. Default is Cyprus Greek, the primary market locale.
    -- Allowed values are validated at the application layer against a locale allowlist.

    timezone                    TEXT            NOT NULL
                                                DEFAULT 'Asia/Nicosia',
    -- IANA timezone identifier. Default is Cyprus Standard Time (UTC+2/UTC+3 DST).
    -- Validated at the application layer via the IANA timezone database.

    notification_preferences_id UUID            NULL
                                                REFERENCES notification_preferences(id)
                                                ON DELETE SET NULL,
    -- Nullable on creation. A default notification_preferences row is created
    -- alongside the user_profile by the trigger (see Section 4) in the same transaction.
    -- SET NULL ensures profile is not deleted if the preferences row is removed.

    last_active_at              TIMESTAMPTZ     NOT NULL
                                                DEFAULT now(),
    -- Updated by the application layer on each authenticated request. Not updated by trigger.

    created_at                  TIMESTAMPTZ     NOT NULL
                                                DEFAULT now(),

    updated_at                  TIMESTAMPTZ     NOT NULL
                                                DEFAULT now()
);

ALTER TABLE user_profiles ENABLE ROW LEVEL SECURITY;
```

## 3. Indexes

```sql
-- Primary key index is implicit.

-- Lookups by display_name for @-mention search and member directory.
CREATE INDEX idx_user_profiles_display_name
    ON user_profiles USING gin(to_tsvector('simple', display_name));

-- Staleness detection for last_active_at queries.
CREATE INDEX idx_user_profiles_last_active_at
    ON user_profiles (last_active_at DESC);
```

## 4. Trigger: create profile on auth.users INSERT

A `BEFORE INSERT` trigger on `auth.users` is not supported by Supabase Auth. The trigger is
defined as an `AFTER INSERT` trigger on `auth.users`, running in the `SECURITY DEFINER` context
to allow writes to `user_profiles` from within the Auth schema context.

```sql
CREATE OR REPLACE FUNCTION auth.on_user_created()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
    v_prefs_id UUID;
BEGIN
    -- Create default notification preferences first so the FK can be satisfied.
    INSERT INTO notification_preferences (id, created_at, updated_at)
    VALUES (gen_uuid_v7(), now(), now())
    RETURNING id INTO v_prefs_id;

    INSERT INTO user_profiles (
        id,
        display_name,
        preferred_locale,
        timezone,
        notification_preferences_id,
        last_active_at,
        created_at,
        updated_at
    )
    VALUES (
        NEW.id,
        COALESCE(NEW.raw_user_meta_data->>'display_name', split_part(NEW.email, '@', 1)),
        COALESCE(NEW.raw_user_meta_data->>'preferred_locale', 'el-CY'),
        COALESCE(NEW.raw_user_meta_data->>'timezone', 'Asia/Nicosia'),
        v_prefs_id,
        now(),
        now(),
        now()
    );

    RETURN NEW;
END;
$$;

CREATE TRIGGER on_auth_user_created
    AFTER INSERT ON auth.users
    FOR EACH ROW
    EXECUTE FUNCTION auth.on_user_created();
```

The `display_name` seed value comes from `raw_user_meta_data` if present, falling back to the
local part of the email address. The user can update their display name after first login.

## 5. Updated-at trigger

```sql
CREATE TRIGGER set_user_profiles_updated_at
    BEFORE UPDATE ON user_profiles
    FOR EACH ROW
    EXECUTE FUNCTION set_updated_at();
```

`set_updated_at()` is the platform-wide trigger function defined in the base migration.

## 6. RLS policies

```sql
-- Users can read and update their own profile.
CREATE POLICY user_profiles_self_access ON user_profiles
    FOR ALL
    USING (id = rls_get_user_id())
    WITH CHECK (id = rls_get_user_id());

-- Org admins and owners can read profiles of members in their business.
-- This policy requires a join through org_members to establish business membership.
CREATE POLICY user_profiles_org_admin_read ON user_profiles
    FOR SELECT
    USING (
        EXISTS (
            SELECT 1
            FROM   org_members om
            WHERE  om.user_id   = user_profiles.id
              AND  om.business_entity_id = rls_get_business_id()
        )
        AND rls_get_user_role() = ANY(ARRAY['owner', 'admin'])
    );
```

Org admins can read profiles of members within their business to support member directory and
assignment features. They cannot update another user's profile.

---

## 7. Columns excluded from the profile

The following data is **not** stored in `user_profiles`:

| Data point | Where it lives |
|---|---|
| Email address | `auth.users.email` (Supabase Auth) |
| Password hash | `auth.users` (Supabase Auth; never accessible) |
| MFA devices | `mfa_devices` table (Block 02) |
| Session tokens | `sessions` table (Block 02) |
| Business role | `org_members` table (Block 02) |
