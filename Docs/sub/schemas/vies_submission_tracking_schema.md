# VIES Submission Tracking Schema

**Category:** Schemas ¬∑ **Owning block:** 16 ‚Äî Dashboard & Reporting ¬∑ **Stage:** 4 sub-doc (Layer 2)

The `vies_submissions` table tracks EC Sales List (ESL) submissions to the Cyprus Tax Department under the VIES (VAT Information Exchange System) reporting obligation. Cyprus-registered VAT-payers that make intra-EU B2B supplies of goods or services must file a quarterly ESL. Block 16 generates the compliant XML file; this table tracks the lifecycle of each submission from XML generation through to tax-authority acceptance or rejection.

Submission to the Cyprus Tax Department portal is a manual operator step. The platform generates and stores the XML and tracks the status; the operator downloads the file and submits it via the Cyprus Tax Department's online portal. This table records what was submitted and what response was received.

## 3. Submission lifecycle

A `vies_submissions` row progresses through the following states. The status transitions are unidirectional forward; no backward transitions are permitted.

```
DRAFT ‚Üí SUBMITTED ‚Üí ACCEPTED
                  ‚Üí REJECTED
```

**DRAFT** ‚Äî The XML has been generated by `report.generate_vies_xml` and stored in `export-temp`. The operator can download the file but has not yet submitted it to the tax portal.

**SUBMITTED** ‚Äî The operator has downloaded the XML, submitted it via the Cyprus Tax Department's online portal, and recorded the submission in the platform by calling `report.record_vies_submission`. This marks `submitted_at` and `submitted_by_user_id`. The submission is now awaiting a response from the tax authority.

**ACCEPTED** ‚Äî The operator has received an acceptance confirmation from the tax portal and recorded it in the platform by calling `report.record_vies_accepted`. The `tax_authority_reference` is populated with the reference number from the tax authority. This is a terminal state.

**REJECTED** ‚Äî The operator has received a rejection from the tax portal and recorded it in the platform by calling `report.record_vies_rejected`. The `rejection_reason` is populated. This is a terminal state. A rejected filing typically requires a corrective XML to be generated (a new `DRAFT` row with a corrective flag per `vies_xml_schema`) and re-submitted.

## 4. RLS

Tenant isolation is enforced. All authenticated roles with a valid session on the business may read `vies_submissions` rows for their business. Write operations (INSERT and status-update transitions) are gated by the `REPORT_EXPORT_FULL` permission surface per `permission_matrix`, restricted to Owner, Admin, and Accountant.

```sql
-- Tenant isolation: all authenticated roles may read for their business
CREATE POLICY vies_submissions_read
  ON vies_submissions
  FOR SELECT
  USING (business_id = ANY (auth.business_ids_for_session()));

-- INSERT: only users with REPORT_EXPORT_FULL permission
CREATE POLICY vies_submissions_insert
  ON vies_submissions
  FOR INSERT
  WITH CHECK (
    business_id = ANY (auth.business_ids_for_session())
    AND auth.has_permission(business_id, 'REPORT_EXPORT_FULL')
  );

-- UPDATE: only users with REPORT_EXPORT_FULL permission (status transitions)
CREATE POLICY vies_submissions_update
  ON vies_submissions
  FOR UPDATE
  USING (
    business_id = ANY (auth.business_ids_for_session())
    AND auth.has_permission(business_id, 'REPORT_EXPORT_FULL')
  );

-- DELETE: blocked unconditionally ‚Äî submissions are permanent records
CREATE POLICY vies_submissions_no_delete
  ON vies_submissions
  FOR DELETE
  USING (false);
```

## 5. Eligibility and period scope

A business is eligible to file a VIES return only if it has VIES-relevant intra-EU supplies in the filing period. The eligibility rules are defined in `vies_quarterly_eligibility_policy`. A `vies_submissions` row is created only for eligible periods.

The quarterly period covers all `OUT_MONTHLY` runs whose `period_end` falls within the quarter. If a business has no VIES-relevant transactions in a quarter, no `DRAFT` row is created for that quarter.

---

## 6. XML storage and integrity

The generated XML is stored in the `export-temp` bucket at the path recorded in `xml_storage_key`. The `xml_sha256_hex` column provides the integrity check: when the operator downloads the file, the platform verifies the downloaded bytes' hash against `xml_sha256_hex` before delivery. If the hash does not match (e.g., due to storage corruption), the download is rejected and an alert is raised.

The `export-temp` bucket has a TTL; files are deleted after 24 hours per `export_pipeline_policy`. If a DRAFT submission's file has expired from `export-temp` before submission, the operator must re-generate the XML (via `report.generate_vies_xml`), which creates a new `DRAFT` row superseding the expired one. The expired row is retained in the database for the audit trail even after its XML file has been deleted.

---

## 7. Mobile rejection

`report.record_vies_submission`, `report.record_vies_accepted`, and `report.record_vies_rejected` are write operations. Mobile clients attempting to invoke these endpoints receive HTTP 405 `MOBILE_WRITE_REJECTED`. These endpoints are listed in `mobile_write_rejection_endpoints.md`. Read access to VIES submission status is available on mobile.

## 8. Audit events

| Event | Severity | When |
|---|---|---|
| `VIES_SUBMISSION_CREATED` | LOW | A new `DRAFT` row is inserted when XML is generated |
| `VIES_SUBMISSION_ACCEPTED` | LOW | Status transitions to `ACCEPTED`; `tax_authority_reference` is recorded |
| `VIES_SUBMISSION_REJECTED` | MEDIUM | Status transitions to `REJECTED`; `rejection_reason` is recorded |

All events are emitted on the business-scoped hash chain per `audit_log_policies`. They are in the `VIES` domain of `audit_event_taxonomy`.

`VIES_SUBMISSION_REJECTED` is MEDIUM because a rejection requires operator action (a corrective filing) and may indicate a data quality issue in the underlying VIES-relevant transactions.
