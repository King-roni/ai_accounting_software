# Classification Rule Schema

**Category:** Schemas · **Owning block:** 08 — Transaction Classification · **Stage:** 4 sub-doc (Layer 2)

Canonical DDL and evaluation conventions for the `classification_rules` table. This table holds the L1 rule set used by the Layer 1 deterministic classifier. Rules are evaluated before vendor memory (L2) and the AI fallback (L3). A matching L1 rule produces a final classification result without invoking higher layers.

---

## Purpose

L1 classification rules encode business-specific and global transaction classification logic as structured predicate expressions. The rule engine evaluates rules in priority order and returns the first match. Rules may be scoped to a specific business or applied globally as defaults.

The schema is the shared source of truth for:

- The `classification.apply_rules` tool, which reads and evaluates active rules
- The rule management API used by Owners and Admins to create and update rules
- The audit log, which records rule lifecycle events
- The live integration test runbook, which validates rule evaluation order and predicate matching

---

## Table DDL

```sql
CREATE TABLE classification_rules (
  id                  UUID            NOT NULL DEFAULT gen_uuid_v7(),
  business_id         UUID            REFERENCES business_entities(id),
                                      -- NULL = global rule
  rule_name           TEXT            NOT NULL,
  predicate           JSONB           NOT NULL,
  target_category     TEXT            NOT NULL,
  priority            INTEGER         NOT NULL,
  confidence_override NUMERIC(5, 4),
                                      -- NULL = use computed confidence
  is_active           BOOLEAN         NOT NULL DEFAULT TRUE,
  version             INTEGER         NOT NULL DEFAULT 1,
  created_at          TIMESTAMPTZ     NOT NULL DEFAULT now(),
  updated_at          TIMESTAMPTZ     NOT NULL DEFAULT now(),

  PRIMARY KEY (id)
);
```

## Field notes

### `id`

Primary key. Generated by `gen_uuid_v7()` per `data_layer_conventions_policy`. One row per rule version; the `version` column identifies the edition within the rule's history.

### `business_id`

When `NULL`, the rule is a global rule that applies to all businesses. When populated, the rule is scoped to that specific business. RLS prevents a non-owner user from reading or modifying rules belonging to a different business. Global rules (`business_id IS NULL`) are visible to the rule engine for all businesses but are only manageable by platform admins.

### `rule_name`

A short, human-readable label for the rule. Not required to be unique across businesses; display-only. Unique constraint is not enforced.

### `predicate`

A JSONB predicate expression tree. The top-level node is always an operator node. Leaf nodes compare a transaction field against a value. See the predicate shape section below.

### `target_category`

The category string assigned to a transaction when the predicate matches. Must be a valid category value from the active tag taxonomy. The rule engine validates this against the taxonomy at rule creation time.

### `priority`

Integer. Lower numeric value = higher priority. Rules are sorted ascending by `priority`; the first matching rule wins. Business-specific rules are evaluated before global rules at any given priority value (i.e. business rules shadow global rules at the same priority). Within a single business, rules are sorted strictly by `priority` ascending.

Priority values do not need to be contiguous. Gaps are acceptable and allow insertion of new rules between existing ones without renumbering.

### `confidence_override`

When set (non-null), bypasses the computed confidence for any transaction matched by this rule and substitutes the override value. Range: `[0.0000, 1.0000]`. This is used when an operator has high certainty about a particular counterparty or pattern and wants to prevent the match from landing in the human review queue. When `NULL`, the rule engine computes confidence from the predicate specificity heuristic.

### `is_active`

Only active rules (`is_active = TRUE`) are evaluated by the rule engine. Deactivated rules are retained for audit purposes. Deactivation does not delete the row; it sets `is_active = FALSE` and emits `CLASSIFICATION_RULE_DEACTIVATED`.

### `version`

Incremented by one on every edit to the rule's `predicate`, `target_category`, `priority`, `confidence_override`, or `rule_name`. The previous version row is NOT retained in-place; instead, a new row is inserted with the incremented `version` value and the same logical rule identity (tracked via the original `id` of the first version). Only the row with the highest `version` for a given logical rule is evaluated.

Implementation note: the rule versioning model uses a `root_rule_id` column (not shown above) that references the first version's `id`. All subsequent versions carry the same `root_rule_id`. The rule engine filters on `is_active = TRUE` and selects the row with the maximum `version` per `root_rule_id`. This sub-schema detail is owned by `layer1_rule_evaluation_schema.md`.

## Predicate JSONB shape

Predicate nodes are one of three kinds: operator nodes, comparison nodes, and range nodes.

### Operator node

```json
{
  "op": "AND" | "OR" | "NOT",
  "children": [/* array of predicate nodes */]
}
```

- `AND` — all children must match.
- `OR` — at least one child must match.
- `NOT` — the single child must not match. `children` must have exactly one element for `NOT`.

### Comparison node

```json
{
  "op": "EQ" | "CONTAINS",
  "field": "<transaction field name>",
  "value": "<string or number>"
}
```

- `EQ` — the transaction field value must equal `value` exactly (case-insensitive string comparison for text fields).
- `CONTAINS` — the transaction field value must contain `value` as a substring (case-insensitive).

### Range node

```json
{
  "op": "RANGE",
  "field": "<transaction field name>",
  "value": { "min": <number | null>, "max": <number | null> }
}
```

Both `min` and `max` are inclusive. A `null` bound means unbounded in that direction.

### Supported field names

| Field name | Type | Maps to |
|---|---|---|
| `counterparty_name` | text | `transactions.counterparty_name` |
| `counterparty_iban` | text | `transactions.counterparty_iban` |
| `description_raw` | text | `transactions.description_raw` |
| `amount_eur` | numeric | `transactions.amount_eur` (absolute value for RANGE) |
| `transaction_type` | text | `transactions.transaction_type` |

Field names outside this list are rejected at rule creation time with a validation error.

### Example predicate

Match all transactions from Revolut with an absolute amount under 10 EUR (typically bank fees):

```json
{
  "op": "AND",
  "children": [
    {
      "op": "EQ",
      "field": "counterparty_name",
      "value": "Revolut"
    },
    {
      "op": "RANGE",
      "field": "amount_eur",
      "value": { "min": 0, "max": 10 }
    }
  ]
}
```

## Evaluation order

1. Retrieve all active rules for the business (`business_id = :business_id AND is_active = TRUE`), sorted by `priority` ascending. The rule engine selects the highest-version active row per `root_rule_id`.
2. Retrieve all active global rules (`business_id IS NULL AND is_active = TRUE`), sorted by `priority` ascending.
3. Evaluate business-specific rules first. Return the first match.
4. If no business-specific rule matches, evaluate global rules. Return the first match.
5. If no rule matches, pass to Layer 2 (vendor memory).

This order means a business-specific rule at priority 100 takes precedence over a global rule at priority 1. Business rules always shadow global rules regardless of priority value.

---

## Rule versioning

Each edit to a rule creates a new version row. The workflow for an edit is:

1. INSERT a new row with incremented `version`, `is_active = TRUE`, updated fields.
2. UPDATE the previous version's row to `is_active = FALSE`.
3. Emit `CLASSIFICATION_RULE_UPDATED` with `rule_id` (the new row's `id`), `previous_version`, `new_version`, `changed_fields`.

Reads by the rule engine always filter on `is_active = TRUE`, so version transitions are instantaneous at the row level. Old version rows (with `is_active = FALSE`) are retained indefinitely for audit trail purposes and must not be deleted.

## Indexes

```sql
-- Primary lookup for rule engine
CREATE INDEX idx_classification_rules_business_active
  ON classification_rules (business_id, is_active, priority)
  WHERE is_active = TRUE;

-- Global rules lookup
CREATE INDEX idx_classification_rules_global_active
  ON classification_rules (priority)
  WHERE business_id IS NULL AND is_active = TRUE;
```

## Audit events

| Event | Trigger | Severity |
|---|---|---|
| `CLASSIFICATION_RULE_CREATED` | New rule row inserted with `version = 1` | LOW |
| `CLASSIFICATION_RULE_UPDATED` | New version row inserted (`version > 1`) | LOW |
| `CLASSIFICATION_RULE_DEACTIVATED` | `is_active` set to `FALSE` via the deactivation API | LOW |

`CLASSIFICATION_RULE_CREATED` payload: `rule_id`, `business_id` (nullable), `rule_name`, `target_category`, `priority`, `has_confidence_override`.

`CLASSIFICATION_RULE_UPDATED` payload: `rule_id`, `business_id` (nullable), `previous_rule_id`, `previous_version`, `new_version`, `changed_fields`.

`CLASSIFICATION_RULE_DEACTIVATED` payload: `rule_id`, `business_id` (nullable), `rule_name`, `deactivated_by_user_id`.

---

## RLS

RLS is enabled on `classification_rules`. Business-scoped rules are readable and writable only by sessions with an active role on the matching `business_id`. Global rules (`business_id IS NULL`) are readable by all authenticated sessions but only writable by the service role (platform admin operations).
