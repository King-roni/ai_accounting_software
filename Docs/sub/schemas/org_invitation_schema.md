# Org Invitation Schema

**Block:** 02 — Tenancy, Auth & Org Management  
**Layer:** 2 — Sub-Doc  
**Status:** Draft

## Overview

This document defines the `org_invitations` table, which manages the lifecycle of business-membership invitations. An invitation allows an existing platform user or a new registrant to join a specific business entity in a specified role. Invitation tokens use `gen_random_uuid()` because they function as opaque bearer tokens (like password-reset tokens and OAuth state IDs) — UUID v7 would leak timing information and is reserved for internal PK identifiers.

---

## 1. Enum Types

### 1.1 `invitation_status_enum`

```sql
CREATE TYPE invitation_status_enum AS ENUM (
    'PENDING',    -- Token issued, not yet acted upon
    'ACCEPTED',   -- Invitee has successfully joined the business
    'DECLINED',   -- Invitee explicitly declined (optional flow)
    'EXPIRED',    -- 24-hour validity window elapsed without action
    'REVOKED'     -- Explicitly cancelled by an OWNER or ADMIN before expiry
);
```

Status transitions are unidirectional. Once a record reaches `ACCEPTED`, `DECLINED`, `EXPIRED`, or `REVOKED`, no further transitions are permitted. A `PENDING` record may transition to any of the four terminal states.

### 1.2 `invited_role_enum`

Invitation role values are a subset of the platform's full role enum. Only roles that can be granted to invited members are permitted.

```sql
-- References the platform-wide role enum defined in tenancy_schema_definition.md
-- Valid values for invitations: 'MEMBER', 'ADMIN'
-- OWNER role cannot be granted via invitation; it is assigned at business creation.
```

## 2. Table Definition

```sql
CREATE TABLE org_invitations (
    id              UUID                    PRIMARY KEY DEFAULT gen_random_uuid(),
    -- gen_random_uuid() is intentional: this column serves as the bearer token
    -- for the invitation link. Timing-predictable UUIDs (v7) would weaken security.

    business_id     UUID                    NOT NULL
                                            REFERENCES business_entities(id)
                                            ON DELETE CASCADE,

    invited_email   TEXT                    NOT NULL,
    -- Stored in lowercase. Email is validated at insert time against RFC 5321.

    invited_by      UUID                    NOT NULL
                                            REFERENCES auth.users(id)
                                            ON DELETE SET NULL,
    -- SET NULL: if the inviting user is deactivated, the invitation record is
    -- retained with invited_by = NULL. The invitation remains valid.

    role            TEXT                    NOT NULL
                                            CHECK (role IN ('MEMBER', 'ADMIN')),
    -- OWNER cannot be granted via invitation. See tenancy_schema_definition.md.

    token           TEXT                    NOT NULL UNIQUE,
    -- A separate random token value used in the invitation link URL.
    -- Distinct from the PK. Generated by the application layer using
    -- cryptographically secure random bytes (32 bytes, base64url-encoded).
    -- The PK (id) identifies the record; the token is the secret bearer value.

    status          invitation_status_enum  NOT NULL DEFAULT 'PENDING',

    expires_at      TIMESTAMPTZ             NOT NULL
                                            DEFAULT now() + interval '24 hours',

    accepted_at     TIMESTAMPTZ,
    -- Populated when status transitions to ACCEPTED.

    declined_at     TIMESTAMPTZ,
    -- Populated when status transitions to DECLINED.

    revoked_at      TIMESTAMPTZ,
    -- Populated when status transitions to REVOKED.

    revoked_by      UUID                    REFERENCES auth.users(id),
    -- Populated on REVOKED transition. Records the OWNER or ADMIN who revoked.

    created_at      TIMESTAMPTZ             NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ             NOT NULL DEFAULT now()
);
```

## 3. Indexes

```sql
-- Primary lookup path: token redemption (invitation link click)
CREATE UNIQUE INDEX idx_org_invitations_token
    ON org_invitations (token);

-- Business-level admin view: list pending invitations for a business
CREATE INDEX idx_org_invitations_business_status
    ON org_invitations (business_id, status)
    WHERE status = 'PENDING';

-- Find all invitations sent to a specific email address (dedup check before re-send)
CREATE INDEX idx_org_invitations_invited_email
    ON org_invitations (invited_email, business_id, status)
    WHERE status = 'PENDING';

-- Expiry sweep job index
CREATE INDEX idx_org_invitations_expires_at
    ON org_invitations (expires_at)
    WHERE status = 'PENDING';
```

## 4. Row-Level Security

```sql
ALTER TABLE org_invitations ENABLE ROW LEVEL SECURITY;

-- Business OWNER or ADMIN can view all invitations for their business
CREATE POLICY org_invitations_select_admin
    ON org_invitations FOR SELECT
    USING (
        EXISTS (
            SELECT 1 FROM org_members
            WHERE user_id     = auth.uid()
              AND business_id = org_invitations.business_id
              AND role        IN ('OWNER', 'ADMIN')
        )
    );

-- The invitee can view their own invitation by token lookup
-- (used during the acceptance flow before the user is a member)
CREATE POLICY org_invitations_select_invitee
    ON org_invitations FOR SELECT
    USING (
        invited_email = (
            SELECT email FROM auth.users WHERE id = auth.uid()
        )
    );

-- Only OWNER or ADMIN can insert (create invitations)
CREATE POLICY org_invitations_insert
    ON org_invitations FOR INSERT
    WITH CHECK (
        EXISTS (
            SELECT 1 FROM org_members
            WHERE user_id     = auth.uid()
              AND business_id = NEW.business_id
              AND role        IN ('OWNER', 'ADMIN')
        )
    );

-- Only OWNER or ADMIN can update (revoke); system can update (accept/expire)
CREATE POLICY org_invitations_update_admin
    ON org_invitations FOR UPDATE
    USING (
        EXISTS (
            SELECT 1 FROM org_members
            WHERE user_id     = auth.uid()
              AND business_id = org_invitations.business_id
              AND role        IN ('OWNER', 'ADMIN')
        )
    );

-- Service role (system processes) bypasses RLS for expiry sweeps and acceptance flows
-- handled via auth.service_role, not auth.uid()
```

## 5. Expiry Enforcement

Invitation links are valid for **24 hours** from creation. The `expires_at` column is set to `now() + interval '24 hours'` at insert time.

**Lazy expiry check:** The redemption endpoint checks `expires_at <= now()` before processing a token. If expired, the record is transitioned to `EXPIRED` and the `AUTH_INVITATION_EXPIRED` audit event is emitted.

**Sweep job:** A background job runs every 15 minutes. It transitions all `PENDING` records where `expires_at <= now()` to `EXPIRED` status. The `AUTH_INVITATION_EXPIRED` event is emitted for each transitioned record.

```sql
-- Expiry sweep query (runs inside a transaction)
UPDATE org_invitations
SET    status     = 'EXPIRED',
       updated_at = now()
WHERE  status     = 'PENDING'
  AND  expires_at <= now()
RETURNING id, business_id, invited_email, expires_at;
-- Emit AUTH_INVITATION_EXPIRED per returned row
```

## 6. Re-Send Logic

If an invitation is re-sent to the same email address for the same business before the original invitation has been accepted, the following procedure applies:

1. Check for any `PENDING` invitation matching `(invited_email, business_id)`.
2. If one exists, transition it to `EXPIRED` (`revoked_at` is not set; `updated_at` is set).
3. Insert a new `org_invitations` row with `status = PENDING` and a fresh `expires_at`.
4. Emit `AUTH_INVITATION_EXPIRED` for the superseded record.
5. Emit `AUTH_INVITATION_SENT` for the new record.

This ensures that only one active invitation per `(invited_email, business_id)` pair exists at any time. The superseded invitation link, if clicked, will fail the expiry check.

---

## 7. Acceptance Flow

1. User clicks the invitation link. The link contains the `token` value (not the `id`).
2. Application looks up the invitation by `token`.
3. If `status != PENDING` or `expires_at <= now()`, return an error and transition to `EXPIRED` if not already terminal.
4. If the invitee is not a registered user, the registration flow runs first. On completion, the acceptance flow resumes.
5. An `org_members` row is inserted for `(user_id, business_id, role)`.
6. The `org_invitations` row is updated: `status = ACCEPTED`, `accepted_at = now()`.
7. `AUTH_INVITATION_CONSUMED` audit event is emitted.

The acceptance flow runs inside a database transaction. If the `org_members` insert fails (e.g., the user is already a member), the transaction rolls back. The invitation status remains `PENDING`.

---

## 8. Revocation Flow

An `OWNER` or `ADMIN` may revoke a `PENDING` invitation at any time. Revocation:

1. Sets `status = REVOKED`, `revoked_at = now()`, `revoked_by = auth.uid()`.
2. Does not delete the record; the history is retained.
3. Emits `AUTH_INVITATION_REVOKED` audit event with `revocation_reason` (optional).

Revoking an already-expired invitation is a no-op (the record is already in a terminal state).

## 9. Audit Events

All audit events reference the canonical taxonomy in `audit_event_taxonomy.md`.

| Event | Severity | Trigger |
|-------|----------|---------|
| `AUTH_INVITATION_SENT` | LOW | New `org_invitations` row inserted and email dispatched. Payload: `token_id` (the record `id`, not the `token` value), `business_id`, `invited_by_user_id`, `invitee_email`, `assigned_role`, `expires_at`. |
| `AUTH_INVITATION_CONSUMED` | LOW | `status` transitions to `ACCEPTED`. Payload: `token_id`, `business_id`, `consumed_by_user_id`, `assigned_role`, `consumed_at`. |
| `AUTH_INVITATION_REVOKED` | MEDIUM | `status` transitions to `REVOKED` by an admin. Payload: `token_id`, `business_id`, `revoked_by_user_id`, `revocation_reason`. |
| `AUTH_INVITATION_EXPIRED` | LOW | `status` transitions to `EXPIRED` via sweep job or lazy check on redemption attempt. Payload: `token_id`, `business_id`, `expires_at`, `attempted_at`. |

The raw `token` value is never included in any audit payload. The `id` column (PK) is used as the `token_id` reference in payloads.

---

## 10. Capacity Constraint

The number of `PENDING` invitations per business is limited by the org member capacity policy defined in `org_member_capacity_policy.md`. Attempting to create an invitation that would exceed the business's member capacity limit (including pending invitations) returns a validation error before the row is inserted.
