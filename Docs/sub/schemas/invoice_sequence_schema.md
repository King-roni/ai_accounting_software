# invoice_sequence_schema

**Category:** Schemas · **Owning block:** 13 — IN Workflow + Invoice Generator · **Stage:** 4 sub-doc (Layer 2)

The `invoice_sequences` table manages gap-free, per-business, per-year sequence numbers for the three invoice series. This document is the normative source for the sequence table definition, the permitted write path, and the gap-free requirement rationale. All conventions bind to `data_layer_conventions_policy` (UUID v7, canonical JSON, decimal-precise currency).

---

## Table definition

```sql
CREATE TYPE invoice_series_enum AS ENUM (
  'TAX',
  'PRO_FORMA',
  'CREDIT_NOTE'
);

CREATE TABLE invoice_sequences (
  sequence_id            uuid PRIMARY KEY DEFAULT gen_uuid_v7(),
  business_id            uuid NOT NULL REFERENCES business_entities(id),

  -- Series and year identify the sequence
  series                 invoice_series_enum NOT NULL,
  year                   integer NOT NULL
    CHECK (year >= 2020 AND year <= 2099),

  -- The next counter value to be allocated for this (business, series, year)
  next_counter           integer NOT NULL DEFAULT 1
    CHECK (next_counter >= 1),

  -- Authorship
  created_at             timestamptz NOT NULL DEFAULT now(),
  updated_at             timestamptz NOT NULL DEFAULT now(),

  -- Uniqueness: exactly one row per (business, series, year)
  CONSTRAINT uq_invoice_sequences_business_series_year
    UNIQUE (business_id, series, year)
);
```

## Column rationale

### `sequence_id` (UUID v7)

Primary key generated by `gen_uuid_v7()` (Block 04 Phase 01 helper). Business-data table; UUID v7 is correct per `data_layer_conventions_policy`.

### `series` (invoice_series_enum)

Three series in the closed MVP enum:

| Series | Invoice type | Number format |
| --- | --- | --- |
| `TAX` | `TAX_INVOICE` | `INV-YYYY-NNNN` |
| `PRO_FORMA` | `PRO_FORMA` | `PRO-YYYY-NNNN` |
| `CREDIT_NOTE` | `CREDIT_NOTE` | `CN-YYYY-NNNN` |

The `decisions_log.md` 2026-05-08 amendment is binding: `PRO_FORMA` uses a distinct sequence so the `TAX` (`INV`) series remains gap-free. Converting a pro-forma to a tax invoice consumes a fresh `INV` number from the `TAX` series; the `PRO` number is retired, not reused.

### `year` (integer)

Calendar year of the sequence. Each business gets one row per series per year. A new year's row is created lazily on first allocation. The `year` column matches the year in the formatted invoice number (e.g., `INV-2026-0001` → `year = 2026`).

### `next_counter` (integer, default 1)

The next sequence number to be allocated for this `(business_id, series, year)`. Default 1 means no invoices have been issued yet for this series in this year. After the first allocation, this value is 2; after the second, 3; and so on. This counter never decrements.

## Write path — `in_workflow.next_invoice_number`

`in_workflow.next_invoice_number` is the **only** permitted write path to this table. Direct `UPDATE invoice_sequences SET next_counter = ...` from application code is forbidden; the CI lint rule enforces this by rejecting any SQL that writes to `invoice_sequences` outside the designated tool.

The tool executes the following inside a single serializable transaction:

```sql
-- 1. Lock the sequence row (or create it if it doesn't exist yet for this year)
INSERT INTO invoice_sequences (business_id, series, year, next_counter)
VALUES ($business_id, $series, $year, 1)
ON CONFLICT (business_id, series, year) DO NOTHING;

SELECT sequence_id, next_counter
FROM invoice_sequences
WHERE business_id = $business_id
  AND series = $series
  AND year = $year
FOR UPDATE;

-- 2. Increment and write back
UPDATE invoice_sequences
SET next_counter = next_counter + 1,
    updated_at = now()
WHERE business_id = $business_id
  AND series = $series
  AND year = $year;

-- 3. Return the formatted number
-- e.g., series=TAX, year=2026, next_counter (pre-increment) = 42
-- → 'INV-2026-0042'
```

The `SELECT ... FOR UPDATE` serializes concurrent allocation attempts on the same `(business_id, series, year)` row. Two concurrent calls will not receive the same number.

### Number formatting

| Series | Format expression | Example |
| --- | --- | --- |
| `TAX` | `'INV-' || year || '-' || LPAD(next_counter::text, 4, '0')` | `INV-2026-0042` |
| `PRO_FORMA` | `'PRO-' || year || '-' || LPAD(next_counter::text, 4, '0')` | `PRO-2026-0007` |
| `CREDIT_NOTE` | `'CN-' || year || '-' || LPAD(next_counter::text, 4, '0')` | `CN-2026-0003` |

The zero-padded 4-digit format supports up to 9999 invoices per series per business per year. If `last_sequence_number` exceeds 9999, the format expands naturally (e.g., `INV-2026-10000`). The 4-digit pad is a display convention, not a hard cap.

## Gap-free requirement

Cyprus VAT law requires sequential, unbroken invoice numbering for tax invoices. Gaps in the `INV` sequence trigger audit flags during a tax authority review. This requirement is the reason:

1. Sequence numbers are allocated only at first exit from `DRAFT` — never at row creation.
2. A `DRAFT` invoice that is deleted without being issued consumes no sequence number.
3. Voided invoices retain their allocated number; the number is not recycled.
4. Sequence numbers are never decremented, even on error.

The `PRO_FORMA` (`PRO`) series is maintained separately so that the `TAX` series remains unbroken regardless of pro-forma creation and expiry volume. Credit notes (`CN`) follow the same gap-free rule as tax invoices.

---

## Allocation timing by invoice type

| Invoice type | Number allocated at |
| --- | --- |
| `TAX_INVOICE` | First `DRAFT → SENT` transition (via `in_workflow.send_invoice`) |
| `PRO_FORMA` | First `DRAFT → SENT` transition (via `in_workflow.send_invoice`) |
| `CREDIT_NOTE` | At issuance (creation; credit notes are never `DRAFT`) |

Credit notes are issued in a single write; there is no `DRAFT` state. The number is allocated at creation time by `in_workflow.issue_credit_note` which calls `in_workflow.next_invoice_number` for the `CREDIT_NOTE` series.

## RLS

```sql
CREATE POLICY invoice_sequences_isolation ON invoice_sequences
  FOR ALL
  USING (business_id = ANY(auth.business_ids_for_session()));
```

Write access is additionally controlled by the tool-only write path rule. No user-facing API exposes `invoice_sequences` mutations.

## Audit events

| Event | Severity | Trigger |
| --- | --- | --- |
| `INVOICE_NUMBER_ALLOCATED` | LOW | `in_workflow.next_invoice_number` completes; number written to `invoices` or `credit_notes` |
| `INVOICE_NUMBER_GAP_DETECTED` | MEDIUM | Integrity job detects a gap in the sequence (post-allocation forensic check only) |

`INVOICE_NUMBER_ALLOCATED` payload: `business_id`, `series`, `year`, `allocated_number`, `invoice_id` or `credit_note_id`.
