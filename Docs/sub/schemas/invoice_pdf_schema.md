# invoice_pdf_schema

**Category:** Schemas Â· **Owning block:** 13 â€” IN Workflow + Invoice Generator Â· **Stage:** 4 sub-doc (Layer 2)

The `invoice_pdfs` table records every generated PDF artifact for every invoice version. PDFs are regenerated on amendment (e.g., a corrected line item before sending, or a re-render after a template update); each version is stored separately and only one version per invoice is the current canonical PDF. The table is the authoritative record of where each PDF lives in object storage and its integrity fingerprint.

---

## Table definition

```sql
CREATE TYPE invoice_pdf_bucket_enum AS ENUM (
  'ARCHIVE',
  'EXPORT_TEMP'
);

CREATE TABLE invoice_pdfs (
  invoice_pdf_id      uuid PRIMARY KEY DEFAULT gen_uuid_v7(),
  invoice_id          uuid NOT NULL REFERENCES invoices(invoice_id),
  invoice_version     integer NOT NULL
    CHECK (invoice_version >= 1),
  storage_key         text NOT NULL,
  bucket              invoice_pdf_bucket_enum NOT NULL,
  sha256_hex          char(64) NOT NULL,
  generated_at        timestamptz NOT NULL DEFAULT now(),
  generated_by_tool   text NOT NULL,
  page_count          integer NOT NULL
    CHECK (page_count >= 1),
  file_size_bytes     bigint NOT NULL
    CHECK (file_size_bytes > 0),
  is_current          boolean NOT NULL DEFAULT false,

  CONSTRAINT sha256_hex_format
    CHECK (sha256_hex ~ '^[0-9a-f]{64}$'),
  CONSTRAINT generated_by_tool_format
    CHECK (generated_by_tool ~ '^[a-z][a-z0-9_]*\.[a-z][a-z0-9_]*$')
);
```

## Column rationale

### `invoice_pdf_id` (UUID v7)

Primary key generated by `gen_uuid_v7()` per `data_layer_conventions_policy`. UUID v7 is appropriate â€” this is a business-data table; the time-ordered prefix aids time-range queries over generated PDFs.

### `invoice_id` (FK to `invoices`)

References `invoices.invoice_id`. One invoice may have multiple PDF rows across versions. The FK is non-nullable â€” every PDF row must belong to an invoice. There is no orphaned-PDF concept; if an invoice is deleted (only possible for `DRAFT` invoices), the PDF rows are cascade-deleted along with it.

### `invoice_version` (integer, min 1)

PDF generation version counter. Version 1 is the first PDF generated for the invoice. Version 2 is generated if the invoice is amended before sending (e.g., a corrected line item) and a re-render is triggered. Versions are never deleted; only `is_current` is updated when a newer version supersedes the prior one. The audit event `INVOICE_PDF_SUPERSEDED` is emitted when a previously-current row has `is_current` set to `false`.

### `storage_key` (text NOT NULL)

The object storage path for this PDF. Format is bucket-dependent:
- `ARCHIVE` bucket: `pdfs/<business_id>/<year>/<invoice_number>_v<version>.pdf`
- `EXPORT_TEMP` bucket: `tmp/<job_id>/<invoice_number>_v<version>.pdf`

The `storage_key` is unique within a bucket but not globally â€” the `(bucket, storage_key)` pair is the unique locator. Governance of the bucket paths is owned by Block 15 Phase 04 (ARCHIVE) and Block 16 Phase 03 (EXPORT_TEMP).

### `bucket` (invoice_pdf_bucket_enum)

Two valid values:

| Value | Usage |
| --- | --- |
| `ARCHIVE` | Long-term storage; Object-Locked after finalization (Block 15). The canonical permanent copy. |
| `EXPORT_TEMP` | Short-lived export job staging area (Block 16). Rows in `EXPORT_TEMP` are deleted by the retention sweeper after the export job completes or expires. These rows are not included in the finalization integrity check. |

A FINALIZED invoice must have exactly one row with `bucket = 'ARCHIVE'` and `is_current = true` at all times. This invariant is enforced by Block 15's finalization precondition check.

### `sha256_hex` (char(64))

SHA-256 hash of the PDF file bytes, encoded as 64 lowercase hex characters, per `data_layer_conventions_policy`. Used for integrity verification after storage. On retrieval, the file bytes are re-hashed and compared to this value; a mismatch emits `ARCHIVE_TAMPER_DETECTED`.

The format CHECK constraint `^[0-9a-f]{64}$` ensures only lowercase hex is accepted (uppercase hex encoding is a convention violation per `data_layer_conventions_policy`).

### `generated_by_tool` (text NOT NULL)

The tool name that generated this PDF, following the `<block_short_name>.<action>` naming convention from `tool_naming_convention_policy`. Examples: `in_workflow.generate_invoice_pdf`, `in_workflow.regenerate_invoice_pdf`. The CHECK constraint enforces the two-segment snake_case format. The tool name is stored (not a FK) because tool versions may be deprecated over time; the stored name is the generation-time snapshot.

### `page_count` (integer, min 1)

Number of pages in the generated PDF. Minimum 1 (a zero-page PDF is a generation error). Used for display in the UI and for quick integrity sanity checks.

### `file_size_bytes` (bigint)

Size of the PDF file in bytes. Must be greater than 0. Used for storage accounting and as a secondary integrity indicator alongside `sha256_hex`.

### `is_current` (boolean)

When `true`, this row is the current canonical PDF for this invoice at this `invoice_version`. Only one row per invoice may have `is_current = true` â€” this invariant is enforced by the PDF generation tool before flipping the flag (setting the prior current row to `false` and the new row to `true` in a single transaction).

When a new PDF version is generated, the prior current row's `is_current` is set to `false` and `INVOICE_PDF_SUPERSEDED` is emitted; the new row's `is_current` is set to `true` and `INVOICE_PDF_GENERATED` is emitted.

## Invariant: FINALIZED invoice has exactly one current ARCHIVE PDF

For every invoice with `status = 'FINALIZED'` (per `invoice_schema`), the following must hold:

```sql
SELECT COUNT(*) = 1
FROM invoice_pdfs
WHERE invoice_id = $invoice_id
  AND bucket = 'ARCHIVE'
  AND is_current = true;
```

This invariant is checked by Block 15's finalization precondition evaluator before the lock sequence proceeds. A finalized invoice without a current ARCHIVE PDF is a `BLOCKING` precondition failure (`FINALIZATION_PRECONDITION_FAILED`). Block 15 Phase 04's `archive.lock_period` tool calls `in_workflow.generate_invoice_pdf` for any FINALIZED invoice that lacks a current ARCHIVE PDF before sealing the archive.

## Indexes

```sql
CREATE INDEX idx_invoice_pdfs_invoice_current
  ON invoice_pdfs(invoice_id, is_current);

CREATE INDEX idx_invoice_pdfs_invoice_version
  ON invoice_pdfs(invoice_id, invoice_version DESC);

CREATE INDEX idx_invoice_pdfs_bucket
  ON invoice_pdfs(bucket)
  WHERE bucket = 'EXPORT_TEMP';
```

The `(invoice_id, is_current)` index is the hot path for "find the current PDF for this invoice." The `(invoice_id, invoice_version DESC)` index supports version history queries. The partial index on `EXPORT_TEMP` bucket supports the retention sweeper's scan for expired temporary PDFs.

## Mobile write rejection

PDF generation is a write operation. Any attempt to trigger PDF generation from a session where `client_form_factor = MOBILE` is rejected before the permission check with `MOBILE_WRITE_REJECTED`. Mobile clients may read the `invoice_pdfs` metadata rows and retrieve PDFs via signed URLs (read-only). Reference: `mobile_write_rejection_endpoints.md`.

## Audit events

| Event | Severity | Trigger |
| --- | --- | --- |
| `INVOICE_PDF_GENERATED` | LOW | New PDF row inserted with `is_current = true` for the first version of an invoice |
| `INVOICE_PDF_SUPERSEDED` | LOW | Prior current row has `is_current` set to `false` when a new version is generated |

Both events exist in `audit_event_taxonomy` under the `INVOICE` domain. Payloads include `invoice_pdf_id`, `invoice_id`, `invoice_version`, `bucket`, `sha256_hex`, `generated_by_tool`, and `generated_at`.
