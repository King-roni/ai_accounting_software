# Block 10 — Matching Engine

## Role in the System

The matching engine connects transactions to the documents that justify them — invoices to outgoing payments, invoices to incoming payments, receipts to expenses, contracts to recurring obligations. It also detects duplicate matches (one invoice claimed against multiple transactions) and produces a stored, plain-language reason for every match it makes.

The engine is deterministic-first. AI is consulted only when deterministic scoring is genuinely ambiguous, and AI never changes a match outcome — at most it rephrases or explains it. The engine's authority over what is and isn't a match comes from rules, not models.

---

## Scope

### In scope
- Match scoring across the four match levels (Exact, Strong Probable, Weak Possible, No Match)
- The match status taxonomy
- Duplicate detection (one document → multiple transactions; same document from multiple sources; same statement row matched twice)
- Match reason generation (structured + plain-language)
- One-to-many and many-to-one match relationships (split payments, consolidated invoices)
- Routing of unconfirmed and weak matches to Block 14 (Review Queue)

### Out of scope (covered elsewhere)
- Producing the documents in the first place → Block 09 (Document Intake & Extraction)
- Producing the transactions → Block 07 (Bank Statement Pipeline)
- Classifying transactions → Block 08 (Transaction Classification & Tagging)
- Translating matched evidence into ledger entries → Block 11 (Ledger & Cyprus VAT Engine)
- Audit logging → Block 05 (Security & Audit Layer)

## The Four Match Levels

### Level 1 — Exact Match
- Amount and currency exact
- Counterparty / supplier name exact (after normalization)
- Date within the accepted exact-window
- Invoice number or payment reference matches if either is present

Result: `MATCHED_AUTO_HIGH_CONFIDENCE`. Auto-confirmed; no user prompt unless the run is in strict-confirmation mode.

### Level 2 — Strong Probable Match
- Amount and currency exact
- Counterparty fuzzy match (normalized supplier name distance below threshold)
- Date within a wider window
- Recurring pattern from earlier matches reinforces the link

Result: `MATCHED_AUTO_HIGH_CONFIDENCE` **only when the recurring-pattern signal is strong** (the same supplier has been confirmed repeatedly before for this business); otherwise `MATCHED_NEEDS_CONFIRMATION` and routed to Block 14.

### Level 3 — Weak Possible Match
- Some signals align (amount + counterparty fuzzy, but date far; or amount close + supplier exact)
- No strong recurring pattern

Result: `POSSIBLE_MATCH`, always sent to Block 14 for user confirmation.

### Level 4 — No Match
- No document candidate scores above the minimum threshold

Result: `NO_MATCH`, raises a "Missing Documents" review issue.

## Match Score Components

The scorer combines these signals (weights and thresholds calibrated in phase docs and tuned in operation):

- Amount exact match
- Currency match
- Supplier / counterparty exact match (after normalization)
- Supplier / counterparty fuzzy match (string distance)
- Date proximity (days between transaction and document) — **default windows: ±3 days for Exact, ±10 days for Strong Probable, ±30 days for Weak Possible**
- Invoice number / reference match
- Known recurring vendor history
- Email sender domain alignment with supplier
- Drive folder relevance signal
- Business name appearing on the invoice
- VAT number relevance (when both sides have one)

**Cross-period matching:** the engine looks back **1–2 months** in unmatched documents — covers the common case of an invoice issued late in one period and paid early in the next.

**Cross-currency matching:** uses the **bank-recorded FX rate** from the transaction's FX leg (Revolut's own rate). When the bank rate is missing, the engine falls back to the ECB daily reference rate for the transaction date.

## Match Statuses

```text
MATCHED_CONFIRMED          (user has explicitly confirmed)
MATCHED_AUTO_HIGH_CONFIDENCE (auto-confirmed by the engine)
MATCHED_NEEDS_CONFIRMATION (auto-suggested; awaiting user)
POSSIBLE_MATCH             (weak signals; user must decide)
NO_MATCH                   (no candidate above threshold)
REJECTED_MATCH             (user explicitly rejected a previous suggestion)
```

A match record carries the status, the score, the structured signal breakdown, the plain-language reason, and references to both the transaction and the document.

---

## Match Reason

Every match record stores a structured signal breakdown plus a plain-language summary. Example:

> Matched because the invoice amount is EUR 49.00, the transaction amount is EUR 49.00, the supplier is Google Ireland Ltd, and the invoice date is one day before the bank transaction.

The plain-language reason is generated by Block 06's End-Scan / explanation pipeline (Tier 2 by default; Tier 3 only when the case is complex enough to warrant it). The structured breakdown is what the engine actually decides on; the plain-language version is for users and auditors.

## Duplicate Detection

The engine tracks several distinct duplicate patterns:

- **One document, multiple transactions** — same invoice attached to more than one transaction without a documented split-payment relationship.
- **Multiple sources for the same document** — same invoice arrives via both Gmail and Drive; deduplicated by content hash but flagged for awareness.
- **Same statement row matched to multiple unrelated documents** — usually a sign of a wrong-supplier match.
- **Duplicate uploads** — same file hash uploaded twice.

Every duplicate produces a review issue (severity tuned by case) and never silently auto-resolves.

---

## Multi-Match Relationships

The engine supports both one-to-many and many-to-one relationships:

- **Split payments** — one invoice paid in two transactions. The engine performs **proactive combinatorial detection**: when it sees a partial-amount match, it searches recent unmatched invoices for combinations that sum to the transaction amount. Candidate splits are surfaced as review issues for user confirmation. Confirmed splits produce two match records, both linked to the same invoice, with explicit `split_payment` flag and total reconciliation.
- **Consolidated payments** — one transaction paying multiple invoices. Multiple match records, each linked to one invoice, totals reconciled.

These cases are common in the IN workflow (clients paying multiple invoices in one transfer) and in the OUT workflow (paying a contractor's monthly bundle).

## Rejection Memory

When a user explicitly rejects a suggested match (`REJECTED_MATCH`), the engine **remembers the rejection forever** for that specific `(transaction, document)` pair. The pair is never re-suggested in any subsequent run. Other transactions and other documents are still evaluated independently — the rejection is pair-scoped, not global.

## Interfaces

### Inputs
- Transactions (Block 07), with type and tag from Block 08
- Document candidates with extracted fields from Block 09
- Recurring vendor memory and per-business pattern history (operational DB)
- AI calls through Block 06 for explanation generation

### Outputs
- `Match Record` rows in the operational DB
- Review issues for `MATCHED_NEEDS_CONFIRMATION`, `POSSIBLE_MATCH`, `NO_MATCH`, and duplicates (consumed by Block 14)
- Audit events for every match decision

## Operating Rules

- **Principle 3 (AI Assists, Rules Decide):** match outcomes are determined by rule-based scoring; AI rewrites or explains, but never overrides a Level 4 to a Level 1.
- **Principle 2 (Structured Data is Truth):** every match has a stored, structured signal breakdown — not just an AI explanation.
- **Principle 5 (Simple Interface):** users see plain-language reasons; the structured score is available on drill-down.
- **AI cache (Stage 1 decision):** repeated identical scoring or explanation calls within a run return cached results.
