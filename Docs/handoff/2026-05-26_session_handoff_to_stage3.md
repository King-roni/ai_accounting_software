## Stage 3 cadence — sub-doc backlog workflow

Each Plane ticket points to a **candidate sub-doc path** in `Docs/sub/<category>/<slug>.md` and a **hook description** from the parent phase's "Sub-doc Hooks (Stage 4)" section.

### Two paths per ticket

**🟢 VERIFY-ONLY (~80% of tickets):**
1. Confirm the candidate sub-doc exists at the path listed in the ticket's "Candidate sub-doc specs" field.
2. If it exists: read it; confirm coverage matches the hook description; set ticket to In Progress; post a DoD trace comment summarizing what the doc covers; close to Done.
3. **NO bullets / propose-and-wait cycle needed** for verify-only.
4. File 1–2 KG triples (verified status + any notable observation).

**🟠 WRITE-REQUIRED (~20% of tickets):**
1. Set ticket to In Progress.
2. Bootstrap: read the parent phase doc to understand the hook scope.
3. Propose 5–10 bullets describing the sub-doc structure you'll write.
4. **Wait for user's "go"** before writing.
5. Write the file under `Docs/sub/<category>/<slug>.md` (NEVER to project root).
6. Post DoD trace comment + close.
7. File 2–3 KG triples.

### Pass-3 staleness caveat — CRITICAL

The ticket field `"Sub-doc spec: no file match found in Docs/sub/"` was generated by an automated Pass-3 indexer that **misses files indexed under different slugs**. **Before assuming write-required:**
- Search for the file by hook keyword (e.g., `tenancy_schema_definition.md` instead of `business_entities_schema.md`).
- Check sibling sub-docs' "Cross-references" sections — the canonical file is often referenced there.
- Example: BOOK-162 said "no file match found" but `Docs/sub/schemas/tenancy_schema_definition.md` existed (182 lines, fully canonical). Pass-3 missed it; close as verify-only.

### When candidates list says multiple files

Some tickets list 2–3 candidate sub-docs ranked by token overlap. The **first** (highest overlap) is usually the right target — verify by reading its first ~60 lines + matching the hook description.

## Critical context from Blocks 02–16 (don't re-derive)

Already pinned in the project-meta drawer's "Critical conventions to honor" section, but the highlights:

- **`audit.emit_audit` signature**: `p_after_state jsonb, p_request_context jsonb` — NO `p_payload`; returns `audit.audit_events` ROW; immutable via `fn_block_update` trigger.
- **`audit_events_actor_kind_chk`**: xor — USER + actor_user_id OR SYSTEM + actor_system, never both.
- **`can_perform` is REAL RBAC** (audit C1, 2026-05-24): joins `business_user_roles × permission_matrix`. Decisions: ALLOW / DENY / REQUIRE_STEP_UP.
- **`transition_run`** accepts both `STEP_UP` and `REQUIRE_STEP_UP` can_perform decisions (B15·P04).
- **Postgres lowercases unquoted function names** → `_compose_manifest_vN_json` becomes `_compose_manifest_vn_json`.
- **PG nested aggregates forbidden** — SUM() inside string_agg() fails; use CTE to flatten.
- **`pgcrypto.digest`** lives in `extensions` schema; SECURITY DEFINER funcs with SET search_path must add `extensions`.
- **`tool_registry.tr_version_semver`** requires X.Y.Z semver; `1.0` fails.
- **ALTER TYPE ADD VALUE** has deferred visibility — split into two migrations when the new value is used in the same migration (e.g., B15·P07 ARCHIVE_PACKAGE, B16·P12 SYSTEM_OBSERVABILITY).
- **No writes from `authenticated` role anywhere** — SECURITY DEFINER RPCs only.
- **archive.locked_ledger_entries** INSERTs require SET LOCAL `app.original_lock_active='1'` (manifest v=1) or `app.adjustment_lock_active='1'` (v≥2).
- **business_entities PK = `id`** (no separate `business_id` column).
- **vat_treatment_enum** uses sentinel `UNKNOWN` (column NOT NULL; never IS NULL).
- **Mobile-write rejection** (B16·P12) is UX guard NOT security event — `_reject_mobile_write` raises P0001 `MOBILE_WRITE_REJECTED`; NO audit emit.
