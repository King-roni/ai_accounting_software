TimeFuserBooks

A premium bookkeeping-automation product for Cyprus SMEs

This document explains what was built, the design system behind it, every screen in the product, and how the code is organised. The whole suite — 14 screens — runs on one shared, token-driven visual system with first-class light and dark modes.

01 Overview

TimeFuserBooks is operating software for the monthly bookkeeping cycle: it ingests bank transactions and documents, classifies and matches them, prepares a Cyprus-VAT-correct ledger, surfaces exceptions in plain language, and finalizes each period into a tamper-evident archive. It serves both business owners and the accountants who work their books.

The brief was to take a product that "looked like a default Tailwind admin scaffold" and give it a real, distinctive identity — without making it flashy. The guiding rule throughout: design the target state, but never invent fields, statuses or entities that aren't in the domain. Features whose backend isn't live yet (analytics charts, the document viewer, e-filing) are designed as first-class but shown in an honest "awaiting data / coming soon" state, so nothing is load-bearing on fake data.

02 Design direction — "Quiet-Luxury Light"

The aesthetic target is Mercury × Stripe × a private-bank statement: warm, calm, precise, trustworthy. It deliberately rejects the cold blue-grey SaaS template look.

  • Warm over cold. Every neutral is a warm stone, never a cool grey. The canvas is warm paper; cards are white lifting on hairline borders.
  • Restraint signals expense. Hairline borders and subtle background shifts do most of the work; heavy shadow is reserved for things that truly float (drawers, modals).
  • One ink, one bronze. A single deep-ink blue carries all actions, links and active states. A muted bronze is the signature accent — used in barely two or three places total: the active-period underline, KPI hero underline, and finalized/verified seals.
  • Data is the hero. This is operating software, so tables run wide and dense (with a comfortable/compact toggle), numbers are tabular monospace and right-aligned, and dates never wrap.
Typeface decision: we use Inter as the single UI typeface (titles just go heavier and tighter), with JetBrains Mono reserved for every number, ID and date. The optional editorial serif was switched off in favour of a cleaner operating feel.

05 Status & badges

Status is the backbone of this product — there are dozens of domain statuses across runs, invoices, transactions and reviews. Every one maps to one of five tones, and colour is never the only signal: each badge carries a subtle background, a hairline border, dark legible text, and an icon.

A four-level severity quartet (Blocking / High / Medium / Low) drives the review queue, and a dedicated bronze seal marks anything finalized or integrity-verified.

06 Components

A small, consistent component kit is defined once in tokens.css and reused everywhere:

  • Buttons — primary, secondary, tertiary, danger, ghost (+ small variant), each with hover / focus / disabled states.
  • Inputs & selects — hairline fields with a focus ring; selects use a single custom chevron.
  • Tabs — underlined (section nav) and pill (lifecycle filters with counts).
  • Cards — plain and left-accent (for severity-coded review items).
  • Tables — sticky headers, sortable columns, hover lift, comfortable/compact density.
  • Right-slide drawers — the primary detail surface across the product.
  • Toast, alert, skeleton, empty state — for feedback and loading/empty conditions.

All of these are rendered live on the Foundation.html style board.

08 Dark mode

Dark mode is first-class, not an afterthought — a warm charcoal, never pure black or cold slate. It's a complete token override under [data-theme="dark"]: the brand lightens for contrast, bronze warms, money colours shift to readable variants, and status tints become low-alpha overlays. Toggle it from the topbar on any screen (try the switch top-right of this page).

Because the theme is driven entirely by CSS custom properties on the root element, a single toggle re-themes all 14 screens instantly, with no per-component work.

09 Information architecture

Every screen lives inside one shared shell: a top bar (logo · business switcher · period switcher · search · notifications · language · theme · user) and a collapsible left sidebar. The business and period switchers are cockpit controls — they re-scope everything below them.

11 Interaction patterns

A few patterns repeat across the product and give it its coherence:

  • Drawer-for-detail. Clicking any row or card opens a right-slide drawer with the full record — never a full page navigation. Esc or the scrim closes it.
  • Allowed and denied states. Every primary action has both a working state and a "denied-with-reason" state, surfaced inline rather than as a generic error. Examples: a blocking review can't be snoozed; a draft invoice can't allocate a number until it has a client; a locked ledger entry can't be edited; a period can't be finalized while gates fail.
  • Honest stubs. Backend-stubbed features render their real target shape behind an "awaiting data" / "coming soon" treatment — dashboard analytics charts, the document viewer, report generation, and Tax-Department e-filing.
  • Step-up for irreversible actions. Finalizing a period requires step-up authentication (a 6-digit modal) and produces a bronze, tamper-evident seal.
  • Plain language for humans. The review queue is written for a non-accountant — calm, reassuring, never alarming or gamified.

12 Architecture

The product is built as plain, dependency-free HTML/CSS/JS — no framework, no build step. It is intentionally simple so it can be opened directly in a browser and handed to engineering as a faithful reference.

  • styles/tokens.css — the single source of truth for all design tokens (light + dark), plus base component classes (badges, buttons, fields, cards, tables, toast, alert, skeleton).
  • styles/shell.css — the app frame: top bar, sidebar, page header, segmented controls, and the right-slide drawer.
  • shared/shell.js — injects the top bar + sidebar into every page, owns the icon set (inline SVG, no icon font), and handles theme, period and sidebar state. Each page calls CB.renderShell({ active: '…' }).
  • shared/periods.js / shared/invoices.js — the two data-heavy screens have their run/phase/gate and invoice/line-item/VAT logic split into modules to keep page files lean.
Each page is a thin HTML file: it links the two stylesheets, contains its own screen markup and demo data, then calls the shared shell. Adding a screen means copying this pattern — the visual system comes for free.

14 Running it

No install, no build. Open Dashboard.html in any modern browser and click through the sidebar — every screen links to the next. Toggle light/dark from the topbar; switch periods with the ‹ May 2026 › control. All data is illustrative (Demo Trading Ltd, May 2026).

Because the pages reference styles/ and shared/ by relative path, keep the folder structure intact when moving the bundle.
